Online refresh tokens are a type of [refresh token](https://auth0.com/docs/secure/tokens/refresh-tokens) designed specifically for browser-based applications, such as [single-page applications](/docs/get-started/auth0-overview/create-applications/single-page-web-apps) (SPAs). Unlike standard refresh tokens, online refresh tokens are bound to the user's Auth0 [session](/docs/manage-users/sessions) and can only be used while that session is active.

Online refresh tokens solve the challenge of maintaining user sessions in SPAs when browser privacy features like Intelligent Tracking Prevention (ITP) block access to third-party cookies, making traditional [silent authentication](/docs/authenticate/login/configure-silent-authentication#configure-silent-authentication) unreliable.

With standard refresh tokens, each application receives its own independent token that can be revoked separately. This creates inconsistent user experiences across applications, a user might be logged out of one application while remaining logged in to others, even though they expect a unified session. 
Online Refresh Tokens address this by binding all tokens to a single session, ensuring that when a user logs out, all applications using that session are logged out together.

## How Online Refresh Tokens work

When you request an Online Refresh Token:

1. Your application includes the `online_access` scope in the authorization request.
2. Auth0 returns an Online Refresh Token bound to the user's current session.
3. Each time you exchange the Online Refresh Token for a new access token, Auth0 extends the session's idle timeout.
4. The token remains valid as long as the underlying Auth0 session is active.

The key benefit is that [Single Sign-On](https://auth0.com/docs/authenticate/single-sign-on) (SSO) continues to work across all your applications because the session is actively maintained server-side, independent of browser cookie limitations.

Because all Online Refresh Tokens within a session are bound together, if you revoke any one of them, the entire session terminates. This means all other Online Refresh Tokens associated with that session become invalid immediately, and SSO ends for the user across all applications. This unified revocation model ensures consistent logout behavior.

## Online vs. offline refresh tokens

The following table highlights the key differences between online and standard refresh tokens:

| **Feature** | **Online refresh token** | **refresh token** |
| ----- | ----- | ----- |
| **Scope requested** | `online_access` | `offline_access` |
| **Session binding** | Bound to Auth0 session | Independent of session |
| **Lifetime** | Determined by session lifetime | Configured token lifetime |
| **Token rotation** | Does not rotate | Supports rotation (configurable) |
| **[Demonstration of Proof of Possession](/docs/secure/sender-constraining/demonstrating-proof-of-possession-dpop)  (DPoP) requirement** | Required for public clients (SPA, Native) | Optional (resource server config) |
| **Revocation** | Revokes the entire session | Revokes only the token |
| **Management API** | Not visible | Visible via the Management API |
| **Session idle timeout** | Extended on each exchange | No effect on session |

### Key behavioral differences

**Session extension:** When you exchange an Online Refresh Token, Auth0 resets the session idle timeout, keeping the session alive. The session's absolute lifetime is not extended—this is a hard limit that cannot be changed via token exchange.

**No rotation:** Online Refresh Tokens do not rotate. Even if you have refresh token rotation enabled for your application, Online Refresh Tokens are not replaced when exchanged. This is because they are stateless and bound to the session rather than stored individually.

**DPoP enforcement:** For public clients (such as SPAs), DPoP is required when using Online Refresh Tokens. Because Online Refresh Tokens operate at the same security level as a session cookie—granting access to the user's authenticated session—they must be protected with the same rigor. DPoP provides this protection by cryptographically binding the token to a specific client, preventing token theft and replay attacks.

**Stateless:** Online Refresh Tokens are bound to a session, so they cannot be seen or managed independently. This means you cannot see them in the Management API or revoke them individually.

## Configure Online Refresh Tokens

Online Refresh Tokens must be enabled at the API (Resource Server) level. You can configure this through the Auth0 Dashboard or the Management API.

### Configure using the Dashboard

1. Go to [Dashboard > Applications > API](https://manage.auth0.com/#/apis).
2. Select the API you want to configure its settings.
3. In the *Settings** tab, enable the **Allow Online Access** toggle.
5. Select **Save**.

To configure appropriate idle timeout and absolute lifetime values, review your session settings under your Auth0 [Tenant Session Expiration](https://manage.auth0.com/#/tenant/advanced) settings. To learn more, read [Configure Session lifetime](https://auth0.com/docs/manage-users/sessions/configure-session-lifetime)

### Configure using the Management API

To enable Online Refresh Tokens, use the [Update Resource Server](/docs/api/management/v2/resource-servers/patch-resource-servers-by-id) endpoint:

```shell
curl --request PATCH \
  --url 'https://<YOUR_DOMAIN>/api/v2/resource-servers/<YOUR_RESOURCE_SERVER_ID>' \
  --header 'authorization: Bearer <YOUR_MANAGEMENT_API_TOKEN>' \
  --header 'content-type: application/json' \
  --data '{"allow_online_access": true}'
```

To verify the configuration, get the resource server:

```shell
curl --request GET \
  --url 'https://<YOUR_DOMAIN>/api/v2/resource-servers/<YOUR_RESOURCE_SERVER_ID>' \
  --header 'authorization: Bearer <YOUR_MANAGEMENT_API_TOKEN>'
```

The response includes the `allow_online_access` property:

```json
{
  "id": "resource-server-id",
  "name": "My API",
  "identifier": "https://my-api.example.com",
  "allow_online_access": true,
  ...
}
```

## Get an Online Refresh Token

To get an Online Refresh Token, include the `online_access` scope in your authorization request using the [Authorization Code Flow](/docs/get-started/authentication-and-authorization-flow/authorization-code-flow) or [Authorization Code Flow with PKCE](/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).

```shell
https://<YOUR_DOMAIN>/authorize?
    audience=<YOUR_API_AUDIENCE>&
    scope=openid profile online_access&
    response_type=code&
    client_id=<YOUR_CLIENT_ID>&
    redirect_uri=<https://YOUR_APP/callback>&
    state=<OPAQUE_VALUE>
```

After the user authenticates, exchange the authorization code for tokens:

```shell
curl --request POST \
  --url 'https://<YOUR_DOMAIN>/oauth/token' \
  --header 'content-type: application/x-www-form-urlencoded' \
  --data grant_type=authorization_code \
  --data 'client_id=<YOUR_CLIENT_ID>' \
  --data 'code=<YOUR_AUTHORIZATION_CODE>' \
  --data 'redirect_uri=<https://YOUR_APP/callback>' \
  --data 'code_verifier=<YOUR_CODE_VERIFIER>'
```

The token response includes the Online Refresh Token:

```json
{
  "access_token": "eyJ...",
  "id_token": "eyJ...",
  "refresh_token": "ORT...",
  "token_type": "Bearer",
  "expires_in": 86400
}
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
Online Refresh Tokens are prefixed with `ORT` to distinguish them from Offline Refresh Tokens. However, you should treat the token as opaque and not rely on its internal structure.
</Callout>

## Use an Online Refresh Token

Exchange the Online Refresh Token for a new access token when the current access token expires or is about to expire:

```shell
curl --request POST \
  --url 'https://<YOUR_DOMAIN>/oauth/token' \
  --header 'content-type: application/x-www-form-urlencoded' \
  --data grant_type=refresh_token \
  --data 'client_id=<YOUR_CLIENT_ID>' \
  --data 'refresh_token=<YOUR_ONLINE_REFRESH_TOKEN>'
```

The response contains a new access token and optionally a new ID token:

```json
{
  "access_token": "eyJ...",
  "id_token": "eyJ...",
  "token_type": "Bearer",
  "expires_in": 86400
}
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
No new `refresh_token` is returned. Online Refresh Tokens do not rotate—you continue using the same token for subsequent exchanges.
</Callout>

### Session extension behavior

Each successful token exchange:

* **Resets the session idle timeout:** The idle timeout is restored to its full duration.
* **Does not extend absolute lifetime:** The session's absolute expiration remains unchanged.
* **Maintains SSO:** Other applications can get tokens via SSO as long as the session is alive.

## Revoke an Online Refresh Token

When you revoke an Online Refresh Token, it terminates the entire Auth0 session, not just the token. This invalidates all Online Refresh Tokens bound to that session and ends SSO for the user.

```shell
curl --request POST \
  --url 'https://<YOUR_DOMAIN>/oauth/revoke' \
  --header 'content-type: application/x-www-form-urlencoded' \
  --data 'client_id=<YOUR_CLIENT_ID>' \
  --data 'token=<YOUR_ONLINE_REFRESH_TOKEN>'
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
Unlike standard Refresh Tokens, you cannot revoke an Online Refresh Token independently. Revocation always ends the user's session.
</Callout>

## Use Online Refresh Tokens with Actions

Online Refresh Tokens are accessible in Auth0 Actions during the post-login trigger. The `event.refresh_token` object includes information about the token type.

During a token refresh operation with an Online Refresh Token, the `event.session` object is populated with the session the token belongs to. Use the `event.session` object to access session data and make decisions based on the current session state.

In the post-login trigger, you can detect when Auth0 issues an Online Refresh Token:

```javascript
exports.onExecutePostLogin = async (event, api) => {
  // Check if online_access scope was requested
  const requestedScopes = event.transaction?.requested_scopes || [];
  if (requestedScopes.includes('online_access')) {
    console.log('Online Refresh Token will be issued');
    console.log('Session ID:', event.session?.id);
  }
};
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
The `api.refreshToken.revoke()` method, when called for an Online Refresh Token, revokes the entire session rather than just the token. All Online Refresh Tokens bound to that session become invalid.
</Callout>

## Best practices

1. **Store tokens in memory:** Avoid storing Online Refresh Tokens in `localStorage` or `sessionStorage`. Keep them in JavaScript memory for better security. Since they are bound to the session, the cookie should be there next time the browser is open.
2. **Proactive refresh:** Exchange tokens before they expire (for example, 60 seconds before expiration) to avoid interrupting user workflows.
3. **Handle session expiration gracefully:** Implement proper error handling for cases when the session has expired. Redirect users to re-authenticate without losing their work.
4. **Use DPoP for public clients:** Online Refresh Tokens require DPoP for public clients. Ensure your implementation generates and uses DPoP proofs correctly.
5. **Do not rely on token structure:** Treat Online Refresh Tokens as opaque strings. Don't parse or rely on any internal structure—this may change.
6. **Monitor session lifetime:** Be aware of both idle and absolute session timeouts. The absolute timeout cannot be extended via refresh token exchange.

## Limitations

### Unsupported flows

Online Refresh Tokens are only available for flows that create an interactive browser session. The following flows **cannot** issue Online Refresh Tokens:

| **Flow** | **Reason** |
| ----- | ----- |
| [Device Authorization Flow](/docs/get-started/authentication-and-authorization-flow/device-authorization-flow) | Session created in browser, token used on different device |
| [Implicit Flow](/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post) | OAuth spec prohibits refresh tokens in implicit flow |
| [Resource Owner Password Grant](/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow) | No browser session is created |
| [Client Credentials Flow](/docs/get-started/authentication-and-authorization-flow/client-credentials-flow) | No user or session involved |
| [Custom Token Exchange](/docs/get-started/authentication-and-authorization-flow/token-exchange-flow) | No session to bind the token to |

If you request `online_access` scope in any of these flows, the scope is silently ignored and no refresh token is issued.

### Scope conflicts

You cannot request both `online_access` and `offline_access` in the same authorization request. If both scopes are included, Auth0 returns an error:

```json
{
  "error": "invalid_request",
  "error_description": "Cannot request both online_access and offline_access scopes"
}
```

Choose the appropriate scope based on your use case:

- Use `online_access` for browser-based applications that benefit from SSO continuity.
- Use `offline_access` for applications that need tokens to work independently of the browser session.
