Configure your Auth0 tenant as the Requesting App so it can obtain an ID-JAG from the enterprise IdP and exchange it for an access token to call the Resource App's API. To learn about setting up your tenant as the Resource App instead, read Set up Auth0 as Resource App.

To set up your Requesting App:

- Configure and register your Requesting App in Auth0: This includes registering a Requesting App in your Auth0 tenant and updating your IdP (e.g. Okta) to link it with the Resource App. To learn more, read [Requesting App setup](#requesting-app-setup).
- Configure how your Auth0 tenant federates with your customer’s enterprise IdP: To learn more, read [Configure an Okta Workforce Enterprise connection in Auth0](docs/secure/call-apis-on-users-behalf/xaa/idp/okta-as-oidc-idp#configure-an-okta-workforce-enterprise-connection-in-auth0).

## Requesting App setup

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">

In a production environment, you configure each Requesting App once to enable its connection with your Resource App.

</Callout>

To set up your Requesting App, you need to: 

- [Create the Requesting App in Auth0](#create-the-requesting-app-in-auth0)
- [Register the Requesting App in Okta](/docs/secure/call-apis-on-users-behalf/xaa/idp/okta-as-oidc-idp#register-the-requesting-app-in-okta)

### Create the Requesting App in Auth0

Create and register an application that behaves as the Requesting App. This can be a first-party or third-party client, and can be a Regular Web Application, Native App, or Single-Page App (SPA) — Auth0 supports confidential and public clients for XAA. Applications registered via [Client ID Metadata Document (CIMD)](/docs/get-started/applications/dynamic-client-registration) can also be used.

To [create an application](/docs/get-started/auth0-overview/create-applications) representing the Requesting App in your Auth0 tenant:

- Navigate to **Applications > Applications** and select **Create Application**.
- Enter a name and select **Regular Web Application**, **Native**, or **Single Page Application** depending on the type of application you want to use.

<Frame>![](/docs/images/xaa/xaa_create_regular_web_app.png)</Frame>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">

If your application is a third-party application, which is the most common case, you must explicitly configure its API access permissions. To learn more, read [API Access Policies for Third-Party Applications](https://auth0.com/docs/get-started/apis/api-access-policies-for-applications#api-access-policies-and-third-party-applications).

</Callout>

- In the application details, note the **Client ID** of the application. This is required during [Register the Requesting App in Okta](/docs/secure/call-apis-on-users-behalf/xaa/idp/okta-as-oidc-idp#register-the-requesting-app-in-okta).

<Frame>![](/docs/images/xaa/xaa_req_app_details.png)</Frame>

- Once you’ve created the application, scroll to **Settings** and enable the **Cross App Access** toggle.

<Frame>![](/docs/images/xaa/allow_xaa_auth0_app.png)</Frame>

Once you’ve created and configured your application, you must provide Okta with the application’s `client_id` and the issuer URL of your Auth0 tenant. This enables the connection between the Requesting App, identified by the `client_id`, and the Resource App, identified by the issuer URL. To learn more, read [Test the end-to-end XAA flow](/docs/secure/call-apis-on-users-behalf/xaa/end-to-end-testing).

For Okta-specific configuration, see [Register the Requesting App in Okta](/docs/secure/call-apis-on-users-behalf/xaa/idp/okta-as-oidc-idp#register-the-requesting-app-in-okta).
