This page walks through configuring Okta as the SAML enterprise identity provider for Cross App Access (XAA). You'll set up an Okta tenant, register the Resource and Requesting Apps in Okta, and configure a SAML Enterprise connection so Auth0 can federate with Okta.

## Create and configure your Okta tenant

To set up your end-to-end test environment for the Resource App, you need to create and configure your Okta tenant for Cross App Access.

- On the [Okta Developer website](https://developer.okta.com/signup/), sign up for an **Okta Integrator Free Plan**. Once you sign up, you should be redirected to your new Okta tenant.
- If Cross App Access is not enabled in your Okta tenant, email `developers@okta.com` to request XAA enablement for your Okta tenant.

Once Cross App Access feature is enabled, the following two flags should be On in under your Okta Admin > Settings > Features.

<Frame>![](/docs/images/xaa/xaa_okta_flags_ea.png)</Frame>

## Register the Requesting App in Okta

### Create Requesting App in Okta
Your Requesting application is a standard SAML Application. Go to **Applications > Applications > Create App Integration** and select **SAML 2.0** as type.

<Frame>![](/docs/images/xaa/xaa_okta_req_app_new_type_saml.png)</Frame>

Enter application name and Single Sign On URL back to your web application.

<Frame>![](/docs/images/xaa/okta_req_app_new_application_saml.png)</Frame>

Once requesting application is created, it's assigned SAML 2.0 metadata.

<Frame>![](/docs/images/xaa/xaa_okta_req_app_sign_on_saml.png)</Frame>

### Assign Requesting Application to Test Users
Finally, allow your test user to log into the Requesting App in Okta.

In the Okta Admin Console:

1. Navigate to **Applications** and select the requesting application you created.
2. Select **Assign > Assign to People** and select your test user.
3. Select **Save**.

<Frame>![](/docs/images/xaa/xaa_okta_req_app_user_assignment_saml.png)</Frame>

## Register the Resource App in Okta

### Create Resource App in Okta
Similar to Requesting Application, your Resource Application in a SAML 2.0 Application.

1. Go to **Applications > Applications > Create App Integration** and select **SAML 2.0** as type.
2. Under Configure SAML:
    - Set  **Single Sign On URL** field to your **Auth0 Tenant’s callback URL**. Your callback URL will be your Auth0 domain followed by `/login/callback`
    - Set **Audience URI (SP Entity ID)** to `urn:auth0:TENANT:CONNECTION` format. Pick a connection name that we will later reuse when establishing SAML connection between Okta and Auth0.
3. Select **Next** and **Finish**.

<Frame>![](/docs/images/xaa/xaa_okta_resource_app_details_saml.png)</Frame>

### Assign Resource Application to Test Users
Finally, allow your test user to log into the Requesting App in Okta.

In the Okta Admin Console:

1. Navigate to **Applications** and select the resource application you created.
2. Select **Assign > Assign to People** and select your test user.
3. Select **Save**.

<Frame>![](/docs/images/xaa/xaa_okta_resource_app_user_assignment_saml.png)</Frame>

### Enable Cross-app Access (XAA) in Resource Application
Go to **Resource Server** tab and enable Cross-app access (XAA) for Resource Application. **Issuer URL** of this app's authorization server
is `https://AUTH0_DOMAIN/`.

<Frame>![](/docs/images/xaa/xaa_okta_resource_app_resource_server_saml.png)</Frame>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">

    Issuer URL must end with the trailing forward slash **`/`**. That's the issuers format Auth0 issues and accepts tokens.

</Callout>

## Register AI Agent in Okta
### Create AI Agent

Visit Directory > AI Agents > Register AI Agent and select **Register manually**. Pick a name for your agent and assign an owner.

<Frame>![](/docs/images/xaa/xaa_okta_register_ai_agent_saml.png)</Frame>

At this stage, your AI Agent is `STAGED`. Before activating the agent, we need to configure the following steps.

<Frame>![](/docs/images/xaa/xaa_okta_ai_agent_staged_saml.png)</Frame>

### Add Agent Credentials
AI Agents authenticate on behalf of your applications with private key JWT. In the Credentials tab, select **Add public key**.

You can either provide public key of your own key pair or **Generate new key** and download its private key and key ID (kid).
Once credentials are set correctly, checkbox appears on the side of the tab.

<Frame>![](/docs/images/xaa/xaa_okta_ai_agent_credentials_set_saml.png)</Frame>

### Add Agent Delegations
In the Delegations tab, select **Add Caller** in the **User sign-on** section.
Select the **Requesting** application, then select **Add caller**

<Frame>![](/docs/images/xaa/xaa_okta_ai_agent_delegations_assign_saml.png)</Frame>

Once delegations are set correctly, checkbox appears on the side of the tab.

<Frame>![](/docs/images/xaa/xaa_okta_ai_agent_delegations_set_saml.png)</Frame>

### Add Agent Resource connections
In the Resource connections tab, select **Add resource connections** and select **Application** as resource type.

In the Application section, select Resource application you created and set **AI agent's client ID registered in this app** to **client_id** of the **Agent0** application in your **Auth0 tenant**.

<Frame>![](/docs/images/xaa/xaa_okta_ai_agent_resource_connections_tab_saml.png)</Frame>

Once delegations are set correctly, checkbox appears on the side of the tab.

### Activate the Agent
From the Agent's Actions drop down, select **Activate** and check "MANAGED STATUS" is showing up as **Active**.

## Configure a SAML Enterprise connection in Auth0

Use your **Resource App**’s `client_id` and `client_secret` to [create a SAML Enterprise connection](/docs/authenticate/identity-providers/enterprise-identity-providers/saml) in your Auth0 tenant.
In your Auth0 Tenant's Management Console go to Authentication > Enterprise > SAML and select **+ Create Connection**.

Copy **Sign In URL** and **X509 Signing Certificate** from Resource Application's **Sign On** tab.

<Frame>![](/docs/images/xaa/xaa_auth0_new_okta_saml_connection.png)</Frame>

When creating the SAML Enterprise connection, activate the **Cross App Access - Resource Application** role. This enables your Resource App to accept ID-JAGs issued by the enterprise IdP associated with that connection, in this case, your Okta tenant.
Since your connection is SAML, you need to enter OpenID Connect **Issuer or discovery URL** pointing to your Okta tenant.

<Frame>![](/docs/images/xaa/xaa_auth0_connection_saml_xaa_enabled.png)</Frame>

After creating the Okta Workforce Enterprise connection, check that the **Callback URL provided by Auth0** in the connection's settings, matches the **Redirect URI** configure the sign-on policies of the **Resource App in your Okta** tenant.

### Testing Connection in Auth0
In the Auth0 Dashboard:

- Navigate to **Authentication > Enterprise > Okta Workforce**:
- Enter the Okta Workforce Enterprise connection you created and select the **Applications** tab. Then, enable the Requesting App you created for the connection.
- Go back to the list of Okta Workforce connections. Select the three dots on the right for your connection and select **Try**. You will be redirected to authenticate in your Okta tenant to complete the login with your test user.

<Frame>![](/docs/images/xaa/xaa_auth0_try_saml_connection.png)</Frame>

- Login with the user you assigned to XAA Resource Applications

<Frame>![](/docs/images/xaa/xaa_okta_saml_login.png)</Frame>

- Verify login was successful

<Frame>![](/docs/images/xaa/xaa_auth0_saml_try_success.png)</Frame>

Once user successfully logins with federated connection, their user-profile is created in Auth0

<Frame>![](/docs/images/xaa/xaa_auth0_try_profile_saml.png)</Frame>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">

    It's important that users login with XAA-enabled enterprise connection at least once so that user profile exists in Auth0.
    Auth0 does not support JIT user creation with ID-JAG at the moment.

</Callout>
