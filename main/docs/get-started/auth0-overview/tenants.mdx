An Auth0 <Tooltip tip="Tenant: A logical isolation unit in Auth0. Each tenant has its own set of applications, connections, users, and configuration." cta="View Glossary" href="/docs/glossary?term=tenant">tenant</Tooltip> is a logically isolated unit that holds all of your Auth0 configuration and data. When you sign up for Auth0, you create your first tenant. Everything you configure after that — applications, connections, users, security policies, branding — lives inside that tenant.

Think of a tenant as a single identity environment. Each tenant has its own user directory, its own login experience, its own set of credentials, and its own logs. Nothing is shared between tenants unless you explicitly build an integration to do so.

Your tenant name becomes part of your Auth0 domain. For example, if you name your tenant `example-co` and select the US region, your domain is `example-co.us.auth0.com`. This domain is the base URL for your authentication requests, your Management API calls, and your Universal Login page.

## What lives inside a tenant

A tenant contains every resource you need to build and operate an identity integration:

| Resource | What it does |
| --- | --- |
| **[Applications](/docs/get-started/applications)** | The web apps, SPAs, native apps, and machine-to-machine clients that use Auth0 for authentication. Each application gets its own client ID and credentials. |
| **[APIs](/docs/get-started/auth0-overview/set-up-apis)** | The resource servers you protect with access tokens. You define permissions (scopes) here that your applications can request. |
| **[Connections](/docs/authenticate/identity-providers)** | The identity sources your users log in with — databases, social providers (Google, GitHub, Apple), enterprise providers (SAML, Microsoft Entra ID, Okta Workforce Identity Cloud), and passwordless options (email, SMS). |
| **[Users](/docs/manage-users)** | Every user profile in your directory, including credentials, metadata, linked accounts, login history, and role assignments. |
| **[Branding](/docs/customize)** | The look and feel of your Universal Login page, email templates, phone message templates, and custom domain configuration. |
| **[Actions](/docs/customize/actions)** | Serverless functions that run at specific points in the Auth0 pipeline — login, registration, password reset, M2M token exchange — to add custom logic. |
| **[Security policies](/docs/secure)** | Attack protection rules (bot detection, brute-force protection, breached password detection), multi-factor authentication settings, and access control policies. |
| **[Logs](/docs/deploy-monitor/logs)** | A record of every authentication event and administrative action in your tenant, searchable by event type, user, IP address, and timestamp. |
| **[Tenant settings](/docs/get-started/tenant-settings)** | Global configuration that affects the entire environment — session timeouts, signing keys, environment tags, and team member access. |

## Tenant isolation

Every tenant is fully independent. Tenants do not share users, configuration, credentials, or data. A change to one tenant — adding a connection, updating a security policy, modifying branding — has no effect on any other tenant.

This isolation is what makes tenants the natural boundary for separating environments, business units, and compliance domains. For example, you can give your engineering team a development tenant where they test freely, while your production tenant stays untouched. If your company operates multiple brands that need independent identity configurations, each brand gets its own tenant. And because tenants are region-specific, you can satisfy data-residency requirements by placing each tenant in the appropriate geography.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">

If your application serves business customers who each need their own login configuration but share the same user directory, [Organizations](/docs/manage-users/organizations) may be a better fit than separate tenants. Organizations provide per-customer branding and connection routing within a single tenant.

</Callout>

## When to use multiple tenants

Most teams start with one tenant and add more as their needs grow. The most common reason is the development lifecycle: one tenant for development, one for staging, and one for production. This lets you test configuration changes and new connections without risk to live users, then promote those changes when they're ready. To learn more, read [Set Up Multiple Environments](/docs/get-started/auth0-overview/create-tenants/set-up-multiple-environments).

Beyond environment separation, you might create additional tenants when you have distinct user populations that should never interact (for example, an internal workforce directory and a customer-facing directory), or when users in different regions must have their data stored in specific geographies.

To learn more, read [Create Multiple Tenants](/docs/get-started/auth0-overview/create-tenants/create-multiple-tenants) and [Multi-Tenant Application Best Practices](/docs/get-started/auth0-overview/create-tenants/multi-tenant-apps-best-practices).

## Tenant naming and domains

When you create a tenant, you choose a name that becomes part of your Auth0 domain. This name is permanent — it cannot be changed or reused after deletion. Tenant names must be between 3 and 63 characters long and can contain only lowercase alphanumeric characters and hyphens (`-`). A name cannot begin or end with a hyphen.

Your Auth0 domain follows the pattern `{tenant-name}.{locality}.auth0.com`. The locality value is determined by the region you select when creating the tenant:

| Region | Example domain |
| --- | --- |
| United States | `example-co.us.auth0.com` |
| Europe | `example-co.eu.auth0.com` |
| Australia | `example-co.au.auth0.com` |
| Japan | `example-co.jp.auth0.com` |
| United Kingdom | `example-co.uk.auth0.com` |
| Canada | `example-co.ca.auth0.com` |

For production environments, you can replace the default Auth0 domain with a custom domain (for example, `login.example-co.com`). Custom domains give your users a branded login experience and avoid third-party cookie issues. To learn more, read [Custom Domains](/docs/customize/custom-domains).

## Manage your tenant

The [Auth0 Dashboard](https://manage.auth0.com/) is the web-based management console and the best starting point for exploration, debugging, and one-off configuration tasks. To learn more, read [The Auth0 Dashboard](/docs/get-started/auth0-overview/dashboard).

Every Dashboard operation is also available programmatically through the [Management API](/docs/api/management/v2). For terminal-based workflows, the [Auth0 CLI](https://auth0.github.io/auth0-cli/) lets you manage tenants, applications, and other resources from the command line. And for infrastructure-as-code, the [Terraform provider](/docs/deploy-monitor/auth0-terraform-provider) lets you define and version your Auth0 configuration alongside the rest of your infrastructure.

Most teams use the Dashboard for interactive work and the API, CLI, or Terraform for automation and CI/CD workflows.
