The [Agent Experience Score](https://auth0.com/agent-experience) measures how well AI agents implement Auth0 across different models and frameworks. Visit the live results page to compare scores, filter by framework, and see how Auth0 tools improve agent performance.

This page explains the scoring methodology — how each score is calculated, what the dimensions measure, and how grades are assigned.

## What gets tested

Real AI agents — Claude Code, GitHub Copilot, Gemini CLI — run real Auth0 integration tasks in real development environments. Each agent gets a workspace, a shell, and file tools, just like a developer would. The prompts are short and realistic: "add authentication to my Next.js app," not step-by-step recipes.

Each model is tested **with and without Auth0 tools** ([MCP Server](/docs/get-started/build-with-ai-tools#auth0-docs-mcp-server) and [Agent Skills](/docs/quickstart/agent-skills)). The difference between those scores is the measurable impact of Auth0's AI tooling on the developer experience.

## Scoring dimensions

Every run is scored across 7 dimensions, then combined into a single weighted score out of 100.

### Process — did the agent work smoothly?

| Dimension | What it measures |
|-----------|------------------|
| **Setup Friction** | Did the agent complete the task autonomously, or did it stop to ask questions and hit errors? |
| **Setup Speed** | How quickly did the agent finish? Measured by active execution time, so results are comparable across environments. |
| **Efficiency** | How many steps did it take? Fewer tool calls means less cost and less complexity. |
| **Error Recovery** | How often did infrastructure errors (rate limits, timeouts) disrupt the run? |

### Output — does the code work?

| Dimension | What it measures |
|-----------|------------------|
| **Correctness** | Does the generated code import real packages, call real methods, and wire components correctly? |
| **Hallucination** | Did the agent invent packages that don't exist or use wrong SDK variants? |
| **Security** | Did the agent hardcode secrets, store tokens insecurely, or commit credentials to source code? |

Each dimension is scored 0–100 individually, then weighted and summed into the overall score you see on the [results page](https://auth0.com/agent-experience).

## Grades

Overall scores map to letter grades:

| Grade | Min score | What it means |
|-------|-----------|---------------|
| A | 90 | Production-ready. Minimal issues. |
| B | 75 | Solid, but with a few gaps to fix. |
| C | 60 | Usable, but needs cleanup. |
| D | 40 | Significant problems. |
| F | &lt; 40 | Not useful — faster to start from scratch. |

Grades are calibrated to match developer intuition. A score of 91 should feel like code you'd accept with minimal review. A score of 55 should feel like something that needs real work to fix.

## How results are validated

Every grader verifies generated code — not prose or explanations. Graders check that code compiles, imports real packages, calls actual SDK methods, and doesn't introduce security vulnerabilities.

Results are validated at multiple levels:

- **Presence checks**: Required SDK symbols, imports, and config keys exist in the output.
- **Hallucination detection**: Invented packages, wrong SDK variants, and fabricated API methods are caught.
- **Security checks**: Hardcoded credentials, tokens in insecure storage, and secrets in source code are flagged.
- **Structural validation**: Code is correctly wired — right components in right files, lifecycle hooks handled, middleware in the correct order.
- **Version correctness**: The agent uses current APIs, not deprecated patterns (only checked when the agent has access to current docs).
- **Holistic review**: An LLM judge evaluates overall correctness of the implementation.

## Improve your agent's results

The same tools used in testing are available to every developer. Enabling them gives your AI agent access to current Auth0 documentation and SDK best practices.
