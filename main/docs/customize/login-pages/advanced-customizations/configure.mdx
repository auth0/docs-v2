You can configure ACUL using the Auth0 Management API, Dashboard and Infrastructure-as-Code(IaC) tools.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
To configure ACUL, make sure your custom configuration files and assets are hosted on a public CDN. 
</Callout>

## The ACUL rendering configuration object

The ACUL rendering configuration is a single JSON object that is used to enable ACUL and list all the available properties, per screen, which consist of specific assets, context data, and applicable conditional logic. 

ACUL rendering configuration object: 

```json
{
  "rendering_mode": "advanced",
  "context_configuration": [...],
  "default_head_tags_disabled": false,
  "head_tags": [...],
  "filters": {
    "match_type": "includes_any",
    "clients": [...],
    "organizations": [...],
    "domains": [...]
  },
  "use_page_template": false
}
``` 

ACUL rendering configuration object properties:

| Property      | Required |  Description   |
| ----------- | ----------- | ----|
| `rendering_mode`      | Yes | String. It enables ACUL for the screen.<br /> Possible values: <ul><li><code>standard</code> Renders the standard Universal Login screen. </li> <li><code>advanced</code> Enables ACUL and uses the ACUL rendering configuration object. </li></ul> Defaults to `standard`. |
|`use_page_template` |  Yes | Boolean. Defines whether your custom screen is rendered inside your tenant's existing custom page template.<br /> Possible values: <ul><li>`True` If a custom page template is defined, the screen renders within it. A warning is logged if no template exists.</li><li>`False` The screen renders using a minimal template.</li></ul>Defaults to `false`. |
|`default_head_tags_disabled`| No | Boolean. Defines whether Auth0 injects its default set of `head` tags into the page.<br /> Possible values: <ul><li>`True` Disables all default tags, giving you complete control over the `head`. You are responsible for providing necessary tags.</li> <li>`False` Auth0 will include its standard default tags.</li></ul> Defaults to `false`.|
| `context_configuration` | Yes | Boolean. Array of Strings. An array specifying additional data points to include in the `universal_login_context` object passed to your UI.<br /> Possible values: <ul> <li>`branding.settings`</li> <li>`branding.themes.default`</li> <li>`client.logo_uri`</li> <li>`client.description`</li> <li>`client.metadata.[keyName]`</li> <li>`organization.display_name`</li> <li>`organization.branding`</li> <li>`organization.metadata.[keyName]`</li> <li>`request.ip`</li> <li>`request.geoip`</li> <li>`request.hostname`</li> <li>`request.language`</li> <li>`screen.texts`</li> <li>`tenant.name`</li> <li>`tenant.friendly_name`</li> <li>`tenant.enabled_locales`</li> <li>`transaction.domain.metadata.[keyName]`</li> <li>`untrusted_data.submitted_form_data`</li> <li>`untrusted_data.authorization_params.ui_locales`</li> <li>`untrusted_data.authorization_params.login_hint`</li> <li>`untrusted_data.authorization_params.screen_hint`</li> <li>`untrusted_data.authorization_params.ext-[keyName]`</li>  <li>`user.organizations`</li>  <li>`user.app_metadata.[keyName]`</li>  <li>`user.user_metadata.[keyName]`</li> </ul>Limit of 100 entries. |
| `head_tags` | Yes | Array of Objects. Injects HTML tags into the `head` of the host page.<br /> Each object defines and HTML tag and can contain: <ul> <li>`tag` (String, required) Must be an HTML element valid for use in the `head`tag.</li> <li>`content` (String, optional)  Supports using the context data as liquid variables, for example ` client.logo_url `. Limit of 250 characters. </li> <li>`attributes` (Object, optional) Key-value map. <ul> <li>[integrity](https://developer.mozilla.org/en-US/docs/Web/Security/Subresource_Integrity) can be an array. Entity Limit: 10.</li> <li>URL atributes (src, href, etc) support dynamic segments and allow the usage of following variables: <ul> <li> `locale` </li> <li> `screen.name` </li> <li> `prompt.name` </li> <li> `client.id` </li> <li> `client.name` </li> <li> `client.metadata.[keyName]` </li> <li> `organization.id` </li> <li> `organization.name` </li> <li> `organization.metadata.[keyName]` </li> </ul> </li> </ul> </li> Limit of 10 attributes. </ul>  |
| `filters` | No | Object. Allows the conditional application of the entire ACUL configuration.<br /> Each object can contain:  <ul> <li>`match_type` (String, required)<br /> Possible values: <ul> <li> includes_any </li> <li> excludes_any </li> </ul> </li> <li>`clients` (Array of Objects, optional)<ul> <li> Uses `id` or `metadata` </li> <li> Limit of 25 objects. </li> </ul> </li> <li>`organizations` (Array of Objects, optional) <ul> <li> Uses `id` or `metadata` </li> <li> Limit of 25 objects. </li> </ul> </li>  </ul> |

## Auth0 Dashboard
You can use the Dashboard to customize your screens with the ACUL:

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
The Dashboard settings follow the properties of the ACUL rendering configuration object. 
</Callout>

1. Navigate to [Dashboard > Branding > Universal Login](https://manage.auth0.com/dashboard/#/universal-login/customizations-new)
	1. Select **Customize authentication screens**
	2. Select a <code>screen</code> to  customize
	3. From **Settings > Rendering mode** select **Advanced mode**
	4. (Optional) Select **Use custom page template**  
	5. (Optional) Select **Advanced mode filters**
	6. Select **Head Tags** to add your own html head tags
	7. Select **Additional Data** to add data from the <code>universal_login_context object</code>
2. Select Save & Publish

## Infrastructure-as-Code (IaC)

You can configure ACUL with IaC tools, such as Auth0 Terraform, Auth0 Deploy CLI, and Auth0 CLI.

### **Auth0 CLI** 
You can use the Auth0 CLI `aclu` command to configure ACUL. 
To learn more, read [Auth0 CLI](https://github.com/auth0/auth0-cli) on GitHub.  

### **Auth0 Terraform Provider**
You can use the Auth0 Terraformâ€™s `auth0_prompt_renderer` resource to configure ACUL. 
To learn more, read [prompt_screen_renderer](https://registry.terraform.io/providers/auth0/auth0/latest/docs/resources/prompt_screen_renderer) on Github. 

### **Auth0 Deploy CL**
You can use Auth0 Deploy CLI `prompts` resource to configure ACUL. 
To learn more, read [prompts](https://github.com/auth0/auth0-deploy-cli/blob/master/docs/resource-specific-documentation.md#prompts) on GitHub. 

## Auth0 Management API

You can use the [/v2/prompts/{prompt}/screen/{screen}/rendering](https://auth0.com/docs/api/management/v2/prompts/patch-rendering) endpoint to enable ACUL using the ACUL rendering configuration object. 

Make a `PATCH` request to the [/v2/prompts/[{prompt}]/screen/[{screen}]/rendering](https://auth0.com/docs/api/management/v2/prompts/patch-rendering) endpoint . 


Sample `PATCH` request to configure ACUL for the login-id screen:

```curl
# PATCH to /api/v2/prompts/login-id/screen/login-id/rendering
{
  "rendering_mode": "advanced",
  "context_configuration": [
    "branding.themes.default",
    "client.logo_uri",
    "client.description",
    "client.metadata.google_tracking_id",
    "screen.texts",
    "tenant.enabled_locales",
    "untrusted_data.submitted_form_data",
    "untrusted_data.authorization_params.ext-my_param"
  ],
  "head_tags": [
    {
      "tag": "script",
      "attributes": {
        "src": "https://cdn.sass.app/auth-screens/{{client.name}}.js",
        "defer": true,
        "integrity": [
          "sha256-someHash/Abc+123",
          "sha256-someHash/cDe+456"
        ]
      }
    },
    {
      "tag": "link",
      "attributes": {
        "rel": "stylesheet",
        "href": "https://cdn.sass.app/auth-screens/{{client.name}}.css"
      }
    }
  ]
}
```
