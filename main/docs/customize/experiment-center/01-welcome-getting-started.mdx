## What is Experiment Center?

Experiment Center (EC) is a native Auth0 experimentation engine. It lets you A/B test changes to your authentication experience and see the impact in your auth event logs.

Without EC, every configuration change in Auth0 is global and immediate. Enabling passkeys affects every user at once. Tuning bot detection thresholds is a blind guess. Changing a signup flow is a full deployment with no control group. EC breaks that pattern. You define your test in Auth0, traffic splits deterministically, and your existing auth events come back enriched with experiment metadata so you can analyze results in your own tools.

EC is not a general-purpose feature-flag SaaS. It is native to the Auth0 auth pipeline, which means it can reach things external platforms cannot: which authentication methods are available, how MFA is enforced, what security thresholds apply, what your login and signup flows render. This is only possible because Auth0 owns the authentication runtime.

## What's in Beta

Beta is API-first. You manage everything through the Auth0 Management API and Auth0 CLI. There is no dashboard UI in Beta.

Beta runs on test traffic only. You will run experiments on a test tenant, not on production end-user traffic. The goal is to validate that the engine works correctly and that the workflow fits your team.

**Feature flags and lifecycle management**

Feature flags define the feature being tested. A flag has a name, typed configuration parameters, and at least two variations (control and one or more treatments). Flags follow a stored lifecycle: `draft` on creation, then `active` once you have at least two variations, then `archived` when the testing program for that flag is complete. Archived is terminal; you create a new flag if you need a fresh start.

**Variations with structured parameters**

Each variation specifies overrides on the flag's baseline parameter values. Parameters are typed: `string`, `boolean`, `number`, `array`, or `object`. In Beta, parameters are always structured. There is no free-form string injection mode.

**Customer-defined segments and segment rules**

Segments are reusable groups of authentication requests that match a set of rules. Rules can match on fields available at the start of an `/authorize` transaction: `client_id`, `connection`, `organization_id`, `domain`, `device_type`, `browser`, `platform`, `user_agent`, and IP-derived geographic fields (`country`, `region`). You define your own segments; Auth0-managed segments (pre-built audience groups) come at EA.

**Experiments with percentage and targeted allocation**

An experiment binds a feature flag to a traffic allocation strategy. You choose percentage-based (for example, 90% control / 10% treatment) or segment-targeted (route a named segment to a specific variation). Both strategies support a default fallback variation.

**Deterministic assignment**

Assignment is computed at runtime via a hash of the subject identifier and the experiment ID. The same subject always gets the same variation for the life of the experiment. No assignment table is stored; the hash is deterministic, so there is nothing to look up.

For pre-auth flows (login, signup), the subject is a device identifier derived at request time. Cross-session consistency on the same device and browser is guaranteed. Cross-device linking is not required at Beta.

**Enriched tenant log events**

When an auth transaction runs through an active experiment, the resulting tenant log event is enriched with experiment metadata: `experiment_id`, `variation_id`, and `segment_id` at minimum. You pull these enriched events into your own analytics tool (Statsig, Amplitude, Snowflake, Datadog, or anything else your log stream feeds). EC does not build metrics dashboards or statistical analysis at Beta. You bring your own analytics.

Tenant logs are delivered in real time. Auth events in Snowflake are on a 12-hour ingestion cycle.

**Actions integration**

Experiment context is injected into three Actions triggers: `post_login`, `pre_user_registration`, and `post_user_registration`. Inside your Action, `event.experiment` contains the resolved context: `experiment_id`, `variation_id`, `config` (merged baseline and overrides), and `is_control`. You branch your Action logic on these values.

**ACUL (Auth0 Custom Universal Login) integration**

If you are using ACUL, you can opt a screen in to receive experiment context via the screen's `context_configuration.experimentation` setting. When opted in, the `ExperimentContext` object is available in the component's props with `experiment_id`, `variation_id`, `config`, and `is_control`. You branch your component rendering on these values.

**Page templates and partials integration**

Experiment context is available in page templates and partials for conditional rendering. The `config` values for the resolved variation are available in the template context.

**Auth0 Management API and CLI**

All Beta entities (feature flags, variations, experiments, segments) are manageable through the Auth0 Management API v2 and the Auth0 CLI. Every operation you can do via API, you can do via CLI.

**One active experiment per tenant**

At Beta, you can have one active experiment per tenant at a time. Design your test plan around this constraint. You can have any number of experiments in `draft`, `paused`, `completed`, or `archived` states; only one can be `active`.

:::note
The one-active-experiment limit is a Beta constraint. Multiple concurrent experiments with context rules return at EA.
:::

## What's not in Beta

These capabilities are scoped to EA or later. Do not plan your Beta experiment around them.

| Capability | Target phase |
|---|---|
| Inbound integrations (Statsig, Split.io, Optimizely, LaunchDarkly) | EA |
| Auth0-managed segments (pre-built audience groups) | EA |
| Multiple concurrent experiments per tenant | EA |
| Dashboard UI for experiment management | EA |
| Structured metric configuration (primary, guardrail metrics) | EA |
| Funnel analytics and in-product results visualization | EA |
| Pre-auth (pre-login fingerprint) experiments on cross-device traffic | EA |
| Terraform provider and Deploy CLI support | EA |
| MCP / agentic surface | GA |

If your use case depends on an inbound integration (you want Statsig to assign users and have Auth0 honor those assignments), this is supported at Beta via query parameter overrides on `/authorize`. You pass `experiment_id` and `variation_id` as query parameters; Auth0 honors the assignment without re-bucketing. This is the supported Beta workaround for external platform workflows.

## The Beta program

### What each side commits to

| Auth0 commits to | You commit to |
|---|---|
| Recorded kickoff Zoom (Week 1) | Attend the kickoff Zoom |
| Recorded pre-EA follow-up Zoom (Week 7) | Attend the pre-EA follow-up Zoom |
| Same-day Slack response during US business hours (best effort) | Run at least one experiment to completion |
| PM, design, and engineering access | Respond to async feedback forms (3 during the program) |
| Direct PM access for scope and roadmap questions | Provide written feedback at the end of the program |

### Milestones

Beta runs from the week of June 8 through early August 2026. You will move through six milestones with target completion windows.

| # | Milestone | What "done" means | Target |
|---|---|---|---|
| 1 | Kicked off | Kickoff Zoom recorded; eng owner and analytics owner named; first experiment plan written down | Week 1 |
| 2 | Provisioned | EC enabled on your tenant; CLI authenticates; smoke-test query confirms enrichment is reaching tenant logs | Week 1-2 |
| 3 | Experiment activated | Experiment in `active` status; validation passed; assignment event observed | Week 2-3 |
| 4 | Data flowing | Enriched events with `event.experiment` metadata reach your analytics tool; at least one event per variation observed end-to-end | Week 3-4 |
| 5 | Experiment completed | Experiment in `completed` status; outcome tagged (`promoted`, `discarded`, or `inconclusive`) | Week 5-7 |
| 6 | EA conversion captured | Pre-EA follow-up Zoom; conversion temperature recorded | Week 7-8 |

Milestone 5 is the completion gate for the Beta program. At least one partner reaching Milestone 5 is the minimum bar for EA to launch on schedule. If you hit a blocker at any milestone, follow the escalation path below.

### Support channel and response SLAs

You have a shared Slack channel with the Auth0 EC team. This is your primary support channel during Beta. Response times are:

| Priority | Definition | Response SLA |
|---|---|---|
| P0 | You cannot run your experiment; data integrity risk | Within 4 business hours |
| P1 | Blocked on a non-critical feature; data quality issue | Within 1 business day |
| P2 | Friction, feedback, suggestions that don't block | Within 2 business days |

For P0 incidents, post in your shared channel and ping the Auth0 PM directly. Engineering is engaged the same day.

Async feedback forms arrive at the end of Week 2, Week 4, and Week 6. They are short (5-7 questions) and tied to specific research questions the team is tracking. Responses shape EA scope decisions.

## Prerequisites

Before you start, you need the following.

**A test tenant with EC Beta enabled**

The Auth0 ULX team enables EC per tenant. Your Auth0 TAM or the EC PM will arrange this during provisioning (Milestone 2). You cannot self-enable EC; it is gated by a per-tenant feature flag.

**Auth0 Management API credentials**

You need a Management API token with the following scopes:

```
read:experimentation
create:experimentation
update:experimentation
delete:experimentation
```

You can provision a Machine-to-Machine application in the Auth0 dashboard and grant these scopes. See the [Auth0 Management API documentation](https://auth0.com/docs/secure/tokens/access-tokens/management-api-access-tokens) for details.

**Auth0 CLI (Beta version)**

The CLI is the primary tool for managing EC entities. Install or upgrade to the Beta version that includes EC commands:

```bash
# Install the Auth0 CLI
brew install auth0
```

:::note
EC commands require a specific Beta version of the Auth0 CLI. Your Auth0 TAM will provide the required version number and installation instructions in your shared Slack channel before Milestone 2.
:::

Authenticate the CLI against your test tenant:

```bash
auth0 login --domain your-test-tenant.us.auth0.com
```

**Access to tenant logs**

You need a way to read enriched auth events. Options:

- Auth0 dashboard: **Monitoring > Logs** on your test tenant
- Log streaming: if you already stream tenant logs to an analytics tool (Datadog, Splunk, Segment, etc.), enriched experiment metadata flows through the same stream automatically
