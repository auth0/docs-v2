# Tenant Log Enrichment Field Reference

The Experiment Center enriches Auth0 tenant log events with experiment metadata. This page documents every field added, where they appear, and how to stream the enriched data to your analytics tools.

## How enrichment works

When an auth transaction completes and an experiment was active during that transaction, the Experiment Center stamps experiment metadata onto the resulting tenant log event. No new event types are created at Beta. Existing event types (successful login, failed password, successful signup, etc.) gain an additional `experiment` object.

This is zero-integration measurement from the customer's perspective: you activate an experiment, and experiment metadata appears in your existing log stream. No instrumentation changes needed.

---

## The `event.experiment` object

Every enriched auth event includes an `experiment` field at the top level of the event object. The field is `null` (or absent) when no experiment was active during the transaction.

:::note
The field name is `event.experiment`, not `event.experimentation`. These are different: `event.experimentation` is the runtime context object injected into Actions; `event.experiment` is the field stamped on tenant log events. See DRAFT-NOTES.md item 13 (carried forward from round 1) for engineering confirmation needed.
:::

### Fields

| Field | Type | Description |
|---|---|---|
| `experiment_id` | string | The ID of the active experiment (e.g., `exp_9Km3nZ7vQwRx2yDjFs5gCp`) |
| `variation_id` | string | The ID of the variation assigned to the subject (e.g., `var_4Nq8mR2xPkTv6wYjHs3fBn`) |
| `is_control` | boolean | `true` when the assigned variation is the control group for this experiment |
| `segment_id` | string, nullable | The segment that matched (segment-allocation experiments only); `null` for percentage-allocation experiments |
| `config` | object | The merged configuration for the assigned variation. Keys are parameter names; values are `{ value: ... }` objects |

### `assignment_method` field

:::note
The `assignment_method` field (`"auto"`, `"explicit"`, `"forced"`) is in the internal `VariantResolution` object but is **not** exposed in the public `ExperimentContext` or in tenant log enrichment according to the RFD. This field is internal only. Engineering to confirm whether `assignment_method` is present in tenant log events before publishing. See DRAFT-NOTES.md item 19.
:::

The following shows a successful login event (`type: "s"`) with `event.experiment` populated:

```json
  "connection": "Username-Password-Authentication",
  "connection_id": "con_abc123",
  "client_id": "YOUR_CLIENT_ID",
  "client_name": "My App",
  "user_id": "auth0|user_abc123",
  "user_name": "user@example.com",
  "ip": "1.2.3.4",
  "user_agent": "Mozilla/5.0 ...",
  "experiment": {
    "experiment_id": "exp_9Km3nZ7vQwRx2yDjFs5gCp",
    "variation_id": "var_8Xp2qK5nWmJy9vDsHr4gAt",
    "is_control": false,
    "segment_id": null,
    "config": {
      "show_passkey_prompt": { "value": true },
      "prompt_style": { "value": "modal" }
    }
  }
}
```

For a control-group user:

```json
{
  "type": "s",
  "date": "2026-06-01T14:23:00.000Z",
  ...
  "experiment": {
    "experiment_id": "exp_9Km3nZ7vQwRx2yDjFs5gCp",
    "variation_id": "var_4Nq8mR2xPkTv6wYjHs3fBn",
    "is_control": true,
    "segment_id": null,
    "config": {
      "show_passkey_prompt": { "value": false },
      "prompt_style": { "value": "inline" }
    }
  }
}
```

For a segment-allocation experiment with a matched segment:

```json
{
  "type": "s",
  ...
  "experiment": {
    "experiment_id": "exp_3Pv9wN1qXmYz6uRcKt5aFg",
    "variation_id": "var_7Qr3sL8kDmWy4vBjNp2hEo",
    "is_control": false,
    "segment_id": "seg_5Bx4nH9rTkVj1qMcFw6gZd",
    "config": {
      "mfa_policy": { "value": "always" }
    }
  }
}
```

When no experiment was active during the transaction, the `experiment` field is absent from the event entirely (or `null`, depending on the log streaming destination format).

## Which event types get enriched

Any auth event that flows through the Universal Login path during an active experiment gets enriched. At Beta, the following event types are enriched:

| Event type code | Description |
|---|---|
| `s` | Successful login |
| `f` | Failed login |
| `fp` | Incorrect password |
| `fu` | Account blocked |
| `ss` | Successful signup |
| `fs` | Failed signup |

:::note
The complete list of event types enriched at Beta is determined by Spike 2 (tenant log event types and enrichment fields). The list above covers the core login/signup flows. Spike 2 may add additional event types. Engineering to confirm the full set before publishing. See DRAFT-NOTES.md item 20.
:::

Pre-login events and database connection script events are not enriched at Beta.

## Delivery timing

Tenant log events are delivered **immediately** (real-time). Experiment metadata is stamped before the event is written to the log.

Auth0 Snowflake data warehouse events are also enriched with the same fields, but are available on the existing **12-hour ingestion cycle** (not real-time).

---

## Streaming enriched logs to your analytics tool

The Experiment Center enriches events; your existing log streaming infrastructure carries them. If you already have an Auth0 Log Stream configured, enriched events flow to your destination automatically.

To set up log streaming if you have not already, see the [Auth0 Log Streams documentation](https://auth0.com/docs/customize/log-streams).

### Common destinations

| Destination | Notes |
|---|---|
| Datadog | Events appear in Log Management; query on `experiment.experiment_id` |
| Splunk | Events land in your configured index; use SPL to filter and join on experiment fields |
| Amazon S3 | Events arrive as JSON; load into Athena or Redshift for analysis |
| Segment | Events flow to Segment as a custom source; forward to Amplitude, Mixpanel, etc. |
| Snowflake (direct) | Available via Auth0's data warehouse enrichment on the 12-hour cycle |

## Privacy and data handling

- `config` contains only the parameter values for the assigned variation. It does not include PII.
- `experiment_id` and `variation_id` are opaque identifiers; they do not encode user information.
- `segment_id` is an opaque identifier for the matched segment; it does not encode user information.
- The subject identifier used for hash assignment (device ID or `user_id`) is **not** included in the `event.experiment` object. It is used only for hash computation and does not appear in logs.
