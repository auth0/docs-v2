When you have [multiple custom domains](/docs/customize/custom-domains/multiple-custom-domains) configured for your Auth0 tenant, you can designate one as the **default custom domain**. The default domain simplifies configuration and ensures a consistent user experience when a specific custom domain is not explicitly specified.

## What is a default custom domain?

The default custom domain is the custom domain that Auth0 uses automatically when:

* **Email communications**: Sending password reset emails, email verification links, and other Auth0-generated emails when no specific custom domain is specified
* **Authentication flows**: Processing authentication requests that don't specify a custom domain
* **Management API calls**: Handling API requests without the `auth0-custom-domain` header
* **Universal Login**: Redirecting users when no custom domain is specified in the authorization request

Without a default custom domain configured, Auth0 uses your canonical tenant domain (e.g., `YOUR_TENANT.auth0.com` or `YOUR_TENANT.REGION.auth0.com`) in these scenarios.

## Benefits of setting a default domain

Setting a default custom domain provides several advantages:

* **Simplified configuration**: Reduces the need to specify custom domains in every API call or configuration
* **Consistent branding**: Ensures users always see your brand, even when a specific domain isn't specified
* **Optional `auth0-custom-domain` header**: Makes the custom domain header optional in Management API requests
* **Fallback behavior**: Provides sensible defaults for multi-brand or multi-tenant implementations
* **Easier migration**: Simplifies transitioning from a single custom domain to multiple custom domains

## Configure a default domain

You can set a default custom domain through the Auth0 Dashboard or the Management API.

### Using the Auth0 Dashboard

To set a default custom domain in the Auth0 Dashboard:

1. Navigate to **Auth0 Dashboard** > **Branding > Custom Domains**
2. Find the verified custom domain you want to set as default in the list
3. Click the **Set as Default** button for that domain
4. Confirm the action in the dialog that appears

The domain will now show a "Default" badge in the custom domains list.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">

Only verified custom domains can be set as the default. Ensure your custom domain is fully verified and active before designating it as the default.

</Callout>

### Using the Management API

To set a default custom domain via the Management API, use the `PATCH /api/v2/custom-domains/{id}` endpoint:

<AuthCodeGroup>
```bash cURL
curl --request PATCH \
  --url 'https://{yourDomain}/api/v2/custom-domains/{customDomainId}' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
  --header 'content-type: application/json' \
  --data '{
    "is_default": true
  }'
```
```javascript Node.js
const ManagementClient = require('auth0').ManagementClient;

const management = new ManagementClient({
  domain: '{yourDomain}',
  clientId: '{yourClientId}',
  clientSecret: '{yourClientSecret}',
});

await management.customDomains.update(
  { id: '{customDomainId}' },
  { is_default: true }
);
```
```python Python
from auth0.management import Auth0

domain = '{yourDomain}'
mgmt_api_token = '{yourMgmtApiAccessToken}'

auth0 = Auth0(domain, mgmt_api_token)

auth0.custom_domains.update(
    '{customDomainId}',
    {'is_default': True}
)
```
```go Go
import (
    "context"
    "github.com/auth0/go-auth0/management"
)

m, err := management.New(
    domain,
    management.WithClientCredentials(clientID, clientSecret),
)
if err != nil {
    // Handle error
}

isDefault := true
customDomain := &management.CustomDomain{
    IsDefault: &isDefault,
}

err = m.CustomDomain.Update(context.Background(), customDomainID, customDomain)
```
```java Java
import com.auth0.client.mgmt.ManagementAPI;
import com.auth0.json.mgmt.customdomains.CustomDomain;

ManagementAPI mgmt = ManagementAPI.newBuilder(
    "{yourDomain}",
    "{yourMgmtApiAccessToken}"
).build();

CustomDomain customDomain = new CustomDomain();
customDomain.setIsDefault(true);

mgmt.customDomains()
    .update("{customDomainId}", customDomain)
    .execute();
```
</AuthCodeGroup>

## Remove default domain designation

To remove the default designation from a custom domain:

1. Set a different custom domain as the default (only one domain can be default at a time), or
2. Use the Management API to set `is_default: false` on the current default domain

When no custom domain is set as default, Auth0 will use your canonical tenant domain.

## How the default domain is used

### Email notifications

When Auth0 sends email notifications (password resets, email verification, welcome emails), it uses the custom domain in email links and references. The default domain determines which custom domain to use when:

* The `auth0-custom-domain` header is not provided
* The email template doesn't specify a custom domain
* The user operation isn't associated with a specific custom domain

To enable custom domain usage in emails:

1. Navigate to **Auth0 Dashboard** > **Branding > Custom Domains**
2. Enable the **Use Custom Domain in Emails** toggle
3. Emails will now use your default custom domain

### Authentication flows

During authentication, Auth0 uses the default custom domain when:

* An application initiates authentication without specifying a custom domain
* The authorization request doesn't include a custom domain in the URL
* A redirect occurs without explicit domain specification

The token issuer (`iss` claim) will reflect whichever domain was used for authentication. If the default domain is used, tokens will have `iss: "https://your-default-domain.com/"`.

### Management API calls

When making Management API calls, the `auth0-custom-domain` header becomes optional if you have a default domain configured:

<AuthCodeGroup>
```bash Without default domain (header required)
curl --request GET \
  --url 'https://{yourDomain}/api/v2/users' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
  --header 'auth0-custom-domain: your-custom-domain.com'
```
```bash With default domain (header optional)
curl --request GET \
  --url 'https://{yourDomain}/api/v2/users' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}'
```
</AuthCodeGroup>

If you don't provide the `auth0-custom-domain` header and a default domain is configured, Auth0 will automatically use the default domain for the operation.

## Default domain in multi-tenant scenarios

In multi-tenant implementations where different customers use different custom domains, the default domain serves as a fallback. Consider these patterns:

### Pattern 1: Default domain for internal operations

```javascript
// In your application configuration
const auth0Config = {
  domain: process.env.DEFAULT_CUSTOM_DOMAIN, // Your default domain
  clientId: process.env.AUTH0_CLIENT_ID,
  // ...
};

// For customer-specific authentication, override the domain
function getAuth0ConfigForCustomer(customer) {
  return {
    ...auth0Config,
    domain: customer.customDomain || auth0Config.domain, // Use customer domain or default
  };
}
```

### Pattern 2: Default domain for admin portal

Configure your default domain to point to your admin or internal portal, while using specific custom domains for each customer-facing application:

* Default domain: `admin.yourcompany.com` (internal admin portal)
* Customer A: `auth.customer-a.com`
* Customer B: `login.customer-b.com`

### Pattern 3: Default domain with Actions

Use [Actions](/docs/customize/actions) to route users based on custom domain:

```javascript
exports.onExecutePostLogin = async (event, api) => {
  const hostname = event.request.hostname;

  // Check if this is the default domain
  if (hostname === 'default.yourcompany.com') {
    // Apply default domain logic
    api.user.setAppMetadata('login_domain', 'default');
  } else {
    // Handle specific custom domain
    api.user.setAppMetadata('login_domain', hostname);
  }
};
```

## Best practices

When configuring a default custom domain, consider these best practices:

* **Choose a stable domain**: Select a domain that won't change frequently as your default
* **Verify before setting**: Ensure the domain is fully verified and operational before making it default
* **Document the decision**: Record why a particular domain was chosen as default for future reference
* **Test email flows**: After setting a default domain, test password reset and email verification flows
* **Monitor token issuers**: Verify that tokens have the expected `iss` claim after setting a default
* **Plan for multi-brand scenarios**: If you serve multiple brands, choose a generic or admin domain as default
* **Update integrations**: Inform your team about the default domain to ensure proper integration configuration
