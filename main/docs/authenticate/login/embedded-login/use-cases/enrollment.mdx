## What is embedded enrollment?

Embedded enrollment allows users to add credentials, authentication factors, and connected accounts directly within your application without leaving your flow. Users can enroll in new factors during onboarding, manage their authentication methods from a settings page, or link external identity providers to their account—all without redirects.

This keeps users in your app throughout the entire enrollment process, resulting in better completion rates and a seamless experience.

## When to use embedded enrollment

Use embedded enrollment when:

- **Onboarding requirement**: Your app requires users to set up authentication factors during signup or first login.
- **Settings management**: You want to let users manage their enrolled factors and authentication methods from an in-app settings page.
- **Progressive security**: You enforce security progressively—some features available after password login, premium features after MFA enrollment.
- **Security-first products**: Your app targets security-conscious users (banks, financial platforms, health apps) where MFA is expected.
- **Native passkey enrollment**: You're building iOS or Android apps and want platform-native passkey setup.
- **External data access**: You need to access user's data from external services (e.g., Google Drive, GitHub) with their explicit consent.
- **Compliance requirements**: Your organization requires MFA for all users or for access to certain data.

## Real-world scenarios

**Mobile banking apps**: During initial login, users are prompted to set up biometric authentication or OTP. They complete enrollment within the app without external redirects.

**Password manager apps**: Users enroll in native passkeys on iOS/Android directly during onboarding for passwordless login.

**Productivity SaaS**: A task management app allows users to link their Google Drive account to sync documents, with enrollment happening in-app.

**Admin panels**: Team leads can manage their authenticators (add backup codes, set up new phones) directly from their admin settings.

**Health and fitness apps**: Users enrolling in premium features are required to add a second factor for account security.

**B2B platforms**: Enterprise admins enroll in MFA during account creation and can manage factors for their team members.

## MFA Factor Enrollment

MFA factor enrollment lets users add second factors (authenticators, SMS, email, push notifications, biometric) directly within your application.

### When to use

- Onboarding requires a second factor
- Users manage factors from a settings page
- Security policies require MFA for sensitive operations
- You want to reduce friction with in-app enrollment

### How it works

<Steps>
<Step title="App detects need for MFA enrollment">
Your app logic triggers MFA enrollment—either required during signup, offered in settings, or triggered by a security policy.
</Step>

<Step title="Obtain an MFA token">
Get an `mfa_token` from Auth0. This token is returned when a user logs in and MFA enrollment is required. The method depends on how the user authenticates:

<Tabs>
<Tab title="During ROPG Login">
```bash
# Login attempt returns mfa_required with mfa_token
curl --request POST 'https://YOUR_AUTH0_DOMAIN/oauth/token' \
  --header 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'grant_type=password' \
  --data-urlencode 'username=user@example.com' \
  --data-urlencode 'password=SecurePassword123!' \
  --data-urlencode 'client_id=YOUR_CLIENT_ID' \
  --data-urlencode 'client_secret=YOUR_CLIENT_SECRET' \
  --data-urlencode 'audience=YOUR_API_IDENTIFIER' \
  --data-urlencode 'scope=openid profile email'

# Response includes mfa_token when enrollment is required
```
</Tab>

<Tab title="During Passwordless Login">
```bash
# Exchange OTP — returns mfa_required if enrollment needed
curl --request POST 'https://YOUR_AUTH0_DOMAIN/oauth/token' \
  --header 'Content-Type: application/json' \
  --data '{
    "client_id": "YOUR_CLIENT_ID",
    "client_secret": "YOUR_CLIENT_SECRET",
    "grant_type": "http://auth0.com/oauth/grant-type/passwordless/otp",
    "username": "user@example.com",
    "otp": "123456",
    "realm": "email",
    "audience": "YOUR_API_IDENTIFIER",
    "scope": "openid profile email"
  }'

# Response includes mfa_token when enrollment is required
```
</Tab>

<Tab title="Using Refresh Token">
```bash
# Refresh token exchange — returns mfa_required if policy requires enrollment
curl --request POST 'https://YOUR_AUTH0_DOMAIN/oauth/token' \
  --header 'Content-Type: application/json' \
  --data '{
    "client_id": "YOUR_CLIENT_ID",
    "client_secret": "YOUR_CLIENT_SECRET",
    "grant_type": "refresh_token",
    "refresh_token": "CURRENT_REFRESH_TOKEN"
  }'

# Response includes mfa_token when enrollment is required
```
</Tab>
</Tabs>

All three approaches return the same `mfa_required` response:
```json
{
  "error": "mfa_required",
  "error_description": "Multifactor authentication required",
  "mfa_token": "Fe26...Ha"
}
```
</Step>

<Step title="App calls /mfa/associate endpoint">
Initiate MFA factor enrollment by calling the associate endpoint with the desired factor type.

```bash
curl --request POST 'https://YOUR_AUTH0_DOMAIN/mfa/associate' \
  --header 'Authorization: Bearer MFA_TOKEN' \
  --header 'Content-Type: application/json' \
  --data '{
    "authenticator_types": ["otp"],
    "client_id": "YOUR_CLIENT_ID"
  }'
```

This returns an `mfa_token` and enrollment details (e.g., QR code for OTP).
</Step>

<Step title="App displays enrollment UI">
Your app shows the user the enrollment interface for the selected factor:
- **OTP**: Display a QR code for scanning with Google Authenticator or Microsoft Authenticator
- **SMS**: Show a phone number input and confirmation screen
- **Email**: Show a confirmation of the user's registered email
- **Push**: Instructions to approve the enrollment on a linked mobile device
- **Voice**: Phone number input for voice call verification
- **Recovery Codes**: Display single-use backup codes for account recovery
</Step>

<Step title="User completes factor enrollment">
The user scans the QR code, enters their phone number, confirms email, or completes biometric setup.
</Step>

<Step title="App verifies enrollment">
Send verification to Auth0 to complete enrollment.

```bash
curl --request POST 'https://YOUR_AUTH0_DOMAIN/mfa/challenge-response' \
  --header 'Authorization: Bearer MFA_TOKEN' \
  --header 'Content-Type: application/json' \
  --data '{
    "challenge_response": "123456"
  }'
```
</Step>

<Step title="Auth0 confirms enrollment">
Auth0 confirms the factor is enrolled and returns updated factor information. The user can now use this factor for login or step-up authentication.
</Step>
</Steps>

### Supported factors

- **OTP (One-Time Password)** — 6-digit codes from authenticator apps
- **SMS** — Verification codes via text message
- **Email** — OTP codes sent to registered email
- **Push Notifications** — Approve/deny login attempts from a mobile device
- **Voice** — Voice call with key press verification
- **Recovery Codes** — Single-use backup codes for account recovery

<Note>
For complete factor-specific documentation, see the [Manage Authenticator Factors API](/docs/secure/multi-factor-authentication/manage-mfa-auth0-apis/manage-authenticator-factors-mfa-api) reference.
</Note>

### Best practices

- **Make enrollment easy**: Minimize friction—use QR code scanning instead of manual entry for OTP enrollment
- **Provide clear instructions**: Show step-by-step instructions for each factor type
- **Offer choice**: Let users choose their preferred factor (not everyone has a smartphone for SMS)
- **Require backup factors**: Encourage users to enroll in at least two factors so they can recover their account
- **Test factor verification**: Have users verify a factor works before marking it as enrolled
- **Log enrollments**: Track when factors are enrolled or removed for audit and support purposes

**Next steps**: See [MFA for Embedded Flows](/docs/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa) for complete flow implementation.

## Passkey Enrollment

Passkey enrollment allows users to register passkeys directly in your app for web, iOS, and Android applications, enabling biometric or device-PIN-based authentication without passwords.

### When to use

- Native mobile apps (iOS/Android) or web applications targeting passwordless login
- Onboarding flows requiring biometric or device-PIN authentication
- Users upgrading from password-based to passkey-based authentication

### How it works

Passkey enrollment uses the **My Account API** (audience: `https://{yourDomain}/me/`), which provides user-facing self-service endpoints.

<Steps>
<Step title="Get an access token with enrollment scope">
Request an access token with the `create:me:authentication-methods` scope and audience set to `https://{yourDomain}/me/`.

```bash
curl --request POST 'https://YOUR_AUTH0_DOMAIN/oauth/token' \
  --header 'Content-Type: application/json' \
  --data '{
    "client_id": "YOUR_CLIENT_ID",
    "client_secret": "YOUR_CLIENT_SECRET",
    "audience": "https://YOUR_AUTH0_DOMAIN/me/",
    "grant_type": "client_credentials",
    "scope": "create:me:authentication-methods"
  }'
```

<Note>
Alternatively, if the user is already authenticated, use an authorization code flow or refresh token flow with the `create:me:authentication-methods` scope.
</Note>
</Step>

<Step title="Call the My Account API to create a passkey enrollment">
Initiate passkey enrollment by posting to the authentication methods endpoint.

```bash
curl --request POST 'https://YOUR_AUTH0_DOMAIN/me/v1/authentication-methods' \
  --header 'Authorization: Bearer ACCESS_TOKEN' \
  --header 'Content-Type: application/json' \
  --data '{
    "type": "passkey"
  }'
```

This returns enrollment details including a challenge for the client to complete.
</Step>

<Step title="App completes enrollment on device">
Your app uses the returned challenge to trigger passkey enrollment on the user's device:
- **iOS/Android**: Platform-native passkey enrollment (Face ID, Touch ID, or device PIN)
- **Web**: WebAuthn API passkey enrollment
</Step>

<Step title="App confirms enrollment">
After the user completes biometric or device-PIN setup, send the completed credential to Auth0.
</Step>
</Steps>

### Key scopes for passkey management

- **`create:me:authentication-methods`** — Enroll new authentication methods (passkeys)
- **`read:me:authentication-methods`** — View existing authentication methods
- **`update:me:authentication-methods`** — Update authentication method details
- **`delete:me:authentication-methods`** — Remove authentication methods

<Tip>
Use the My Account API scopes in your settings page to let users view, manage, and delete their passkeys directly in your app.
</Tip>

### Sharing passkeys across native and web

Choose your Relying Party ID (rpId) carefully — passkeys enrolled in your iOS, Android, and web apps are interchangeable when they share an rpId, but the rpId also defines the trust boundary. Avoid setting it to your registrable apex (e.g., `example.com`); prefer a parent domain that scopes to your authentication surfaces (e.g., `auth.example.com`). See [Using passkeys across native and web](/docs/authenticate/database-connections/passkeys/passkey-apis#using-passkeys-across-native-and-web) for the full guidance.

**Next steps**: See [My Account API](/docs/manage-users/my-account-api) and [Passkey APIs](/docs/authenticate/database-connections/passkeys/passkey-apis) for complete implementation details.

## Connected Accounts

Connected accounts allow users to link external identity providers (e.g., Google, GitHub, Microsoft) to their profile, granting your app access to their data with their explicit consent.

### When to use

- Your app needs to access user data from external services (Google Drive, GitHub repositories, etc.)
- Users want to sync data across platforms
- You're offering integrations with third-party services

### How it works

Connected accounts are managed through the **My Account API** using the Token Vault pattern.

<Steps>
<Step title="Get an access token with connected accounts scope">
Request an access token with the `create:me:connected_accounts` scope and audience set to `https://{yourDomain}/me/`.

```bash
curl --request POST 'https://YOUR_AUTH0_DOMAIN/oauth/token' \
  --header 'Content-Type: application/json' \
  --data '{
    "client_id": "YOUR_CLIENT_ID",
    "client_secret": "YOUR_CLIENT_SECRET",
    "audience": "https://YOUR_AUTH0_DOMAIN/me/",
    "grant_type": "client_credentials",
    "scope": "create:me:connected_accounts"
  }'
```
</Step>

<Step title="App redirects to provider consent screen">
Direct the user to the connection provider's OAuth consent screen to authorize access to their data.
</Step>

<Step title="App calls My Account API to link account">
After the user authorizes, call the connected accounts endpoint to link their external account.

```bash
curl --request POST 'https://YOUR_AUTH0_DOMAIN/me/v1/connected-accounts' \
  --header 'Authorization: Bearer ACCESS_TOKEN' \
  --header 'Content-Type: application/json' \
  --data '{
    "provider": "google",
    "connection_id": "YOUR_CONNECTION_ID",
    "user_id": "<external_user_id>"
  }'
```
</Step>

<Step title="User data is now accessible">
Your app can now request tokens for the linked external account and access their data on that platform.
</Step>
</Steps>

### Key scopes for connected account management

- **`create:me:connected_accounts`** — Link external accounts
- **`read:me:connected_accounts`** — View linked accounts
- **`delete:me:connected_accounts`** — Unlink external accounts

**Next steps**: See [Token Vault for Connected Accounts](/docs/secure/tokens/connected-accounts-for-token-vault) and [My Account API](/docs/manage-users/my-account-api) for complete implementation.

## Self-Service Authentication Method Management

The My Account API allows users to manage their own authentication methods from a settings page—viewing, adding, updating, and removing authenticators without requiring admin intervention.

### Common settings page operations

**List enrolled authentication methods:**
```bash
curl --request GET 'https://YOUR_AUTH0_DOMAIN/me/v1/authentication-methods' \
  --header 'Authorization: Bearer ACCESS_TOKEN'
```

**Add a new authentication method:**
```bash
curl --request POST 'https://YOUR_AUTH0_DOMAIN/me/v1/authentication-methods' \
  --header 'Authorization: Bearer ACCESS_TOKEN' \
  --header 'Content-Type: application/json' \
  --data '{
    "type": "passkey"
  }'
```

**Delete an authentication method:**
```bash
curl --request DELETE 'https://YOUR_AUTH0_DOMAIN/me/v1/authentication-methods/<method_id>' \
  --header 'Authorization: Bearer ACCESS_TOKEN'
```

### Required scopes

- **`read:me:authentication-methods`** — View user's authentication methods
- **`create:me:authentication-methods`** — Add new authentication methods
- **`update:me:authentication-methods`** — Modify existing methods
- **`delete:me:authentication-methods`** — Remove authentication methods

### Typical settings page features

1. **List of enrolled methods**: Display each method (e.g., "Passkey on iPhone", "Recovery Codes")
2. **Add new method button**: Let users enroll in additional factors
3. **Delete method**: Remove a method with confirmation
4. **View recovery codes**: Show or regenerate recovery codes
5. **Verify method**: Let users confirm a method works before relying on it

<Note>
The My Account API operates with higher rate limits than the Management API, making it ideal for user-facing self-service flows.
</Note>
