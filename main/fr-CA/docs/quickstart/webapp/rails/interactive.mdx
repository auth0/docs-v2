    <Section id={sections[0].id} title={sections[0].title} stepNumber="1">
      Pour utiliser les services Auth0, vous devez d’abord configurer une application dans l’Auth0 Dashboard. L’application Auth0 est
      l’endroit où vous définirez le mode de fonctionnement de l’authentification pour le projet que vous développez.

      ### Configurer une application

      Utilisez le sélecteur interactif pour créer une nouvelle application Auth0 ou sélectionner une application existante correspondant
      au projet que vous souhaitez intégrer. Chaque application dans Auth0 se voit attribuer un ID client alphanumérique unique
      que le code de votre application utilisera pour appeler les API Auth0 au moyen du SDK.

      Tous les paramètres que vous configurez à l’aide de ce guide de démarrage rapide seront automatiquement mis à jour pour votre application dans l’[Auth0 Dashboard](https://manage.auth0.com/#/), où vous
      pourrez gérer vos applications par la suite.

      Si vous préférez explorer une configuration complète, vous pouvez plutôt consulter un exemple d’application.

      ### Configurer les URL de rappel

      Une URL de rappel est une URL de votre application vers laquelle vous souhaitez qu’Auth0 redirige les utilisateurs après leur
      authentification. Si elle n’est pas définie, les utilisateurs ne seront pas redirigés vers votre application après leur connexion.

      <Info>
        Si vous suivez notre projet d’exemple, définissez-la à
        `http://localhost:3000/auth/auth0/callback`.
      </Info>

      ### Configurer les URL de déconnexion

      Une URL de déconnexion est une URL de votre application vers laquelle vous souhaitez qu’Auth0 redirige les utilisateurs après leur
      déconnexion. Si elle n’est pas définie, les utilisateurs ne pourront pas se déconnecter de votre application et recevront une erreur.

      <Info>
        Si vous suivez notre projet d’exemple, définissez-la à `http://localhost:3000`.
      </Info>

      ### Configurer les origines Web autorisées

      Une origine Web autorisée est une URL à laquelle vous souhaitez autoriser l’accès à votre flux d’authentification. Elle doit
      contenir l’URL de votre projet. Si elle n’est pas correctement définie, votre projet ne pourra pas actualiser silencieusement
      les jetons d’authentification; vos utilisateurs seront donc déconnectés lors de leur prochaine visite dans votre application ou lorsqu’ils actualiseront une
      page.

      <Info>
        Si vous suivez notre projet d’exemple, définissez-la à `http://localhost:3000`.
      </Info>
    </Section>

    <Section id={sections[1].id} title={sections[1].title} stepNumber="2">
      Utilisez `omniauth-auth0`, une [stratégie OmniAuth personnalisée](https://github.com/intridea/omniauth#omniauth-standardized-multi-provider-authentication), pour gérer le processus d’authentification.

      Ajoutez les dépendances suivantes à votre `Gemfile` :

      ```gem lines
      gem 'omniauth-auth0', '~> 3.0'
      gem 'omniauth-rails_csrf_protection', '~> 1.0' # empêche les requêtes d'authentification falsifiées
      ```

      Une fois les gems ajoutés, installez-les avec `bundle install`.
    </Section>

    <Section id={sections[2].id} title={sections[2].title} stepNumber="3">
      Créez un fichier de configuration `./config/auth0.yml` pour indiquer les valeurs de votre domaine Auth0, de votre ID client et du secret de l’application, qui se trouvent dans l’Auth0 Dashboard, sous **Settings** de l’application.
    </Section>

    <Section id={sections[3].id} title={sections[3].title} stepNumber="4">
      Créez le fichier d’initialisation suivant `./config/initializers/auth0.rb`, puis [configurez](https://github.com/auth0/omniauth-auth0/blob/master/EXAMPLES.md#send-additional-authentication-parameters) le middleware **OmniAuth** à l’aide du fichier de configuration
      créé à l’étape précédente.

      Assurez-vous que `callback_path` correspond à la valeur indiquée dans le paramètre &quot;Allowed Callback URLs&quot; de votre
      application Auth0.
    </Section>

    <Section id={sections[4].id} title={sections[4].title} stepNumber="5">
      Créez un contrôleur Auth0 pour gérer le rappel d’authentification, l’action `logout` et les méthodes permettant de
      construire l’URL de déconnexion.

      Exécutez la commande :
      `rails generate controller auth0 callback failure logout --skip-assets --skip-helper --skip-routes --skip-template-engine`.

      Dans la méthode callback, assignez le hachage des renseignements de l’utilisateur — renvoyé sous la forme de
      `request.env['omniauth.auth']` — à la session en cours.

      Pour configurer la déconnexion, effacez tous les objets stockés dans la session en appelant la méthode `reset_session`
      dans l’action `logout`. Redirigez ensuite vers le point de terminaison de déconnexion d’Auth0. Pour en savoir plus sur
      `reset_session`, consultez la [documentation d’ActionController pour Ruby on Rails](http://api.rubyonrails.org/classes/ActionController/Base.html#M000668).
    </Section>

    <Section id={sections[5].id} title={sections[5].title} stepNumber="6">
      Ajoutez ces routes à votre fichier `./config/routes.rb`.

      Ces routes doivent être configurées pour que Rails sache comment acheminer les différentes URL de rappel d’Auth0 vers le contrôleur Auth0 que vous
      avez créé à l’étape précédente.

      <Info>
        ##### Vérification

        Exécutez votre application pour vérifier qu’elle fonctionne toujours comme prévu et que vous ne recevez aucune erreur
        liée à Auth0.
      </Info>
    </Section>

    <Section id={sections[6].id} title={sections[6].title} stepNumber="7">
      Un utilisateur peut maintenant ouvrir une session dans votre application en accédant au point de terminaison `/auth/auth0`.

      <Warning>
        Pour [éviter les requêtes d’authentification falsifiées](https://github.com/cookpad/omniauth-rails_csrf_protection), utilisez les méthodes utilitaires `link_to` ou
        `button_to` avec la méthode `:post`.
      </Warning>

      ```ruby lines
      <!-- Placez un bouton de connexion n'importe où dans votre application -->
      <%= button_to 'Login', '/auth/auth0', method: :post %>
      ```

      <Info>
        ##### Point de contrôle

        Ajoutez à votre application un bouton qui redirige l’utilisateur vers le point de terminaison `/auth/auth0` lorsqu’il
        est sélectionné. Notez que vous êtes redirigé vers Auth0 pour vous connecter, puis de nouveau vers votre application une fois
        l’authentification réussie.
      </Info>
    </Section>

    <Section id={sections[7].id} title={sections[7].title} stepNumber="8">
      Maintenant que vous pouvez vous connecter à votre application Rails, il vous faut un
      moyen de vous déconnecter. Déconnectez un utilisateur en le redirigeant vers l’action `auth/logout`, qui le redirige
      vers le point de terminaison de déconnexion d’Auth0.

      <Info>
        Pour tester cela après l’étape précédente, vous devrez peut-être vider votre session, puis rediriger l’utilisateur
        vers le point de terminaison de déconnexion d’Auth0.
      </Info>

      ```ruby lines
      <!-- Placez un bouton de déconnexion n'importe où dans votre application -->
      <%= button_to 'Logout', 'auth/logout', method: :get %>
      ```

      <Info>
        ##### Point de contrôle

        Ajoutez à votre application un bouton qui redirige l’utilisateur vers le point de terminaison `/auth/logout` lorsqu’il
        est sélectionné. Vérifiez que vous êtes redirigé vers Auth0, puis rapidement vers votre application, et que vous n’êtes
        plus connecté.
      </Info>
    </Section>

    <Section id={sections[8].id} title={sections[8].title} stepNumber="9">
      Pour afficher le profil de l’utilisateur, votre application doit définir une route protégée. Vous pouvez utiliser un [Concern](https://guides.rubyonrails.org/getting_started.html#using-concerns) pour contrôler l’accès aux routes pouvant être partagées entre plusieurs
      contrôleurs. Le concern doit rediriger automatiquement vers Auth0 lorsque l’utilisateur n’est pas authentifié. Sinon, le
      concern doit renvoyer le profil de l’utilisateur courant.

      Une fois que vous avez un Concern, incluez-le dans tout contrôleur qui exige qu’un utilisateur soit connecté. Vous pourrez ensuite accéder à
      l’utilisateur à partir de la session `session[:userinfo]`, comme dans l’exemple suivant :

      ```ruby lines
      class DashboardController < ApplicationController
      include Secured
      def show
      @user = session[:userinfo]

      end
      end
      ```

      Une fois l’utilisateur récupéré depuis la session, utilisez-le pour afficher des informations dans votre interface utilisateur :

      ```html lines
      <div>
      <p>Normalized User Profile:<%= JSON.pretty_generate(@user[:info])%></p>
      <p>Full User Profile:<%= JSON.pretty_generate(@user[:extra][:raw_info])%></p>
      </div>
      ```

      <Info>
        ##### Point de contrôle

        Ajoutez le concern `Secured` à votre application, puis incluez-le dans le contrôleur qui exige qu’un
        utilisateur soit authentifié pour y accéder. Vérifiez qu’un utilisateur authentifié a accès aux actions de ce
        contrôleur et que les utilisateurs non authentifiés sont redirigés vers Auth0 pour s’authentifier.
      </Info>
    </Section>
