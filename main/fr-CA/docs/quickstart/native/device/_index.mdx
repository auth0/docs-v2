    <Section id={sections[4].id} title={sections[4].title} stepNumber="5" isSingleColumn>
      Pendant que votre application sur l’appareil attend que l’utilisateur l’active, elle doit appeler périodiquement
      le [point de terminaison POST /oauth/token de l’API d’authentification](https://auth0.com/docs/api/authentication#-post-oauth-token-)
      et traiter la réponse de manière appropriée.

      <Info>
        Assurez-vous que votre application sur l’appareil attend la durée indiquée par `interval` (en secondes) ou jusqu’à
        ce qu’elle reçoive une réponse positive, selon la période la plus longue, afin d’éviter les problèmes de latence du réseau.
      </Info>

      ```bash
      curl --request POST \ 
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=urn:ietf:params:oauth:grant-type:device_code \
        --data device_code=AUTH0_SCOPES \
        --data 'client_id={yourClientId}'
      ```

    <Section id={sections[6].id} title={sections[6].title} stepNumber="7" isSingleColumn>
      Une fois que l’utilisateur a autorisé l’application sur l’appareil, celle-ci reçoit une réponse HTTP 200 et la charge utile suivante :

      ```json lines
      {
        "access_token": "eyJz93a...k4laUWw",
        "refresh_token": "GEbRxBN...edjnXbL",
        "id_token": "eyJ0XAi...4faeEoQ",
        "token_type": "Bearer",
        "expires_in": 86400
      }
      ```

      <Info>
        Vous devriez valider vos jetons avant de les enregistrer. Pour savoir comment faire, consultez Valider les jetons d’accès et Valider les ID Tokens.
      </Info>

      Les jetons d’accès servent à appeler le [point de terminaison Get User Info](https://auth0.com/docs/api/authentication#get-user-info) de l’API d’authentification (si votre
      application pour appareil a demandé le scope `openid`) ou l’API spécifiée par le paramètre `audience`.
      Si vous appelez votre propre API, votre application pour appareil doit vérifier le jeton d’accès avant de l’utiliser.

      Les jetons d’identité contiennent des informations sur l’utilisateur qui doivent être [décodées et extraites](/fr-CA/docs/secure/tokens/id-tokens#id-token-payload). L’API d’authentification
      ne renvoie un `id_token` que si votre application pour appareil a demandé le scope `openid`.

      Les jetons d’actualisation servent à obtenir un nouveau jeton d’accès ou
      un ID Token après l’expiration du précédent. L’API d’authentification ne renvoie un `refresh_token` que si
      l’option **Allow Offline Access** est activée pour l’API spécifiée par le paramètre `audience`,
      et si votre application pour appareil a demandé le scope `offline_access`.

    <Section id={sections[7].id} title={sections[7].title} stepNumber="8" isSingleColumn>
      Pour appeler votre API, votre application sur l’appareil doit transmettre le jeton d’accès en tant que jeton Bearer dans l’en-tête
      `Authorization` de votre requête HTTP.

      ```bash lines
      curl --request GET \
        --url https://myapi.com/api \
        --header 'authorization: Bearer AUTH0_API_ACCESS_TOKEN' \
        --header 'content-type: application/json'
      ```

    <Section id={sections[8].id} title={sections[8].title} stepNumber="9" isSingleColumn>
      Pour obtenir un nouveau jeton d’accès pour un utilisateur, votre application sur l’appareil peut appeler le [point de terminaison POST /oauth/token de l’API d’authentification](https://auth0.com/docs/api/authentication#-post-oauth-token-) avec le
      paramètre `refresh_token`.

      ```bash lines
      curl --request POST \
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=refresh_token \
        --data 'client_id={yourClientId}' \
        --data 'client_secret={yourClientSecret}' \
        --data refresh_token=AUTH0_REFRESH_TOKEN
      ```

      Si la requête aboutit, l’application de votre appareil reçoit une réponse HTTP 200 avec la charge utile suivante :

      ```json lines
      {
        "access_token": "eyJ...MoQ",
        "expires_in": 86400,
        "scope": "openid offline_access",
        "id_token": "eyJ...0NE",
        "token_type": "Bearer"
      }
      ```

      Pour en savoir plus sur les Jetons d’actualisation, consultez [Jetons d’actualisation](https://auth0.com/docs/secure/tokens/refresh-tokens).

      Les journaux du locataire sont créés pour toute interaction qui se produit
      et peuvent être utilisés pour résoudre les problèmes.

      ### Réponses de jeton

      Pendant que vous attendez que l’utilisateur autorise l’appareil, vous pourriez recevoir plusieurs réponses HTTP 4xx.

      #### Autorisation en attente

      Vous verrez cette erreur pendant que vous attendez que l’utilisateur agisse. Continuez l’interrogation en utilisant l’`interval`
      suggéré récupéré à l’étape précédente de ce tutoriel.

      ```json lines
      `HTTP 403`
      {
      "error": "authorization_pending",
      "error_description": "..."
      }
      ```

      #### Ralentissez

      Vous envoyez des requêtes trop rapidement. Ralentissez et utilisez l’intervalle suggéré récupéré à l’étape précédente de ce
      tutoriel. Pour éviter de recevoir cette erreur en raison de la latence réseau, vous devriez commencer à compter chaque intervalle après
      avoir reçu la réponse à la dernière requête d’interrogation.

      ```json lines
      `HTTP 429`
      {
      "error": "slow_down",
      "error_description": "..."
      }
      ```

      #### Jeton expiré

      L&#39;utilisateur n&#39;a pas autorisé l&#39;appareil assez rapidement, donc le `device_code` a expiré. Votre
      application doit informer l&#39;utilisateur que le flux a expiré et l&#39;inviter à le relancer.

      <Info>
        L&#39;erreur `expired_token` est renvoyée exactement une seule fois. Après cela, le point de terminaison renvoie l&#39;erreur
        `invalid_grant`.
      </Info>

      ```json lines
      `HTTP 403`
      {
      "error": "expired_token",
      "error_description": "..."
      }
      ```

      #### Accès refusé

      Si l’accès est refusé, vous obtenez :

      ```json lines
      `HTTP 403`
      {
      "error": "access_denied",
      "error_description": "..."
      }
      ```

      Cela peut se produire pour diverses raisons, notamment :

      * L’utilisateur a refusé d’autoriser l’appareil.
      * Le serveur d’autorisation a refusé la transaction.
      * Une Action configurée a refusé l’accès.

    <Section id={sections[11].id} title={sections[11].title} stepNumber="12" isSingleColumn>
      Pour utiliser le flux d’autorisation de l’appareil, une application d’appareil doit :

      * Prendre en charge l’indication du nom du serveur (SNI)
      * Être une application native
      * Avoir la [méthode d’authentification](/fr-CA/docs/secure/application-credentials#application-authentication-methods) définie sur **Aucune**
      * Être [conforme à OIDC](/fr-CA/docs/get-started/applications/application-settings#oauth)
      * Ne pas avoir été créée au moyen de l’enregistrement dynamique de l’application

      De plus, le flux d’autorisation de l’appareil ne permet pas :

      * Les connexions sociales qui utilisent des clés de développeur Auth0, sauf si vous utilisez la nouvelle expérience Universal Login
      * D’accéder aux paramètres de chaîne de requête à partir d’une page de connexion hébergée ou d’Actions.
