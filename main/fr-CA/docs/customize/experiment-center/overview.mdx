Auth0 Experiment Center est un moteur d’expérimentation natif d’Auth0. Il vous permet d’effectuer des tests A/B sur des modifications apportées à votre expérience d’[authentification](/fr-CA/docs/authenticate/login/auth0-universal-login) et d’en voir l’impact dans vos [logs](/fr-CA/docs/deploy-monitor/logs) d’événements d’authentification.

Avec Experiment Center, vous définissez votre test A/B dans Auth0, le trafic est réparti de manière déterministe, et vos événements d’authentification existants sont enrichis de métadonnées d’expérimentation afin que vous puissiez analyser les résultats dans vos propres outils.

<div id="how-it-works">
  ## Fonctionnement
</div>

Experiment Center vous permet d’exécuter des tests A/B contrôlés sur votre pipeline d’authentification Auth0. Au lieu de déployer une modification auprès de tous les utilisateurs en même temps, vous exposez un nouveau comportement à un pourcentage contrôlé du trafic, mesurez le résultat au moyen d’événements d’authentification enrichis, puis adoptez la variante gagnante lorsque vous êtes prêt.

Experiment Center s’articule autour de trois entités :

* **Experiment** : définit la répartition du trafic et la période d’exécution du test.
* **Feature flag** : définit ce que vous testez et les variations possibles.
* **Segment** : définit un ensemble de règles pour orienter le trafic vers des variations précises.

<Warning>
  Pendant la phase bêta, Experiment Center fonctionne uniquement sur les locataires de développement. Les locataires de production ne sont pas pris en charge.
</Warning>

<div id="experiment">
  ## Expérience
</div>

Une expérience est un mécanisme de mesure associé à un feature flag. Elle définit :

* Quel feature flag est testé
* Comment le trafic est réparti entre les variantes
* Quand le test est exécuté

Chaque expérience référence exactement un feature flag.

<div id="experiment-lifecycle">
  ### Cycle de vie de l’expérience
</div>

Les expériences ont cinq états :

| Statut      | Signification                                                                                                                  |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `draft`     | Créée, mais non démarrée. Peut être testée sans risque à l’aide de surcharges de paramètres de requête.                        |
| `active`    | En cours. L’attribution des variantes et l’injection du contexte sont actives pour toutes les transactions d’authentification. |
| `paused`    | Temporairement suspendue. Aucune nouvelle attribution. Les sessions en cours conservent la variante qui leur a été attribuée.  |
| `completed` | Terminée. Aucune nouvelle attribution. La configuration est conservée à titre de référence pendant la promotion manuelle.      |
| `archived`  | Supprimée logiquement. Masquée dans les vues de liste par défaut. Les données analytiques sont conservées.                     |

<div id="allocation-strategies">
  ### Stratégies d’allocation
</div>

Une expérience utilise l’une des deux stratégies d’allocation suivantes :

**Basée sur un pourcentage :** Le trafic est réparti entre les variations selon leur poids. La somme de tous les poids doit être égale à 100. Un poids de 0 est valide (la variation fait partie de la définition de l’expérience, mais ne reçoit aucun trafic).

**Basée sur les segments (ciblée) :** Le trafic est acheminé vers les variations en fonction de l’appartenance à un segment. Les segments sont évalués par ordre de priorité. Le premier segment correspondant est retenu. Si aucun segment ne correspond, l’allocation `is_fallback` reçoit la requête.

Pour en savoir plus sur l’entité d’expérience, consultez [Détails des entités](/fr-CA/docs/customize/experiment-center/entities#experiment).

<div id="experiment-context">
  ### Contexte de l’expérience
</div>

Lorsqu’une expérience est active et qu’une variante est attribuée, Experiment Center injecte un objet `ExperimentContext` dans vos surfaces d’exécution.

Les écrans [ACUL](/fr-CA/docs/customize/login-pages/advanced-customizations#advanced-customizations-for-universal-login-acul) le reçoivent uniquement si vous configurez l’écran à cet effet au moyen de `context_configuration`.
Pour en savoir plus, consultez le [guide d’intégration ACUL](/fr-CA/docs/customize/experiment-center/integrations/acul-integration-guide).

Les [Actions](/fr-CA/docs/customize/actions/actions-overview) et les [modèles de page](/fr-CA/docs/customize/login-pages/universal-login/customize-templates#customize-universal-login-page-templates) le reçoivent automatiquement chaque fois qu’une expérience est active. Voici la structure de l’objet :

```typescript
{
  experiment_id: string;   // L'ID de l'expérience active
  variation_id: string;    // L'ID de la variation assignée
  config: {                // Configuration fusionnée (référence + overrides)
    [paramName]: { value: unknown }
  };
  is_control: boolean;     // Vrai si c'est la variation de contrôle
}
```

Le champ `config` contient l’intégralité de la configuration fusionnée pour la variation attribuée. Chaque paramètre défini pour l’indicateur de fonctionnalité a toujours une valeur. Vous n’avez pas besoin d’écrire de logique de secours.

<div id="assignment">
  ### Affectation
</div>

Une affectation correspond à la façon dont un utilisateur est dirigé vers une variation précise pendant une transaction d’authentification. Les affectations sont déterministes et persistantes : le même utilisateur voit toujours la même variation pour la même expérience sur le même appareil, de sorte que son expérience reste stable d’une connexion à l’autre.

* **Affectation par pourcentage** répartit le trafic entre les variations selon leur pondération. Un sujet donné est toujours affecté à la même variation.
* **Affectation par segment** évalue les propriétés de la requête selon vos règles de segment, dans l’ordre de priorité. Le premier segment correspondant détermine la variation, et une requête qui correspond au même segment aboutit toujours à la même variation.

Les résultats d’affectation sont consignés dans les logs de votre locataire sous `details.experiment` ; ils ne sont pas exposés par une API distincte.

Pour en savoir plus sur l’entité d’affectation, consultez [Détails des entités](/fr-CA/docs/customize/experiment-center/entities#assignment).

<div id="feature-flag">
  ## Indicateur de fonctionnalité
</div>

Un indicateur de fonctionnalité est l’élément de contrôle de ce qui est testé. Il contient :

* Une **configuration de base** : des paramètres typés et leurs valeurs par défaut
* Une ou plusieurs **variantes** : des configurations alternatives qui diffèrent de la configuration de base

Les indicateurs de fonctionnalité sont associés à un locataire et sont réutilisables. Le même indicateur peut être utilisé par plusieurs expérimentations au fil du temps (par exemple, un test au T1 puis un ajustement au T2 pour la même fonctionnalité).

<div id="feature-flag-lifecycle">
  ### Cycle de vie du feature flag
</div>

Les feature flags ont un cycle de vie enregistré comportant trois états :

| Statut     | Signification                                                                                       |
| ---------- | --------------------------------------------------------------------------------------------------- |
| `draft`    | Créé, mais pas encore actif. Ne peut pas être référencé par une expérience active.                  |
| `active`   | Prêt à l’emploi. Requis avant qu’une expérience qui référence ce flag puisse être activée.          |
| `archived` | État final. Aucune nouvelle expérience ne peut référencer ce flag. Créez un nouveau flag au besoin. |

Pour en savoir plus sur l’entité feature flag, consultez [Détails des entités](/fr-CA/docs/customize/experiment-center/entities#feature-flag).

<div id="variation">
  ### Variation
</div>

Une variation est une version de l’expérience définie dans un feature flag. Elle précise quels paramètres de configuration s’écartent de la référence et dans quelle mesure.

* La **variation témoin** correspond à la référence; elle ne comporte aucune redéfinition (aucun paramètre n’est modifié par rapport aux valeurs par défaut du flag).
* Les **variations de traitement** définissent chacune une ou plusieurs redéfinitions de paramètres.

Les variations n’ont pas de marqueur `is_control`. Le fait qu’une variation soit le témoin statistique d’une expérience donnée est défini dans l’allocation, et non dans la variation. Une même variation peut être le témoin dans une expérience et un traitement dans une autre.

Pour en savoir plus sur l’entité variation, consultez [Détails des entités](/fr-CA/docs/customize/experiment-center/entities#variation).

<div id="segment">
  ## Segment
</div>

Un segment est un groupe nommé de requêtes d’authentification qui correspondent à un ensemble de règles. Vous utilisez les segments dans des expériences d’allocation ciblée pour acheminer des cohortes de trafic précises vers des variantes précises.

Les segments sont définis au niveau du locataire et peuvent être réutilisés d’une expérience à l’autre.

Pour en savoir plus sur l’entité de segment, consultez [Détails des entités](/fr-CA/docs/customize/experiment-center/entities#segment).

<div id="limitations">
  ## Limites
</div>

Les limites suivantes s’appliquent pendant la version bêta :

* **Un seul experiment actif par locataire.** Vous pouvez avoir plusieurs experiments dans les états `draft`, `paused` ou `completed`, mais un seul peut être `active` à la fois.
* **Paramètres structurés uniquement.** Les paramètres de feature flag utilisent une structure clé/type/valeur.
* **Trois déclencheurs Actions.** Le contexte d’experiment est disponible dans [`post_login`](/fr-CA/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger), [`pre_user_registration`](/fr-CA/docs/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger) et [`post_user_registration`](/fr-CA/docs/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger).
* **Trafic de test uniquement.** La version bêta s’exécute sur des locataires de développement avec le trafic que vous générez. Les données des utilisateurs finaux en production ne transitent pas par la version bêta.
* **Promotion manuelle.** Lorsqu’un experiment est terminé, vous appliquez manuellement à votre locataire la configuration de la variante gagnante.
