Avec la [Management API](/fr-CA/docs/api/management/v2) d’Auth0, vous pouvez définir les détails des entités dans Experiment Center, y compris les propriétés, les états du cycle de vie et les règles de validation.

<Warning>
  Durant la phase bêta, Experiment Center est offert uniquement pour les locataires de développement. Les locataires de production ne sont pas pris en charge.
</Warning>

<div id="experiment">
  ## Expérimentation
</div>

<div id="experiment-lifecycle">
  ### Cycle de vie des expériences
</div>

Les expériences ont cinq états :

| **Statut**  | **Signification**                                                                                                               |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `draft`     | Créée, mais non active. Peut être testée sans risque à l’aide de surcharges de paramètres de requête.                           |
| `active`    | Active. L’attribution des variantes et l’injection de contexte sont en vigueur pour toutes les transactions d’authentification. |
| `paused`    | Temporairement suspendue. Aucune nouvelle attribution. Les sessions en cours conservent la variante qui leur a été attribuée.   |
| `completed` | Terminée. Aucune nouvelle attribution. La configuration est conservée à titre de référence pendant la promotion manuelle.       |
| `archived`  | Supprimée de façon non définitive. Masquée des vues de liste par défaut. Les données analytiques sont conservées.               |

Pour changer l’état du cycle de vie, utilisez le point de terminaison de la Management API [`/api/v2/experimentation/experiments/{id}/status`](/fr-CA/docs/api/management/v2/experimentation/update-experiment-status).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Il n’est pas possible de passer de `completed → active`. Si vous devez exécuter le même test de nouveau, créez une nouvelle expérience.
</Callout>

<div id="the-is_valid-gate">
  ### La validation `is_valid`
</div>

Avant qu’une expérience puisse être activée, elle doit réussir une vérification de préparation.

Experiment Center stocke un booléen `is_valid` pour chaque expérience et le recalcule à chaque modification. Vous obtenez ainsi un retour immédiat pendant la configuration.

Vous pouvez aussi déterminer explicitement l’état de préparation à l’aide du point de terminaison de la Management API [`/api/v2/experimentation/experiments/{id}/validate`](/fr-CA/docs/api/management/v2/experimentation/validate-experiment).

Cela renvoie `is_valid: true/false` et un tableau `errors` qui énumère tous les éléments bloquants. La même vérification s’exécute automatiquement lorsque vous tentez une activation via le point de terminaison de statut.

**Règles de validation appliquées lors de l’activation :**

* Le feature flag référencé a le statut `active`
* Au moins une allocation existe
* Toutes les valeurs `variation_id` des allocations appartiennent au feature flag de l’expérience
* Une seule allocation a `is_control: true`
* La somme des pondérations des allocations est de 100 (stratégie en pourcentage)
* Une seule allocation a `is_fallback: true` (stratégie par segment)
* Aucune autre expérience n’est actuellement active pour le locataire (bêta seulement)

<div id="assignment">
  ### Attribution
</div>

**Surcharges des paramètres de requête**

Vous pouvez imposer une attribution précise pour n’importe quelle requête `/authorize` en transmettant des paramètres de requête :

| Paramètre       | Description                                               |
| --------------- | --------------------------------------------------------- |
| `experiment_id` | Inscrire à une expérience précise                         |
| `variation_id`  | Imposer une variation précise (nécessite `experiment_id`) |
| `segment_id`    | Imposer un segment précis (nécessite `experiment_id`)     |

Cela fonctionne pour les expériences, quel que soit leur statut, y compris `draft`. Utilisez cette option pendant le développement pour vérifier que les deux variations s’affichent correctement avant l’activation.

<div id="feature-flag">
  ## Drapeau de fonctionnalité
</div>

<div id="feature-flag-lifecycle">
  ### Cycle de vie d’un indicateur de fonctionnalité
</div>

Les indicateurs de fonctionnalité ont un cycle de vie défini comportant trois états :

| **Statut** | **Signification**                                                                                      |
| ---------- | ------------------------------------------------------------------------------------------------------ |
| `draft`    | Créé, mais pas encore actif. Ne peut pas être utilisé par une expérience active.                       |
| `active`   | Prêt à être utilisé. Requis avant qu’une expérience utilisant cet indicateur puisse être activée.      |
| `archived` | État final. Aucune nouvelle expérience ne peut utiliser cet indicateur. Créez-en un nouveau au besoin. |

Pour passer d’un statut à un autre, utilisez le point de terminaison de l’Auth0 Management API [`/api/v2/experimentation/feature-flags/{id}/status`](/fr-CA/docs/api/management/v2/experimentation/update-feature-flag-status).

**Condition préalable à l’activation :** Un indicateur de fonctionnalité ne peut pas passer à `active` tant qu’il ne comporte pas au moins deux variantes. Cela garantit que chaque indicateur actif comporte au minimum un témoin et un traitement avant sa mise en ligne.

```text wrap lines
draft → active   (requires ≥2 variations)
draft → archived
active → archived (terminal; no way back)
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La transition de `archived → active` n’est pas autorisée. Si vous archivez un flag de fonctionnalité et devez effectuer un autre test, créez-en un nouveau.
</Callout>

<div id="variation">
  ### Variante
</div>

**Paramètres et remplacements**

Les paramètres du feature flag définissent les options de configuration de votre expérimentation. Chaque paramètre comprend :

* Un nom (par exemple, `show_passkey_prompt`)
* Un type : `string`, `boolean`, `number`, `array` ou `object`
* Une valeur par défaut (la valeur de référence)

Le champ `overrides` d’une variante précise uniquement les paramètres qui diffèrent. À l’exécution, Experiment Center fusionne la valeur de référence du flag avec les remplacements de la variante et transmet l’objet `config` complet résultant à ACUL, à Actions et aux modèles de page. Chaque paramètre a toujours une valeur dans `config` ; vous n’avez donc jamais besoin d’écrire une logique de secours.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les paramètres utilisent uniquement le **mode structuré**. Chaque paramètre possède une clé nommée et une valeur typée.
</Callout>

<div id="segment">
  ## Segment
</div>

<div id="segment-rules">
  ### Règles de segment
</div>

Le champ `rules` d’un segment est un tableau d’objets de règle. Une requête correspond au segment si **au moins une** des règles du tableau correspond.

* `match` : toutes les conditions doivent être remplies (logique AND)
* `not_match` : toutes les conditions doivent être remplies (logique NOT)

Chaque condition compare un attribut à une valeur au moyen d’un opérateur : `contains`, `starts_with`, `ends_with`, `exists`.

<div id="available-condition-attributes">
  ### Attributs de condition disponibles
</div>

Les segments peuvent uniquement utiliser les attributs disponibles au début de la transaction `/authorize` :

| **Catégorie**                 | **Attributs**                                      |
| ----------------------------- | -------------------------------------------------- |
| Application                   | `client_id`                                        |
| Connexion                     | `connection`, `connection_type`                    |
| Organisation                  | `organization_id`                                  |
| Domaine                       | `domain`                                           |
| Appareil et navigateur        | `device_type`, `browser`, `platform`, `user_agent` |
| Géographique (dérivé de l’IP) | `country`, `region`                                |
