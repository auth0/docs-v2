Pour de meilleurs résultats lors de la mise en œuvre du <Tooltip tip="Single Sign-On (SSO) : service qui, après qu’un utilisateur ouvre une session dans une application, lui ouvre automatiquement une session dans d’autres applications." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=SSO">SSO</Tooltip> natif vers le Web, Auth0 recommande de suivre les bonnes pratiques ci-dessous. Ces recommandations peuvent aider à préserver l’intégrité de la session :

<div id="use-post-login-actions-to-limit-session-lifetime">
  ## Utilisez les Actions post-login pour limiter la durée de vie des sessions
</div>

Utilisez les déclencheurs d’Action [`post-login`](/fr-CA/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) pour vous assurer que les sessions Web créées au moyen de la fonctionnalité SSO natif vers le Web sont correctement limitées dans le temps et expirent rapidement lorsqu’elles sont inactives. 
Vous pouvez utiliser les Actions `post-login` pour détecter quand une session est lancée au moyen d’un `session_transfer_token` et appliquer des délais d’inactivité et des délais absolus plus courts :

```swift lines
exports.onExecutePostLogin = async (event, api) => {
  if (event.session_transfer_token) {
    const now = Date.now();

    // Limiter la session à 30 minutes au total
    api.session.setExpiresAt(now + 30 * 60 * 1000);

    // Définir le délai d'inactivité à 15 minutes
    api.session.setIdleExpiresAt(now + 15 * 60 * 1000);
  }
};
```

<div id="bind-session_transfer_token-to-the-device-or-ip-address">
  ## Associer le session_transfer_token à l’appareil ou à l’adresse IP
</div>

Pour réduire le risque de rejeu de jeton si un jeton est divulgué, consigné ou intercepté, associez toujours le [`session_transfer_token`](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id) à l’environnement d’origine au moyen de `enforce_device_binding`.

<div id="use-secure-cookies-over-query-parameters">
  ## Utilisez des témoins sécurisés plutôt que des paramètres de requête
</div>

Envoyez le `session_transfer_token` à l’application Web au moyen d’un témoin sécurisé, HTTPOnly, associé à votre domaine Auth0 afin d’éviter la journalisation accidentelle du jeton ou son partage dans des URL, et de réduire la surface d’attaque liée à l’interception de jetons. Si vous devez utiliser un paramètre de requête (par exemple, pour Chrome Custom Tabs), assurez-vous que l’URL utilise HTTPS et supprimez le jeton de l’URL après son utilisation.

<div id="avoid-issuing-refresh-tokens-to-web-apps-unless-necessary">
  ## Évitez d’émettre des jetons d’actualisation pour les applications Web, sauf au besoin
</div>

Activez [`allow_refresh_token`](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id) uniquement pour les applications Web qui en ont réellement besoin. Dans la plupart des cas, des <Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=access+tokens">jetons d’accès</Tooltip> de courte durée combinés à une authentification silencieuse suffisent et sont plus sûrs dans un navigateur.

<div id="enable-allow-refresh-tokens-when-appropriate-to-set-the-refresh-tokens-as-online">
  ## Activer Allow Refresh Tokens lorsque c’est approprié pour que les jetons d’actualisation soient « en ligne »
</div>

Pour éviter les identifiants orphelins et empêcher les <Tooltip tip="Jeton d’actualisation : jeton utilisé pour obtenir un nouveau Jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=refresh+tokens">jetons d’actualisation</Tooltip> de rester actifs après la déconnexion, utilisez le paramètre [`allow_refresh_token`](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id) pour faire en sorte que les jetons d’actualisation émis via SSO natif vers le Web soient liés à la session qui les a générés. Si la session est révoquée ou expire, le jeton d’actualisation est automatiquement invalidé.

<div id="enable-enforce-cascade-revocation">
  ## Activer la révocation en cascade forcée
</div>

Pour garantir que toutes les sessions Web et tous les jetons d’actualisation associés à un `session_transfer_token` sont révoqués, activez [`enforce_cascade_revocation`](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id) dans l’application native. Cette mesure est essentielle pour assurer une invalidation sécurisée des sessions d’une application à l’autre.
