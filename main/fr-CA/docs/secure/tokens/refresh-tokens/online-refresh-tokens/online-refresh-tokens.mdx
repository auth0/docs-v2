Les jetons d’actualisation en ligne (ORT) sont un type de [jeton d’actualisation](/fr-CA/docs/secure/tokens/refresh-tokens) spécialement conçu pour les applications basées sur navigateur, comme les [applications monopages](/fr-CA/docs/get-started/auth0-overview/create-applications/single-page-web-apps) (SPA).

Les ORT permettent de suivre la [session](/fr-CA/docs/manage-users/sessions) d’un utilisateur dans les SPA lorsque des fonctionnalités de protection de la vie privée du navigateur, comme Intelligent Tracking Prevention (ITP), bloquent l’accès aux témoins tiers, ce qui rend l’[authentification silencieuse](/fr-CA/docs/authenticate/login/configure-silent-authentication#configure-silent-authentication) traditionnelle peu fiable.

Les ORT sont liés à la [session](/fr-CA/docs/manage-users/sessions) Auth0 de l’utilisateur et ne peuvent être utilisés que tant que cette session est active, ce qui garantit que, lorsqu’un utilisateur se déconnecte, toutes les applications qui utilisent cette session sont déconnectées en même temps.

<div id="how-it-works">
  ## Fonctionnement
</div>

Lorsque vous demandez un ORT :

1. Votre application inclut le scope `online_access` dans la demande d’autorisation.
2. Auth0 renvoie un ORT lié à la session actuelle de l’utilisateur.
3. Chaque fois que vous échangez l’ORT contre un nouveau jeton d’accès, Auth0 prolonge le délai d’inactivité de la session.
4. Le jeton reste valide tant que la session Auth0 sous-jacente est active.

<div id="comparison-of-orts-versus-refresh-tokens">
  ## Comparaison des ORT et des jetons d’actualisation
</div>

| **Fonctionnalité**                                                                                                                            | **ORT**                                                                                                              | **jeton d’actualisation**                                                                                          |
| --------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| **scope demandé**                                                                                                                             | `online_access`                                                                                                      | `offline_access`                                                                                                   |
| **Association à la session**                                                                                                                  | Lié à la session Auth0                                                                                               | Indépendant de la session                                                                                          |
| **Durée de vie**                                                                                                                              | Déterminée par la [durée de vie de la session](/fr-CA/docs/manage-users/sessions/configure-session-lifetime)               | Configurée selon la [durée de vie du jeton](/fr-CA/docs/secure/tokens/refresh-tokens/configure-refresh-token-expiration) |
| **Rotation des jetons**                                                                                                                       | Ne fait pas l’objet d’une rotation                                                                                   | Prend en charge la [rotation](/fr-CA/docs/secure/tokens/refresh-tokens/configure-refresh-token-rotation)                 |
| **Exigence liée à la [Demonstration of Proof of Possession](/fr-CA/docs/secure/sender-constraining/demonstrating-proof-of-possession-dpop) (DPoP)** | Requise pour les applications publiques (SPA, applications natives)                                                  | Facultative (configuration du serveur de ressources)                                                               |
| **Révocation**                                                                                                                                | [Révoque l’ensemble de la session](/fr-CA/docs/manage-users/sessions/manage-sessions-actions#revoke-sessions-with-actions) | [Révoque uniquement le jeton](/fr-CA/docs/secure/tokens/refresh-tokens/revoke-refresh-tokens)                            |
| **Management API**                                                                                                                            | Non visible                                                                                                          | Visible dans la Management API                                                                                     |
| **Délai d’inactivité de la session**                                                                                                          | Prolongé à chaque échange                                                                                            | N’a aucun effet sur la session                                                                                     |

<div id="key-behavioral-differences">
  ### Principales différences de comportement
</div>

**Extension de session :** Lorsque vous échangez un ORT, Auth0 réinitialise le délai d’inactivité de la session, ce qui permet de maintenir la session de l’utilisateur active. Auth0 ne prolonge pas la [durée de vie absolue](/fr-CA/docs/manage-users/sessions/session-lifecycle#absolute-timeout-maximum) de la session.

**Application de DPoP :** Pour les applications publiques (comme les SPA), Auth0 exige DPoP lors de l’utilisation des ORT. Comme les ORT offrent le même niveau de sécurité qu’un cookie de session donnant accès à la session authentifiée de l’utilisateur, ils doivent être protégés avec la même rigueur. DPoP empêche le vol de jetons et les attaques par rejeu.

**Sans état :** Les ORT sont liés à une session et ne peuvent donc pas être consultés ni gérés indépendamment. Vous ne pouvez pas les voir dans la Management API ni les révoquer individuellement.

<div id="best-practices">
  ## Bonnes pratiques
</div>

1. **Stockez les jetons en mémoire :** Évitez de stocker les ORT dans `localStorage` ou `sessionStorage`. Conservez-les dans la mémoire JavaScript pour une meilleure sécurité. Comme les ORT sont liés à la session Auth0, le cookie de session rétablira le contexte lors de la prochaine visite de l’utilisateur, ce qui permettra à votre application de demander un nouvel ORT.
2. **Gérez l’expiration de la session correctement :** Mettez en place une gestion appropriée des erreurs lorsque la session a expiré. Redirigez les utilisateurs pour qu’ils se réauthentifient sans perdre leur travail.
3. **Utilisez DPoP pour les applications publiques :** Les ORT exigent DPoP pour les applications publiques. Assurez-vous que votre implémentation génère et utilise correctement les preuves DPoP.
4. **Ne vous fiez pas à la structure du jeton :** Traitez les ORT comme des chaînes opaques. Ne les analysez pas et ne vous fiez à aucune structure interne — elle pourrait changer.
5. **Surveillez la durée de vie de la session :** Tenez compte à la fois des délais d’expiration de session pour inactivité et du délai d’expiration absolu. Le délai d’expiration absolu ne peut pas être prolongé par l’échange d’un jeton d’actualisation.

<div id="limitations">
  ## Limites
</div>

<div id="unsupported-flows">
  ### Flux non pris en charge
</div>

Les ORT ne sont disponibles que pour les flux qui créent une session interactive dans le navigateur. Les flux suivants **ne peuvent pas** émettre d’ORT :

| **Flux**                                                                                                                                         | **Raison**                                                                        |
| ------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------- |
| [Flux d’autorisation d’appareil](/fr-CA/docs/get-started/authentication-and-authorization-flow/device-authorization-flow)                              | Session créée dans le navigateur, mais jeton utilisé sur un autre appareil        |
| [Flux implicite](/fr-CA/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post)                                           | La spécification OAuth interdit les jetons d’actualisation dans le flux implicite |
| [Octroi avec mot de passe du propriétaire de la ressource](/fr-CA/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow) | Aucune session de navigateur n’est créée                                          |
| [Flux d’identifiants d’application](/fr-CA/docs/get-started/authentication-and-authorization-flow/client-credentials-flow)                             | Aucun utilisateur ni aucune session n’entre en jeu                                |
| [Échange de jetons personnalisé](/fr-CA/docs/get-started/authentication-and-authorization-flow/token-exchange-flow)                                    | Aucune session à laquelle associer le jeton                                       |

Si vous demandez le scope `online_access` dans l’un de ces flux, il est ignoré silencieusement et aucun jeton d’actualisation n’est émis.

<div id="scope-conflicts">
  ### Conflits de scopes
</div>

Vous ne pouvez pas demander à la fois `online_access` et `offline_access` dans la même requête d’autorisation. Si les deux scopes sont inclus, Auth0 renvoie une erreur :

```json
{
  "error": "invalid_request",
  "error_description": "Cannot request both online_access and offline_access scopes"
}
```

Sélectionnez le scope approprié selon votre cas d’utilisation :

* Utilisez `online_access` pour les applications Web qui bénéficient de la continuité du SSO.
* Utilisez `offline_access` pour les applications qui ont besoin de jetons pour fonctionner indépendamment de la session du navigateur.
