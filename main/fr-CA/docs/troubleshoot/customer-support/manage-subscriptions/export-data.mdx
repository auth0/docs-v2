title: Exporter les données
---

Toutes les données de votre locataire Auth0 demeurent en tout temps sous votre contrôle et sont accessibles à tout moment au moyen de la <Tooltip tip="Management API : Un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip>.

<div id="user-passwords">
  ## Mots de passe des utilisateurs
</div>

Les seules informations qui ne sont pas accessibles par l’API (pour des raisons de sécurité) sont les hachages de mots de passe de vos utilisateurs de base de données hébergés par Auth0 et les clés privées. Vous pouvez tout de même demander ces informations en ouvrant un [ticket d’assistance](https://support.auth0.com). Pour en savoir plus, consultez [Exporter les hachages de mots de passe et les secrets MFA](/fr-CA/docs/manage-users/user-migration/export-password-hashes-and-mfa-secrets). Cette opération n’est pas offerte avec notre offre d’abonnement Free, et nous ne pouvons ni accepter ni garantir les demandes d’exportation à une date et une heure précises.

Si vous stockez les mots de passe des utilisateurs dans votre base de données, configurez une connexion de base de données personnalisée qu’Auth0 interrogera chaque fois qu’un utilisateur se connectera. Dans ce cas, Auth0 ne stockera jamais de hachages de mots de passe, sauf si vous choisissez de migrer progressivement les utilisateurs vers Auth0.

<div id="auth0-components">
  ## Composants Auth0
</div>

Auth0 utilise principalement <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker les informations de connexion." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) comme protocole d’authentification; vous devriez donc pouvoir intégrer votre application à l’aide de bibliothèques standard. Il en va de même lorsque vous intégrez Auth0 au moyen de [SAML](/fr-CA/docs/authenticate/single-sign-on/outbound-single-sign-on/configure-auth0-saml-identity-provider). Tous les SDK, bibliothèques et exemples d’Auth0 [sont publiés sur GitHub en tant que logiciels libres](https://github.com/auth0/).

<div id="social-identity-providers">
  ## Fournisseurs d’identité sociaux
</div>

Si vous choisissez de ne pas utiliser Auth0, mais souhaitez continuer d’utiliser les mêmes <Tooltip tip="OAuth 2.0 : Cadre d’autorisation qui définit les protocoles et les flux d’autorisation." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=OAuth">OAuth</Tooltip> <Tooltip tip="OAuth 2.0 : Cadre d’autorisation qui définit les protocoles et les flux d’autorisation." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=client+IDs">ID client</Tooltip> et secrets pour vos <Tooltip tip="fournisseur d’identité (IdP) : Service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=identity+providers">fournisseurs d’identité</Tooltip> sociaux, vous conserverez l’accès aux renseignements sur les utilisateurs sans avoir à afficher de nouveaux écrans de consentement.

<div id="custom-code">
  ## Code personnalisé
</div>

Toutes les fonctionnalités de code personnalisé d’Auth0 (Rules, scripts de base de données personnalisés, connexions OAuth personnalisées, etc.) s’exécutent dans un service de bac à sable Node.js. Toutes les bibliothèques disponibles dans ce service de bac à sable sont aussi disponibles sur npm pour les utiliser avec du code Node.js standard.
