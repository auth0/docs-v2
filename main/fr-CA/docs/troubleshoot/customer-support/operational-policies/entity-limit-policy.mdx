title: Politique sur les limites des entités
---

Les entités dans Auth0 sont des éléments de configuration du locataire, comme les applications, les connexions, les organisations, les <Tooltip tip="Serveur de ressources : serveur hébergeant des ressources protégées. Les serveurs de ressources acceptent les demandes de ressources protégées et y répondent." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=resource+servers">serveurs de ressources</Tooltip> d’API, les autorisations, les rôles et les Actions.

<Warning>
  Les limites d’entités sont des limites absolues appliquées pour préserver la santé de notre plateforme; elles ne font pas partie de la tarification, bien qu’elles puissent varier selon certains groupes de plans tarifaires. Les plans tarifaires individuels comportent des limitations distinctes, qui correspondent aux droits inclus, comme les MAU et les Enterprise Connections, que vous obtenez lorsque vous souscrivez à un plan donné. (Pour connaître les limitations de votre plan tarifaire, consultez [Auth0 Pricing](https://auth0.com/pricing/).)
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Toute utilisation du terme &quot;unlimited&quot; signifie qu’il n’y a pas de limites absolues, mais Okta impose des seuils au niveau du système. Veuillez communiquer avec le soutien si vous avez des questions au sujet de ces limites.
</Callout>

<div id="global-subscription-plan-limits">
  ## Limites globales des forfaits d’abonnement
</div>

Cette section présente les limites qui s’appliquent à tous les forfaits d’abonnement.

<div id="organizations">
  ### Organisations
</div>

| Entity                                                          | Limit   |
| --------------------------------------------------------------- | ------- |
| Organisations par locataire                                     | 100,000 |
| Membres par organisation                                        | 100,000 |
| Connexions par organisation                                     | 10      |
| Attributions de rôles par membre de l’organisation              | 50      |
| Autorisations d’application M2M par organisation (cloud public) | 100     |
| Autorisations d’application M2M par organisation (cloud privé)  | 1000    |
| Profils d’échange de jetons personnalisés                       | 100     |

Les clients ayant un forfait Enterprise peuvent demander une augmentation des limites pour le nombre d’organisations par locataire et de membres par organisation en [communiquant avec le soutien](https://support.auth0.com/).

Ces limites peuvent être augmentées à 2,000,000 organisations par locataire et 2,000,000 membres par organisation dans le cloud public. Dans les instances de cloud privé, ces entités sont illimitées.

<div id="authorization-core-role-based-access-control-rbac">
  ### Contrôle d’accès basé sur les rôles (RBAC) d’Authorization Core
</div>

| Entité                                  | Limite |
| --------------------------------------- | ------ |
| Rôles par locataire                     | 1000   |
| Scopes par API (serveurs de ressources) | 1000   |
| Rôles par utilisateur                   | 50     |
| Autorisations par utilisateur           | 1000   |
| Autorisations par rôle                  | 1000   |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Notez que les limites relatives aux autorisations par utilisateur s’appliquent uniquement aux autorisations attribuées directement.

  Techniquement, un utilisateur peut avoir plus d’autorisations que le nombre indiqué si vous attribuez des autorisations à différents rôles, puis attribuez ces rôles à l’utilisateur.
</Callout>

| Entité                                                    | Limite |
| --------------------------------------------------------- | ------ |
| Jetons d’actualisation par utilisateur et par application | 200    |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous atteignez la limite de <Tooltip tip="Jeton d’actualisation : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Refresh+tokens">jetons d’actualisation</Tooltip> et créez un nouveau jeton d’actualisation,
  le système révoque ou supprime le plus ancien jeton de cet utilisateur pour
  l’application.

  Les jetons révoqués et expirés ne sont pas pris en compte dans cette limite.
</Callout>

<div id="actions">
  ### Actions
</div>

| Entité                                        | Limite |
| --------------------------------------------- | ------ |
| Actions par locataire                         | 100    |
| Actions par déclencheur par locataire         | 20     |
| Modules d’Actions par locataire               | 100    |
| Modules d’Actions par Action                  | 3      |
| Versions par Action                           | 50     |
| Versions par module d’Action                  | 50     |
| Secrets par Action                            | 30     |
| Secrets par module d’Action                   | 30     |
| Dépendances (modules NPM) par Action          | 10     |
| Dépendances (modules NPM) par module d’Action | 10     |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les limites des Actions et des modules d’Actions incluent à la fois les entités déployées et non déployées. Une fois supprimés, ces éléments ne sont plus comptabilisés dans la limite applicable.

  Les limites des versions d’Action et des versions de module d’Action incluent à la fois les brouillons et les versions actives. Une fois la limite atteinte, l’ajout de chaque version supplémentaire amène Auth0 à supprimer la plus ancienne version inutilisée.
</Callout>
