---

Voici quelques points à vérifier pour vous aider à déterminer à quel moment les problèmes surviennent lors de la connexion et de la déconnexion.

<div id="login-issues">
  ## Problèmes de connexion
</div>

<div id="is-the-user-prompted-for-login-credentials">
  ### L’utilisateur est-il invité à saisir ses identifiants ?
</div>

* Le fichier HAR montre-t-il un appel au <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités auxquelles un utilisateur a accès." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=authorization+server">serveur d’autorisation</Tooltip> (point de terminaison `/authorize`)?

  <Warning>
    Avant de partager un fichier HAR avec qui que ce soit (y compris Auth0), assurez-vous de supprimer ou d’obfusquer toutes les données sensibles, comme :

    * Renseignements confidentiels sur l’utilisateur
    * Renseignements personnels identifiables (PII)
    * Renseignements confidentiels sur l’application

    Pour en savoir plus, consultez les articles suivants sur [communauté Auth0](https://community.auth0.com) :

    * [Nettoyer les traces HTTP](https://community.auth0.com/t/sanitizing-http-traces/119488)
    * [Comment nettoyer automatiquement un fichier de trace HTTP](https://community.auth0.com/t/how-to-sanitize-a-http-trace-file-automatically/120583)
    * [Comment caviarder manuellement les renseignements sensibles](https://community.auth0.com/t/how-to-manually-redact-sensitive-information/122554)
    * [Le fichier HAR est trop volumineux pour être téléversé dans le dossier d’assistance](https://community.auth0.com/t/har-file-is-too-large-to-upload-to-the-support-case/122488)
  </Warning>
* La connexion est-elle activée pour l’application ?
* Le service d’autorisation distant est-il disponible ?
* Si vous utilisez la [page Universal Login d’Auth0](/fr-CA/docs/authenticate/login/auth0-universal-login), essayez de désactiver la personnalisation et vérifiez si l’authentification fonctionne. Si la connexion fonctionne sans vos personnalisations, examinez le code de personnalisation de votre page <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur le serveur d’autorisation d’Auth0, afin de vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Universal+Login">Universal Login</Tooltip>.

<div id="is-an-error-message-shown-after-entering-credentials">
  ### Un message d’erreur s’affiche-t-il après la saisie des identifiants ?
</div>

* Pouvez-vous essayer de vous connecter d’une autre façon pour vous assurer que les identifiants sont corrects ?
* Si l’expiration du mot de passe est possible, vérifiez si le mot de passe a expiré.
* Vérifiez les outils de développement de votre navigateur ou la console de l’inspecteur Web pour repérer les erreurs dans le flux avant le retour à Auth0.
* Vérifiez le fichier HAR - montre-t-il un retour vers Auth0 (point de terminaison `/login/callback`) ?

  <Warning>
    Avant de partager un fichier HAR avec qui que ce soit (y compris Auth0), assurez-vous de supprimer ou de masquer toutes les données sensibles, telles que :

    * des renseignements confidentiels sur l’utilisateur
    * des renseignements personnels identifiables (PII)
    * des renseignements confidentiels sur l’application

    Pour en savoir plus, consultez les articles suivants sur la [communauté Auth0](https://community.auth0.com) :

    * [Assainir les traces HTTP](https://community.auth0.com/t/sanitizing-http-traces/119488)
    * [Comment assainir automatiquement un fichier de trace HTTP](https://community.auth0.com/t/how-to-sanitize-a-http-trace-file-automatically/120583)
    * [Comment caviarder manuellement des renseignements sensibles](https://community.auth0.com/t/how-to-manually-redact-sensitive-information/122554)
    * [Le fichier HAR est trop volumineux pour être téléversé dans le dossier d’assistance](https://community.auth0.com/t/har-file-is-too-large-to-upload-to-the-support-case/122488)
  </Warning>

  * Si ce n’est pas le cas, vérifiez que le fournisseur d’identité est configuré avec la bonne URL de rappel pour Auth0.

<div id="is-a-login-session-established-for-the-user-at-the-authorization-server">
  ### Une session de connexion est-elle établie pour l’utilisateur sur le serveur d’autorisation ?
</div>

* Pour le vérifier, ouvrez un deuxième onglet dans le même navigateur et accédez à la même URL. Êtes-vous invité à vous reconnecter ?

  * Si vous n’êtes pas invité à vous reconnecter, une session existe.

<div id="is-a-log-entry-created-in-your-auth0-logs">
  ### Une entrée de journal a-t-elle été créée dans les journaux Auth0?
</div>

* Si aucune entrée de journal n’a été créée, la transaction d’authentification n’a pas abouti ou n’est pas revenue à Auth0.
* Vérifiez la réponse du serveur d’autorisation pour voir s’il contient des messages d’erreur.
* Vérifiez les journaux du serveur d’autorisation (si possible) pour repérer des erreurs.

<div id="is-an-entry-created-in-auth0-users-screen-with-all-correct-profile-info">
  ### Une entrée est-elle créée dans l’écran utilisateur d’Auth0 avec tous les bons renseignements de profil ?
</div>

* Si ce n’est pas le cas, vérifiez la réponse du serveur d’autorisation dans le fichier HAR. Il se peut qu’elle ne renvoie aucune information sur l’utilisateur.

  <Warning>
    Avant de partager un fichier HAR avec qui que ce soit (y compris Auth0), assurez-vous de supprimer ou d’obfusquer toutes les données sensibles, comme :

    * Renseignements confidentiels sur l’utilisateur
    * Renseignements personnels identifiables (PII)
    * Renseignements confidentiels sur l’application

    Pour en savoir plus, consultez les articles suivants dans la [communauté Auth0](https://community.auth0.com) :

    * [Sanitizing HTTP Traces](https://community.auth0.com/t/sanitizing-http-traces/119488)
    * [How to Sanitize an HTTP Trace File Automatically](https://community.auth0.com/t/how-to-sanitize-a-http-trace-file-automatically/120583)
    * [How to Manually Redact Sensitive Information](https://community.auth0.com/t/how-to-manually-redact-sensitive-information/122554)
    * [HAR File is Too Large to Upload to the Support Case](https://community.auth0.com/t/har-file-is-too-large-to-upload-to-the-support-case/122488)
  </Warning>
* Si vous utilisez des Rules, vérifiez s’il y a des problèmes dans vos scripts Rules.
* Si vous utilisez une connexion de base de données personnalisée, vérifiez s’il y a des problèmes dans vos scripts d’action de base de données.
* Si vous utilisez LDAP, vérifiez s’il y a des problèmes dans le script de mappage de profil (`profileMapper.js`).
* Vérifiez la configuration de vos connexions sociales pour voir quelles informations de profil sont demandées.

<div id="does-the-har-file-show-a-token-or-assertion-returned-to-application">
  ### Le fichier HAR montre-t-il un jeton ou une assertion renvoyé à l’application ?
</div>

* Recherchez dans le [fichier HAR](/fr-CA/docs/troubleshoot/troubleshooting-tools/generate-and-analyze-har-files) l’appel à votre Application Callback URL.

  <Warning>
    Avant de partager un fichier HAR avec qui que ce soit (y compris Auth0), assurez-vous de supprimer ou d’obfusquer toutes les données sensibles, par exemple :

    * Des renseignements confidentiels sur l’utilisateur
    * Des renseignements personnels identifiables (PII)
    * Des renseignements confidentiels sur l’application

    Pour en savoir plus, consultez les articles suivants dans la [communauté Auth0](https://community.auth0.com) :

    * [Nettoyer les traces HTTP](https://community.auth0.com/t/sanitizing-http-traces/119488)
    * [Comment nettoyer automatiquement un fichier de trace HTTP](https://community.auth0.com/t/how-to-sanitize-a-http-trace-file-automatically/120583)
    * [Comment masquer manuellement des renseignements sensibles](https://community.auth0.com/t/how-to-manually-redact-sensitive-information/122554)
    * [Le fichier HAR est trop volumineux pour être téléversé dans le dossier d’assistance](https://community.auth0.com/t/har-file-is-too-large-to-upload-to-the-support-case/122488)
  </Warning>
* Repérez le <Tooltip tip="ID Token : jeton destiné à l’application elle-même, et non à l’accès à une ressource." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=ID+Token">ID Token</Tooltip> (`id_token`) et vérifiez s’il contient les renseignements requis par l’application.

<div id="if-you-decode-the-token-or-assertion-does-it-have-information-you-expect">
  ### Si vous décodez le jeton ou l’assertion, contient-il l’information attendue?
</div>

* Consultez les <Tooltip tip="JSON Web Token (JWT) : format standard des ID Token (et souvent des Jetons d’accès), utilisé pour représenter de façon sécurisée des revendications entre deux parties." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=JWTs">JWT</Tooltip> avec [JWT.io](http://jwt.io).
* Consultez les assertions <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des informations d’authentification sans mot de passe." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=SAML">SAML</Tooltip> avec [SAMLTool.io](http://samltool.io).

<div id="if-the-logged-in-user-cannot-access-another-application-with-single-sign-on">
  ### Si l’utilisateur connecté ne peut pas accéder à une autre application avec l’authentification unique
</div>

1. L’utilisateur essaie-t-il de se connecter à la deuxième application dans le même navigateur que celui utilisé lors de sa connexion initiale ?
2. Accédez à [Paramètres du locataire &gt; Avancé](https://manage.auth0.com/#/tenant/advanced) et vérifiez les paramètres de **gestion des sessions de connexion**. La deuxième tentative de connexion a-t-elle eu lieu avant l’expiration de ces délais ?
3. Vérifiez la valeur transmise dans le paramètre `prompt` de l’appel à `/authorize`.
4. La connexion utilisée pour se connecter à la première application est-elle activée pour la deuxième application ?
5. La deuxième application a-t-elle reçu toutes les informations nécessaires du profil utilisateur ?
6. Si vous utilisez un appareil mobile, vous devrez utiliser un flux basé sur le navigateur. Pour en savoir plus, consultez [Meilleures pratiques](/fr-CA/docs/authenticate/login/native-login).

<div id="check-application-logs">
  ### Vérifiez les journaux de l’application
</div>

* Les journaux de votre application affichent-ils des erreurs?
* L’application a-t-elle reçu toutes les informations nécessaires, comme des groupes ou des attributs du profil utilisateur?

<div id="logout-issues">
  ## Problèmes de déconnexion
</div>

* Avez-vous ajouté vos URL de redirection de déconnexion à la liste des URL autorisées ? Si vous utilisez une URL de redirection dans un appel de déconnexion, elle doit être enregistrée dans les paramètres du locataire ou dans les paramètres de l’application.
* Si vous avez besoin d’une déconnexion fédérée, avez-vous ajouté le paramètre `?federated` à l’appel de déconnexion ?
* Assurez-vous que l’URL de redirection de déconnexion est différente de l’URL de rappel de connexion.
* Faites en sorte que l’URL de redirection de déconnexion pointe vers une page anonyme (non protégée par authentification), afin que les redirections vers cette URL ne déclenchent pas immédiatement une connexion, ce qui pourrait semer la confusion chez les utilisateurs.
