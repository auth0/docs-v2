description: Describe las prácticas recomendadas y los estándares para tus API.
title: Prácticas recomendadas para las API
---

Cuando te integres con los servicios de Auth0, te recomendamos configurar tus API de acuerdo con los estándares de red actuales. Revisa las siguientes configuraciones de resolución de DNS, de Transport Layer Security (TLS) y las estrategias de optimización de HTTP, diseñadas para garantizar que tu aplicación gestione los cambios de infraestructura y mantenga conexiones resilientes con los servicios de Auth0. Seguir estos estándares de DNS, TLS y HTTP garantiza la compatibilidad con nuestra red global de distribución, mejora la fiabilidad de las solicitudes mediante una lógica de reintento adecuada y mantiene la seguridad del tráfico de autenticación.

<div id="dns-and-hostname-resolution">
  ## DNS y resolución de nombres de host
</div>

<div id="use-auth0-provided-hostnames">
  ### Use los nombres de host proporcionados por Auth0
</div>

Conéctese siempre a Auth0 usando los nombres de host proporcionados, no la dirección IP subyacente. La infraestructura de Auth0 depende de la gestión del tráfico basada en DNS para el equilibrio de carga, el enrutamiento en la nube y mediante CDN, y la conmutación por error entre regiones. Recomendamos no usar filtrado basado en DNS, ya que los nombres de la infraestructura en la nube pueden cambiar con el tiempo.

<div id="respect-dns-time-to-live-ttl">
  ### Respete el tiempo de vida (TTL) del DNS
</div>

Respete siempre el [TTL del DNS](https://www.ateam-oracle.com/dns-ttls-considerations) proporcionado por el DNS. Los valores de TTL suelen ser bajos (por ejemplo, entre 30 y 300 segundos) para que las aplicaciones respondan rápidamente a los cambios en la infraestructura.

<div id="dns-caching">
  ### Caché DNS
</div>

Use el resolvedor DNS de su sistema operativo siempre que sea posible. Si utiliza su propio resolvedor (a nivel de aplicación), asegúrese de que no almacene en caché las respuestas durante más tiempo que el TTL.

<div id="support-multiple-aaaaa-records">
  ### Compatibilidad con varios registros A/AAAA
</div>

Auth0 puede devolver varias direcciones IP para un nombre de host determinado. Las API deben distribuir las conexiones entre las IP devueltas mediante round-robin o asignación aleatoria. Configure la conmutación por error para que se produzca de inmediato si una dirección no es accesible.

<div id="transport-layer-security-tls">
  ## Seguridad de la capa de transporte (TLS)
</div>

Auth0 solo admite solicitudes HTTP con TLS. Le recomendamos usar las versiones más recientes:

* **TLS 1.3** muy recomendable
* **TLS 1.2**
* **TLS 1.1 y versiones anteriores no son compatibles**

<div id="use-modern-cipher-suites">
  ### Use suites de cifrado modernas
</div>

Evite usar configuraciones de suites de cifrado del cliente muy personalizadas. Use la configuración segura predeterminada de su entorno de ejecución, a menos que tenga requisitos de cumplimiento normativo (por ejemplo, FIPS). Es posible que se rechacen los clientes con compatibilidad limitada con suites de cifrado.

Para consultar los cifrados disponibles, lea [TLS (SSL) Versions and Ciphers](/es/docs/customize/custom-domains/self-managed-certificates/tls-ssl).

<div id="support-server-name-indication-sni">
  ### Compatibilidad con Server Name Indication (SNI)
</div>

Las API deben usar la extensión TLS SNI en todo el tráfico HTTPS dirigido a Auth0. No se aceptará tráfico sin SNI.

<div id="honor-certificate-validation">
  #### Respete la validación de certificados
</div>

**No**:

* Desactive la validación de certificados
* Fije certificados

Auth0 renueva y reemplaza los certificados automáticamente y con frecuencia. La fijación de certificados puede causar interrupciones del servicio.

<div id="allow-intermediate-certificate-rotation">
  #### Permita la rotación de certificados intermedios
</div>

Nuestra cadena de certificados puede cambiar periódicamente. Asegúrese de que su cliente TLS:

* Confíe de forma predeterminada en cualquier autoridad de certificación válida y de confianza pública del almacén de confianza del sistema operativo
* Actualice el almacén de confianza del sistema operativo como parte de las actualizaciones periódicas
* No fije certificados intermedios
* Rechace únicamente cadenas inválidas

<div id="enable-tls-session-resumption-and-reuse">
  #### Habilite la reanudación y reutilización de sesiones TLS
</div>

Permita tickets de sesión TLS o ID de sesión en HTTP/2. Esto reduce la sobrecarga de la negociación TLS.

<div id="http">
  ## HTTP
</div>

<div id="use-a-modern-http-protocol">
  ### Utilice un protocolo HTTP moderno
</div>

Utilice [HTTP/2](https://www.rfc-editor.org/rfc/rfc9113) o [HTTP/3](https://httpwg.org/specs/rfc9114.html). Estos ofrecen multiplexación de conexiones, usan compresión de encabezados HTTP y requieren menos conexiones TCP. Si su cliente admite HTTP/2 automáticamente, use endpoints HTTPS y deje que el cliente negocie HTTP/2 o una versión superior.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Evite HTTP/1.1 salvo que sea necesario. HTTP/1.1 es totalmente compatible, pero menos eficiente. Se desaconseja su uso salvo que sea necesario.
</Callout>

<div id="enable-and-reuse-persistent-connections">
  ### Habilite y reutilice conexiones persistentes
</div>

Configure su API para reutilizar conexiones HTTP, reducir la latencia de las solicitudes y mejorar la confiabilidad. Revise la siguiente configuración recomendada:

* Mantenga las conexiones abiertas durante un máximo de 300 segundos
* Limite el número máximo de conexiones inactivas a una cantidad razonable
* Evite crear un nuevo objeto de API HTTP por solicitud para aprovechar las conexiones existentes
* Rote las conexiones periódicamente para renovar su estado

<div id="request-retries">
  ### Reintentos de solicitudes
</div>

Recomendamos implementar reintentos de solicitudes para operaciones que se puedan reintentar de forma segura: [solicitudes idempotentes](https://developer.mozilla.org/en-US/docs/Glossary/Idempotent).

Al reintentar debido a errores transitorios de red o del servidor, inspeccione el encabezado de respuesta HTTP `Retry-After`, que indica cuánto tiempo debe esperar una aplicación antes de realizar otra solicitud.

Ejemplo: `Retry-After: 300`

Si el encabezado está presente, espere la cantidad de segundos especificada antes de reintentar una solicitud. Respetar `Retry-After` garantiza un uso equitativo y evita fallos en cascada o la limitación de tasa.

Si `Retry-After` no está presente, use [retroceso exponencial](https://en.wikipedia.org/wiki/Exponential_backoff) para determinar el intervalo de reintento. No realice más de tres intentos.

<div id="auth0-sdks">
  ## SDKs de Auth0
</div>

Recomendamos seguir estas prácticas recomendadas al configurar su entorno o aplicación para usar los SDKs de Auth0.

<div id="use-official-auth0-sdks">
  ### Usa los SDK oficiales de Auth0
</div>

Prioriza los SDK [oficiales o referenciados](docs/libraries) para tu plataforma y lenguaje. Usa los SDK oficiales de Auth0 en lugar de crear aplicaciones HTTP personalizadas o integraciones de API. Estas bibliotecas se mantienen siguiendo las prácticas recomendadas y abstraen la complejidad de las redes de bajo nivel y del cumplimiento de los requisitos de seguridad.

<div id="choose-the-applicable-sdk">
  ### Elige el SDK adecuado
</div>

Auth0 proporciona bibliotecas especializadas para distintos patrones de integración. Asegúrate de seleccionar el SDK que se ajuste a tu caso de uso arquitectónico:

* **SDKs de autenticación y autorización**: Usa estos SDKs para el inicio de sesión de usuarios finales y para obtener tokens. Elige el SDK según el tipo de aplicación:
  * **Aplicaciones de una sola página (SPA)**: Las bibliotecas de JS/TS gestionan el ciclo de vida de los tokens y los flujos PKCE en el navegador (p. ej., [React](https://github.com/auth0/auth0-react), [Vue](https://github.com/auth0/auth0-vue), [Angular](https://github.com/auth0/auth0-angular)).
  * **Aplicaciones web tradicionales**: Las bibliotecas del lado del servidor gestionan sesiones y transacciones basadas en cookies (p. ej., [Node.js](https://github.com/auth0/node-auth0), [ASP.NET Core](https://github.com/auth0/auth0-aspnetcore-authentication), [PHP](https://github.com/auth0/auth0-PHP)).
  * **Aplicaciones nativas/móviles**: Las bibliotecas nativas de cada plataforma interactúan con navegadores del sistema y almacenamiento seguro ([Swift](https://github.com/auth0/Auth0.swift), [Kotlin](https://github.com/auth0/Auth0.android), [React Native](https://github.com/auth0/react-native-auth0)).
* **Servicio backend y API**: Valida los Token de acceso entrantes en tus APIs. Estas bibliotecas se encargan de la verificación de firmas criptográficas y del almacenamiento en caché de [JSON Web Key Sets (JWKS)](docs/secure/tokens/json-web-tokens/json-web-key-sets#json-web-key-sets).
* **SDKs de Management API**: Permiten realizar de forma programática tareas administrativas, como crear nuevos usuarios y rotar Secretos del cliente. Gestionan los encabezados de limitación de tasa y la lógica de reintento.

Al usar SDKs oficiales, te aseguras de que tu integración admita automáticamente:

* Agrupación resiliente de conexiones HTTP y configuración de keep-alive.
* Implementación correcta de la lógica de reintento con retroceso exponencial y respeto del encabezado `Retry-After`.
* Cumplimiento de los estándares actuales de TLS y seguridad.
