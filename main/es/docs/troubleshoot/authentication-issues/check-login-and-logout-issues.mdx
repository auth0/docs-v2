Estas son algunas cosas que debe comprobar para acotar cuándo se producen los problemas durante el inicio y cierre de sesión.

<div id="login-issues">
  ## Problemas de inicio de sesión
</div>

<div id="is-the-user-prompted-for-login-credentials">
  ### ¿Se le solicita al usuario que introduzca sus credenciales de inicio de sesión?
</div>

* ¿El archivo HAR muestra una llamada al <Tooltip tip="Servidor de autorización: servidor centralizado que ayuda a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/es/docs/glossary?term=authorization+server">Servidor de autorización</Tooltip> (endpoint `/authorize`)?

  <Warning>
    Antes de compartir un archivo HAR con cualquier persona (incluido Auth0), asegúrese de eliminar u ofuscar todos los datos confidenciales, como:

    * Información confidencial del usuario
    * Información de identificación personal (PII)
    * Información confidencial de la aplicación

    Para obtener más información, consulte los siguientes artículos en [Auth0 Community](https://community.auth0.com):

    * [Sanitización de trazas HTTP](https://community.auth0.com/t/sanitizing-http-traces/119488)
    * [Cómo sanitizar automáticamente un archivo de traza HTTP](https://community.auth0.com/t/how-to-sanitize-a-http-trace-file-automatically/120583)
    * [Cómo redactar manualmente información confidencial](https://community.auth0.com/t/how-to-manually-redact-sensitive-information/122554)
    * [El archivo HAR es demasiado grande para subirlo al caso de soporte](https://community.auth0.com/t/har-file-is-too-large-to-upload-to-the-support-case/122488)
  </Warning>
* ¿La conexión está habilitada para la aplicación?
* ¿Está disponible el servicio de autorización remoto?
* Si utiliza la [página de Universal Login de Auth0](/es/docs/authenticate/login/auth0-universal-login), intente desactivar la personalización y compruebe si la autenticación funciona. Si el inicio de sesión funciona sin sus personalizaciones, revise el código de personalización de su página de <Tooltip tip="Universal Login: su aplicación redirige a Universal Login, alojado en el Servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/es/docs/glossary?term=Universal+Login">Universal Login</Tooltip>.

<div id="is-an-error-message-shown-after-entering-credentials">
  ### ¿Se muestra un mensaje de error después de introducir las credenciales?
</div>

* ¿Puede probar a iniciar sesión de otra forma para confirmar que las credenciales son correctas?
* Si es posible que la contraseña haya caducado, compruebe si ya ha caducado.
* Revise las herramientas para desarrolladores del navegador o la consola del inspector web para detectar errores en el flujo antes de volver a Auth0.
* Revise el archivo HAR: ¿muestra un regreso a Auth0 (endpoint `/login/callback`)?

  <Warning>
    Antes de compartir un archivo HAR con cualquier persona (incluido Auth0), asegúrese de eliminar u ofuscar todos los datos confidenciales, como:

    * Información confidencial del usuario
    * Información de identificación personal (PII)
    * Información confidencial de la aplicación

    Para obtener más información, lea los siguientes artículos en [Auth0 Community](https://community.auth0.com):

    * [Sanitizing HTTP Traces](https://community.auth0.com/t/sanitizing-http-traces/119488)
    * [How to Sanitize an HTTP Trace File Automatically](https://community.auth0.com/t/how-to-sanitize-a-http-trace-file-automatically/120583)
    * [How to Manually Redact Sensitive Information](https://community.auth0.com/t/how-to-manually-redact-sensitive-information/122554)
    * [HAR File is Too Large to Upload to the Support Case](https://community.auth0.com/t/har-file-is-too-large-to-upload-to-the-support-case/122488)
  </Warning>

  * Si no es así, verifique que el proveedor de identidad tenga la URL de devolución de llamada correcta para Auth0.

<div id="is-a-login-session-established-for-the-user-at-the-authorization-server">
  ### ¿Se establece una sesión de Login para el usuario en el Servidor de autorización?
</div>

* Para comprobarlo, abra una segunda pestaña en el mismo navegador y vaya a la misma URL. ¿Se le vuelve a pedir que inicie sesión?

  * Si no se le vuelve a pedir que inicie sesión, hay una sesión activa.

<div id="is-a-log-entry-created-in-your-auth0-logs">
  ### ¿Se creó una entrada de registro en los Logs de Auth0?
</div>

* Si no se creó ninguna entrada de registro, la transacción de autenticación no se completó o no regresó a Auth0.
* Compruebe la respuesta del Servidor de autorización para ver si hay mensajes de error.
* Compruebe los logs del Servidor de autorización (si es posible) para identificar errores.

<div id="is-an-entry-created-in-auth0-users-screen-with-all-correct-profile-info">
  ### ¿Se creó una entrada en la pantalla de usuarios de Auth0 con toda la información de perfil correcta?
</div>

* Si no es así, revise la respuesta del Servidor de autorización en el archivo HAR. Es posible que no esté devolviendo información sobre el usuario.

  <Warning>
    Antes de compartir un archivo HAR con alguien (incluido Auth0), asegúrese de eliminar u ofuscar todos los datos confidenciales, como:

    * Información confidencial del usuario
    * Información de identificación personal (PII)
    * Información confidencial de la aplicación

    Para obtener más información, lea los siguientes artículos en [Auth0 Community](https://community.auth0.com):

    * [Saneamiento de trazas HTTP](https://community.auth0.com/t/sanitizing-http-traces/119488)
    * [Cómo sanear automáticamente un archivo de traza HTTP](https://community.auth0.com/t/how-to-sanitize-a-http-trace-file-automatically/120583)
    * [Cómo ocultar manualmente información confidencial](https://community.auth0.com/t/how-to-manually-redact-sensitive-information/122554)
    * [El archivo HAR es demasiado grande para cargarlo en el caso de soporte](https://community.auth0.com/t/har-file-is-too-large-to-upload-to-the-support-case/122488)
  </Warning>
* Si usa Rules, revise sus scripts de Rules para detectar problemas.
* Si usa una conexión de base de datos personalizada, revise sus scripts de Action de base de datos para detectar problemas.
* Si usa LDAP, revise el script de asignación de perfil (`profileMapper.js`) para detectar problemas.
* Revise la configuración de su conexión social para ver qué información de perfil se solicita.

<div id="does-the-har-file-show-a-token-or-assertion-returned-to-application">
  ### ¿El archivo HAR muestra un token o una aserción devueltos a la aplicación?
</div>

* Busque en el [archivo HAR](/es/docs/troubleshoot/troubleshooting-tools/generate-and-analyze-har-files) la llamada a la URL de devolución de llamada de su aplicación.

  <Warning>
    Antes de compartir un archivo HAR con alguien (incluido Auth0), asegúrese de eliminar u ocultar todos los datos sensibles, como:

    * Información confidencial del usuario
    * Información de identificación personal (PII)
    * Información confidencial de la aplicación

    Para obtener más información, consulte los siguientes artículos en [Auth0 Community](https://community.auth0.com):

    * [Cómo sanear trazas HTTP](https://community.auth0.com/t/sanitizing-http-traces/119488)
    * [Cómo sanear automáticamente un archivo de traza HTTP](https://community.auth0.com/t/how-to-sanitize-a-http-trace-file-automatically/120583)
    * [Cómo redactar manualmente información sensible](https://community.auth0.com/t/how-to-manually-redact-sensitive-information/122554)
    * [El archivo HAR es demasiado grande para subirlo al caso de soporte](https://community.auth0.com/t/har-file-is-too-large-to-upload-to-the-support-case/122488)
  </Warning>
* Busque el <Tooltip tip="ID Token: credencial destinada al propio cliente, no al acceso a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=ID+Token">ID Token</Tooltip> (`id_token`) y compruebe si contiene la información que necesita la aplicación.

<div id="if-you-decode-the-token-or-assertion-does-it-have-information-you-expect">
  ### Si decodifica el token o la aserción, ¿incluye la información esperada?
</div>

* Vea los <Tooltip tip="JSON Web Token (JWT): formato estándar de ID Token (y, a menudo, de Token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/es/docs/glossary?term=JWTs">JWTs</Tooltip> con [JWT.io](http://jwt.io).
* Vea las aserciones <Tooltip tip="Security Assertion Markup Language (SAML): protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/es/docs/glossary?term=SAML">SAML</Tooltip> con [SAMLTool.io](http://samltool.io).

<div id="if-the-logged-in-user-cannot-access-another-application-with-single-sign-on">
  ### Si el usuario que ha iniciado sesión no puede acceder a otra aplicación con inicio de sesión único
</div>

1. ¿El usuario está intentando iniciar sesión en la segunda aplicación desde el mismo navegador que usó en el inicio de sesión inicial?
2. Ve a [Tenant Settings &gt; Advanced Settings](https://manage.auth0.com/#/tenant/advanced) y revisa la configuración de **Log In Session Management**. ¿El segundo intento de inicio de sesión se realizó dentro de los tiempos de espera establecidos?
3. Verifica el valor que se pasa como parámetro `prompt` en la llamada a `/authorize`.
4. ¿La conexión que se usó para iniciar sesión en la primera aplicación está habilitada para la segunda aplicación?
5. ¿La segunda aplicación recibió toda la información necesaria del perfil de usuario?
6. Si usas un dispositivo móvil, tendrás que usar un flujo basado en navegador. Para obtener más información, consulta [Prácticas recomendadas](/es/docs/authenticate/login/native-login).

<div id="check-application-logs">
  ### Revise los registros de la aplicación
</div>

* ¿Los registros de su aplicación muestran algún error?
* ¿La aplicación recibió toda la información que necesita, como grupos o atributos del perfil de usuario?

<div id="logout-issues">
  ## Problemas de cierre de sesión
</div>

* ¿Agregó las URL de redirección de cierre de sesión a la lista de permitidos? Si usa una URL de redirección en una llamada de cierre de sesión, debe registrarla en la configuración del inquilino o de la aplicación.
* Si necesita cierre de sesión federado, ¿agregó el parámetro `?federated` a la llamada de cierre de sesión?
* Asegúrese de que la URL de redirección de cierre de sesión sea distinta de la URL de devolución de llamada de inicio de sesión.
* Haga que la URL de redirección de cierre de sesión apunte a una página accesible de forma anónima (no protegida por inicio de sesión), para que las redirecciones a esa URL no activen inmediatamente un inicio de sesión, lo que puede confundir a los usuarios.
