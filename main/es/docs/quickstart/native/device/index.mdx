* API de autenticación: Siga leyendo para saber cómo llamar directamente a nuestra API. Para una experiencia interactiva, consulte nuestro [entorno de pruebas de Device Flow](https://auth0.github.io/device-flow-playground/).

<div id="prerequisites">
  ## Requisitos previos
</div>

**Antes de comenzar este tutorial:**

* Revisa las [limitaciones](#limitations) para asegurarte de que el flujo de autorización de dispositivos sea adecuado para tu implementación.
* [Registra la aplicación en Auth0](/es/docs/get-started/auth0-overview/create-applications/native-apps).

  * Selecciona un **Tipo de aplicación** de **Native**.
  * Si es necesario, configura **Allowed Web Origins**. Puedes usarlo para permitir localhost como origen para el desarrollo local o para establecer un origen permitido para software de TV específico con una arquitectura sujeta a CORS (por ejemplo, HTML5 + JS). La mayoría de las aplicaciones no usarán esta configuración.
  * Asegúrate de que la opción **OIDC Conformant** esté habilitada. Esta configuración está en el [Dashboard](https://manage.auth0.com/#), en **Application Settings &gt; Advanced &gt; OAuth**.
  * Asegúrate de que los [**Grant Types**](/es/docs/get-started/applications/update-grant-types) de la aplicación incluyan **Device Code**. Esto también está en el [Dashboard](https://manage.auth0.com/#), en **Application Settings &gt; Advanced &gt; Grant Types**.
  * Si quieres que tu aplicación pueda usar [Tokens de actualización](/es/docs/secure/tokens/refresh-tokens), asegúrate de que los [**Grant Types**](/es/docs/get-started/applications/update-grant-types) de la aplicación incluyan **Refresh Token**.
* Configura y habilita al menos una conexión para la aplicación: [Conexiones de base de datos](/es/docs/get-started/applications/set-up-database-connections), [Conexiones sociales](https://marketplace.auth0.com/features/social-connections)
* [Registra tu API en Auth0](/es/docs/get-started/architecture-scenarios/mobile-api/part-2#create-the-api)

  * Si quieres que tu API reciba [Tokens de actualización](/es/docs/secure/tokens/refresh-tokens) para obtener nuevos tokens cuando los anteriores expiren, habilita **Allow Offline Access**.
* [Configura los ajustes del código de usuario del dispositivo](/es/docs/get-started/tenant-settings/configure-device-user-code-settings) para definir el conjunto de caracteres, el formato y la longitud de tu código de usuario generado aleatoriamente.

<div id="steps">
  ## Pasos
</div>

1. [Solicitar un código de dispositivo](#request-device-code) (Flujo de dispositivo): Solicite un código de dispositivo que el usuario pueda usar para autorizar el dispositivo.
2. [Solicitar la activación del dispositivo](#request-device-activation) (Flujo de dispositivo): Solicite que el usuario autorice el dispositivo usando su laptop o smartphone.
3. [Solicitar tokens](#request-tokens) (Flujo de dispositivo): Consulte periódicamente el endpoint de token para solicitar un token.
4. [Autorización del usuario](#user-authorization) (Flujo del navegador): El usuario autoriza el dispositivo para que este pueda recibir tokens.
5. [Recibir tokens](#receive-tokens) (Flujo de dispositivo): Después de que el usuario autorice correctamente el dispositivo, reciba los tokens.
6. [Llamar a su API](#call-your-api) (Flujo de dispositivo): Use el Token de acceso obtenido para llamar a su API.
7. [Actualizar tokens](#refresh-tokens) (Flujo de dispositivo): Use un Token de actualización para solicitar nuevos tokens cuando caduquen los actuales.

<div id="request-device-code">
  ## Solicitar código de dispositivo
</div>

Una vez que el usuario haya iniciado la aplicación del dispositivo y quiera autorizarlo, deberá obtener un código de dispositivo. Cuando el usuario inicie su sesión en el dispositivo basado en navegador, este código quedará vinculado a esa sesión.

Para obtener el código de dispositivo, la aplicación debe solicitar un código en la [URL del código de dispositivo](https://auth0.com/docs/api/authentication#get-device-code), incluido el ID de cliente.

<div id="example-post-to-device-code-url">
  ### Ejemplo de POST a la URL de código de dispositivo
</div>

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/oauth/device/code' \
    --header 'content-type: application/x-www-form-urlencoded' \
    --data 'client_id={yourClientId}' \
    --data 'scope=SCOPE' \
    --data 'audience=AUDIENCE'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/oauth/device/code");
  var request = new RestRequest(Method.POST);
  request.AddHeader("content-type", "application/x-www-form-urlencoded");
  request.AddParameter("application/x-www-form-urlencoded", "client_id={yourClientId}&scope=SCOPE&audience=AUDIENCE", ParameterType.RequestBody);
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"strings"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/oauth/device/code"

  	payload := strings.NewReader("client_id=%24%7Baccount.clientId%7D&scope=SCOPE&audience=AUDIENCE")

  	req, _ := http.NewRequest("POST", url, payload)

  	req.Header.Add("content-type", "application/x-www-form-urlencoded")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

  ```java Java
  HttpResponse<String> response = Unirest.post("https://{yourDomain}/oauth/device/code")
    .header("content-type", "application/x-www-form-urlencoded")
    .body("client_id=%24%7Baccount.clientId%7D&scope=SCOPE&audience=AUDIENCE")
    .asString();
  ```

  ```javascript Node.JS
  var axios = require("axios").default;

  var options = {
    method: 'POST',
    url: 'https://{yourDomain}/oauth/device/code',
    headers: {'content-type': 'application/x-www-form-urlencoded'},
    data: new URLSearchParams({
      client_id: '{yourClientId}',
      scope: 'SCOPE',
      audience: 'AUDIENCE'
    })
  };

  axios.request(options).then(function (response) {
    console.log(response.data);
  }).catch(function (error) {
    console.error(error);
  });
  ```

  ```php PHP
  $curl = curl_init();

  curl_setopt_array($curl, [
    CURLOPT_URL => "https://{yourDomain}/oauth/device/code",
    CURLOPT_RETURNTRANSFER => true,
    CURLOPT_ENCODING => "",
    CURLOPT_MAXREDIRS => 10,
    CURLOPT_TIMEOUT => 30,
    CURLOPT_HTTP_VERSION => CURL_HTTP_VERSION_1_1,
    CURLOPT_CUSTOMREQUEST => "POST",
    CURLOPT_POSTFIELDS => "client_id=%24%7Baccount.clientId%7D&scope=SCOPE&audience=AUDIENCE",
    CURLOPT_HTTPHEADER => [
      "content-type: application/x-www-form-urlencoded"
    ],
  ]);

  $response = curl_exec($curl);
  $err = curl_error($curl);

  curl_close($curl);

  if ($err) {
    echo "Error de cURL #:" . $err;
  } else {
    echo $response;
  }
  ```

  ```python Python
  import http.client

  conn = http.client.HTTPSConnection("")

  payload = "client_id=%24%7Baccount.clientId%7D&scope=SCOPE&audience=AUDIENCE"

  headers = { 'content-type': "application/x-www-form-urlencoded" }

  conn.request("POST", "/{yourDomain}/oauth/device/code", payload, headers)

  res = conn.getresponse()
  data = res.read()

  print(data.decode("utf-8"))
  ```

  ```rb Ruby
  require 'net/http'
  require 'openssl'

  url = URI("https://{yourDomain}/oauth/device/code")

  http.verify_mode = OpenSSL::SSL::VERIFY_NONE

  request = Net::HTTP::Post.new(url)
  request["content-type"] = 'application/x-www-form-urlencoded'
  request.body = "client_id=%24%7Baccount.clientId%7D&scope=SCOPE&audience=AUDIENCE"

  response = http.request(request)
  puts response.read_body
  ```

<div id="device-code-parameters">
  #### Parámetros del código de dispositivo
</div>

Ten en cuenta que, al solicitar un código de dispositivo para llamar a una API personalizada:

* debes incluir un parámetro `audience`
* puedes incluir alcances adicionales compatibles con la API de destino

<Info>
  Si tu aplicación solo quiere un Token de acceso para recuperar información sobre el usuario autenticado, no se requiere ningún parámetro `audience`.
</Info>

| Nombre del parámetro | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `client_id`          | El ID de cliente de tu aplicación. Puedes encontrar este valor en la [Configuración de la aplicación](https://manage.auth0.com/#/Applications/\{yourClientId}/settings).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `scope`              | Los [alcances](/es/docs/get-started/apis/scopes) para los que quieres solicitar autorización. Deben estar separados por espacios. Puedes solicitar cualquiera de los [alcances estándar de OIDC](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims) relacionados con el usuario, como `profile` y `email`, [claims personalizados](/es/docs/secure/tokens/json-web-tokens/json-web-token-claims#custom-claims) que cumplan con un [formato con espacio de nombres](/es/docs/secure/tokens/json-web-tokens/create-custom-claims), o cualquier [alcance compatible con la API de destino](/es/docs/get-started/apis/scopes/api-scopes) (por ejemplo, `read:contacts`). Incluye `openid` para obtener un ID Token o para poder usar el [endpoint /userinfo](https://auth0.com/docs/api/authentication#user-profile) para recuperar la información de perfil del usuario. Incluye `offline_access` para obtener un Token de actualización (asegúrate de que el campo **Allow Offline Access** esté habilitado en la [Configuración de la API](https://manage.auth0.com/#/apis)). Ten en cuenta que este valor debe estar codificado para URL. |
| `audience`           | El identificador único de la API a la que tu aplicación quiere acceder. Usa el valor de **Identifier** en la pestaña [Settings](https://manage.auth0.com/#/apis) de la API que creaste como parte de los prerrequisitos de este tutorial. Ten en cuenta que este valor debe estar codificado para URL.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |

<div id="device-code-response">
  ### Respuesta del código del dispositivo
</div>

Si todo va bien, recibirás una respuesta HTTP 200 con una carga útil que contiene los valores `device_code`, `user_code`, `verification_uri`, `expires_in`, `interval` y `verification_uri_complete`:

```json lines
{
  "device_code": "Ag_EE...ko1p",
  "user_code": "QTZL-MCBW",
  "verification_uri": "https://accounts.acmetest.org/activate",
  "verification_uri_complete": "https://accounts.acmetest.org/activate?user_code=QTZL-MCBW",
  "expires_in": 900,
  "interval": 5
}
```

* `device_code` es el code único del dispositivo. Cuando el usuario vaya a `verification_uri` desde un dispositivo con navegador, este code se vinculará a su sesión.
* `user_code` contiene el code que debe introducirse en `verification_uri` para autorizar el dispositivo.
* `verification_uri` contiene la URL que el usuario debe visitar para autorizar el dispositivo.
* `verification_uri_complete` contiene la URL completa que el usuario debe visitar para autorizar el dispositivo. Esto permite que su aplicación incorpore el `user_code` en la URL, si así lo desea.
* `expires_in` indica el tiempo de validez (en segundos) de `device_code` y `user_code`.
* `interval` indica el intervalo (en segundos) con el que la aplicación debe consultar la URL del token para solicitar uno.

<Info>
  Puede [configurar el conjunto de caracteres, el formato y la longitud del código de usuario generado aleatoriamente](/es/docs/get-started/tenant-settings/configure-device-user-code-settings) en la configuración de su inquilino.

  Para evitar ataques de fuerza bruta, aplicamos los siguientes límites a `user_code`:

  **Longitud mínima**:

  * Letras BASE20: 8 caracteres
  * Números: 9 caracteres

  **Longitud máxima**:

  * 20 caracteres (incluidos los guiones y los espacios, que pueden añadirse como separadores para facilitar la lectura)

  **Tiempo de expiración**:

  * 15 minutos
</Info>

<div id="request-device-activation">
  ## Solicitar la activación del dispositivo
</div>

Una vez que hayas recibido un `device_code` y un `user_code`, debes pedirle al usuario que vaya a la `verification_uri` desde su laptop o smartphone e introduzca el `user_code`:

<Frame>![Solicitar la activación del dispositivo](https://cdn2.auth0.com/docs/1.14550.0/media/articles/flows/guides/device-auth/request-device-activation.png)</Frame>

El `device_code` no está pensado para el usuario directamente y no debe mostrarse durante la interacción para evitar confundirlo.

<Info>
  Al crear una CLI, puedes omitir este paso y abrir de inmediato el navegador con `verification_uri_complete`.
</Info>

<div id="request-tokens">
  ## Solicitar tokens
</div>

Mientras espera a que el usuario active el dispositivo, comience a sondear la URL del token para solicitar un token de acceso. Con el intervalo de sondeo extraído (`interval`) del paso anterior, deberá enviar una solicitud `POST` a la [URL del token](https://auth0.com/docs/api/authentication#device-auth) junto con el `device_code`.

Para evitar errores debidos a la latencia de la red, debe comenzar a contar cada intervalo después de recibir la respuesta de la última solicitud de sondeo.

<div id="token-request-parameters">
  #### Parámetros de la solicitud de token
</div>

| Nombre del parámetro | Descripción                                                                                                                                                                                                                                                                                |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `grant_type`         | Establezca este valor en &quot;urn:ietf:params:oauth:grant-type:device&#95;code&quot;. Se trata de un tipo de concesión de extensión (como se define en la sección 4.5 de [RFC6749](https://tools.ietf.org/html/rfc6749#section-4.5)). Tenga en cuenta que debe estar codificado como URL. |
| `device_code`        | El `device_code` obtenido en el paso anterior de este tutorial.                                                                                                                                                                                                                            |
| `client_id`          | El ID de cliente de su aplicación. Puede encontrar este valor en la [Configuración de la aplicación](https://manage.auth0.com/#/Applications/\{yourClientId}/settings).                                                                                                                    |

<div id="token-responses">
  ### Respuestas del token
</div>

Mientras espera a que el usuario autorice el dispositivo, puede recibir varias respuestas `HTTP 4xx`:

<div id="authorization-pending">
  #### Autorización pendiente
</div>

Verá este error mientras espera a que el usuario realice una acción. Siga sondeando con el intervalo sugerido obtenido en el paso anterior de este tutorial.

```
`HTTP 403`

{
  "error": "authorization_pending",
  "error_description": "..."
}
```

<div id="slow-down">
  #### Reduzca la frecuencia
</div>

Está realizando sondeos con demasiada rapidez. Reduzca la frecuencia y use el intervalo sugerido obtenido en el paso anterior de este tutorial. Para evitar recibir este error debido a la latencia de la red, debe empezar a contar cada intervalo después de recibir la respuesta de la última solicitud de sondeo.

```
`HTTP 429`

{
  "error": "slow_down",
  "error_description": "..."
}
```

<div id="expired-token">
  #### Token expirado
</div>

El usuario no autorizó el dispositivo lo suficientemente rápido, por lo que el `device_code` expiró. La aplicación debe notificar al usuario que el flujo expiró y pedirle que lo reinicie.

<Info>
  El error `expired_token` se devolverá exactamente una vez; después, se devolverá el temido `invalid_grant`. El dispositivo *debe* dejar de realizar sondeos.
</Info>

```
`HTTP 403`

{ 
  "error": "expired_token",
  "error_description": "..."
}
```

<div id="access-denied">
  #### Acceso denegado
</div>

Por último, si se deniega el acceso, recibirá:

```
`HTTP 403`

{
  "error": "access_denied",
  "error_description": "..."
}
```

Esto puede ocurrir por varios motivos, entre ellos:

* el usuario rechazó autorizar el dispositivo
* el servidor de autorización denegó la transacción
* una [Rule](/es/docs/customize/rules) configurada denegó el acceso

<div id="user-authorization">
  ## Autorización del usuario
</div>

El usuario escaneará el código QR o abrirá la página de activación e introducirá el código de usuario:

<Frame>![Introducir el código de usuario](https://cdn2.auth0.com/docs/1.14550.0/media/articles/flows/guides/device-auth/enter-user-code.png)</Frame>

Se mostrará una página de confirmación para que el usuario confirme que se trata del dispositivo correcto:

<Frame>![Confirmar dispositivo](https://cdn2.auth0.com/docs/1.14550.0/media/articles/flows/guides/device-auth/confirm-device.png)</Frame>

El usuario completará la transacción iniciando sesión. Este paso puede incluir uno o varios de los siguientes procesos:

* Autenticar al usuario;
* Redirigir al usuario a un Proveedor de identidad para realizar la autenticación;
* Comprobar si hay sesiones de SSO activas;
* Obtener el consentimiento del usuario para el dispositivo, a menos que ya se haya otorgado previamente.

<Frame>![Autenticar al usuario](https://cdn2.auth0.com/docs/1.14550.0/media/articles/flows/guides/device-auth/user-auth.png)</Frame>

Tras completar correctamente la autenticación y otorgar el consentimiento, se mostrará la pantalla de confirmación:

<Frame>![Confirmación del usuario](https://cdn2.auth0.com/docs/1.14550.0/media/articles/flows/guides/device-auth/user-confirmation.png)</Frame>

En este punto, el usuario ya se ha autenticado y el dispositivo ha sido autorizado.

<div id="receive-tokens">
  ## Recibir tokens
</div>

Mientras el usuario autentica y autoriza el dispositivo, la aplicación del dispositivo sigue consultando la URL del token para solicitar un Token de acceso.

Una vez que el usuario haya autorizado correctamente el dispositivo, recibirás una respuesta `HTTP 200` con una carga útil que contiene los valores `access_token`, `refresh_token` (opcionalmente), `id_token` (opcionalmente), `token_type` y `expires_in`:

```json lines
{
  "access_token":"eyJz93a...k4laUWw",
  "refresh_token":"GEbRxBN...edjnXbL",
  "id_token": "eyJ0XAi...4faeEoQ",
  "token_type":"Bearer",
  "expires_in":86400
}
```

<Warning>
  Debe validar sus tokens antes de guardarlos. Para saber cómo hacerlo, consulte [Validar tokens de acceso](/es/docs/secure/tokens/access-tokens/validate-access-tokens) y [Validar ID Tokens](/es/docs/secure/tokens/id-tokens/validate-id-tokens).
</Warning>

[Tokens de acceso](/es/docs/secure/tokens/access-tokens) se usan para llamar al [endpoint `/userinfo` de la API de autenticación de Auth0](https://auth0.com/docs/api/authentication#get-user-info) o a otra API. Solo podrá usar el Token de acceso para llamar a `/userinfo` si incluyó el scope `openid`. Si llama a su propia API, lo primero que deberá hacer es [verificar el Token de acceso](/es/docs/secure/tokens/access-tokens/validate-access-tokens).

[ID Tokens](/es/docs/secure/tokens/id-tokens) contienen información del usuario que debe [decodificarse y extraerse](/es/docs/secure/tokens/id-tokens#id-token-payload). El `id_token` solo estará presente en la respuesta si incluyó el scope `openid`.

[Tokens de actualización](/es/docs/secure/tokens/refresh-tokens) se usan para obtener un nuevo Token de acceso o ID Token después de que el anterior haya expirado. El `refresh_token` solo estará presente en la respuesta si incluyó el scope `offline_access` y habilitó **Allow Offline Access** para su API en el Dashboard.

<Warning>
  Los Tokens de actualización deben almacenarse de forma segura, ya que permiten que un usuario permanezca autenticado prácticamente de forma indefinida.
</Warning>

<div id="refresh-tokens">
  ## Tokens de actualización
</div>

Ya habrá recibido un [Token de actualización](/es/docs/secure/tokens/refresh-tokens) si ha seguido este tutorial y ha completado lo siguiente:

* configuró su API para permitir el acceso sin conexión
* incluyó el scope `offline_access` al iniciar la solicitud de autenticación a través del endpoint [authorize](https://auth0.com/docs/api/authentication#authorize-application)

Puede usar el Token de actualización para obtener un nuevo Token de acceso. Normalmente, un usuario solo necesitará un nuevo Token de acceso después de que caduque el anterior o cuando obtenga acceso a un nuevo recurso por primera vez. No es una buena práctica llamar al endpoint para obtener un nuevo Token de acceso cada vez que llame a una API, y Auth0 aplica límites de velocidad que restringen la cantidad de solicitudes al endpoint que pueden ejecutarse con el mismo token desde la misma IP.

Para actualizar su token, haga una solicitud `POST` al endpoint `/oauth/token` de la API de autenticación con `grant_type=refresh_token`.

<div id="refresh-token-request-parameters">
  #### Parámetros de la solicitud de Token de actualización
</div>

| Nombre del parámetro | Descripción                                                                                                                                                                                                                                            |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `grant_type`         | Establezca este valor en &quot;refresh&#95;token&quot;.                                                                                                                                                                                                |
| `client_id`          | El ID de cliente de su aplicación. Puede encontrar este valor en la [Configuración de la aplicación](https://manage.auth0.com/#/Applications/\{yourClientId}/settings).                                                                                |
| `client_secret`      | El Secreto del cliente de su aplicación. Puede encontrar este valor en la [Configuración de la aplicación](https://manage.auth0.com/#/Applications/\{yourClientSecret}/settings).                                                                      |
| `refresh_token`      | El Token de actualización que se debe usar.                                                                                                                                                                                                            |
| `scope`              | (Opcional) Una lista de permisos de scope solicitados separada por espacios. Si no se envía, se usarán los alcances originales; de lo contrario, puede solicitar un conjunto reducido de alcances. Tenga en cuenta que debe estar codificado para URL. |

<div id="refresh-token-response">
  ### Respuesta del Token de actualización
</div>

Si todo sale bien, recibirás una respuesta `HTTP 200` con una carga útil que incluye un nuevo `access_token`, `id_token` (opcionalmente), la vigencia del token en segundos (`expires_in`), los valores de `scope` concedidos y `token_type`:

```json lines
{
  "access_token": "eyJ...MoQ",
  "expires_in": 86400,
  "scope": "openid offline_access",
  "id_token": "eyJ...0NE",
  "token_type": "Bearer"
}
```

<Warning>
  Debe validar sus tokens antes de guardarlos. Para saber cómo hacerlo, consulte [Validar tokens de acceso](/es/docs/secure/tokens/access-tokens/validate-access-tokens) y [Validar ID Token](/es/docs/secure/tokens/id-tokens/validate-id-tokens).

<div id="limitations">
  ## Limitaciones
</div>

Para usar el flujo de autorización de dispositivos, los dispositivos deben:

* Admitir Server Name Indication (SNI)
* Tener un [tipo de aplicación de Auth0](/es/docs/get-started/applications) de **Native**
* Tener el [**método de autenticación del endpoint de token**](/es/docs/get-started/applications/application-settings) establecido en **None**
* Ser [compatibles con OIDC](/es/docs/get-started/applications/application-settings#oauth)
* No haber sido creados mediante [Dynamic Client Registration](/es/docs/get-started/applications/dynamic-client-registration)

Además, el flujo de autorización de dispositivos no permite:

* [Conexiones sociales](/es/docs/authenticate/identity-providers) que usen [claves de desarrollador de Auth0](/es/docs/authenticate/identity-providers/social-identity-providers/devkeys), a menos que se use la nueva [experiencia de Universal Login](/es/docs/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience).
* El acceso a parámetros de cadena de consulta desde la página de inicio de sesión alojada o las Rules

Es compatible por completo con el [Draft 15](https://tools.ietf.org/html/draft-ietf-oauth-device-flow-15), excepto para clientes confidenciales.
