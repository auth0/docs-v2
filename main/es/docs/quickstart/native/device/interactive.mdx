    Este tutorial muestra cómo llamar a tu API desde un dispositivo con entrada restringida mediante el Flujo de Autorización de Dispositivos. Te recomendamos que inicies sesión para
    seguir este inicio rápido con ejemplos configurados para tu cuenta.

    <Section id={sections[0].id} title={sections[0].title} stepNumber="1">
      * Registre una aplicación nativa.
      * Asegúrese de que el interruptor **Conforme con OIDC** esté habilitado. Para obtener más información, consulte Autenticación conforme con OIDC.
      * Agregue **Código de dispositivo** a los tipos de concesión de la aplicación. Para obtener más información, consulte Actualizar tipos de concesión.
      * Agregue **Token de actualización** a los tipos de concesión de la aplicación si desea habilitar los tokens de actualización.
      * Configure y habilite al menos una conexión para la
        aplicación.
      * Registre su API en Auth0.
      * Habilite **Permitir acceso sin conexión** si usa tokens de actualización. Para obtener más información, consulte Configuración de la API.
      * Configure los ajustes del código de usuario del dispositivo para definir el
        conjunto de caracteres, el formato y la longitud del código de usuario generado aleatoriamente.
    </Section>

    <Section id={sections[1].id} title={sections[1].title} stepNumber="2">
      Cuando el usuario inicia la aplicación del dispositivo y desea autorizarla, su aplicación debe solicitar un code de dispositivo a la Authentication API de Auth0 para asociarlo con la sesión del usuario.

      Para obtener el código de dispositivo, su aplicación debe llamar al [Authorize endpoint](https://auth0.com/docs/api/authentication#-post-oauth-device-code-) del flujo de autorización de dispositivos de la API de autenticación:

        curl --request post \
          --url 'https://{yourDomain}/oauth/device/code' \
          --header 'content-type: application/x-www-form-urlencoded'

        ```csharp C# lines
        var client = new RestClient("https://{yourDomain}/oauth/device/code");
        var request = new RestRequest(Method.POST);
        request.AddHeader("content-type", "application/x-www-form-urlencoded");

        ```

      Mientras la aplicación de su dispositivo espera a que el usuario la active, debe llamar de forma intermitente al [endpoint POST /oauth/token de la API de autenticación](https://auth0.com/docs/api/authentication#-post-oauth-token-)
      y gestionar la respuesta adecuadamente.

      <Info>
        Asegúrese de que la aplicación de su dispositivo espere la duración de `interval` (en segundos) o hasta
        que reciba una respuesta exitosa, lo que sea mayor, para evitar problemas de latencia de red.
      </Info>

      ```bash
      curl --request POST \ 
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=urn:ietf:params:oauth:grant-type:device_code \
        --data device_code=AUTH0_SCOPES \
        --data 'client_id={yourClientId}'
      ```

    <Section id={sections[6].id} title={sections[6].title} stepNumber="7">
      Una vez que el usuario autoriza la aplicación del dispositivo, esta recibe una respuesta HTTP 200 y la siguiente carga:

      ```json lines
      {
        "access_token": "eyJz93a...k4laUWw",
        "refresh_token": "GEbRxBN...edjnXbL",
        "id_token": "eyJ0XAi...4faeEoQ",
        "token_type": "Bearer",
        "expires_in": 86400
      }
      ```

      <Info>
        Debe validar sus tokens antes de guardarlos. Para saber cómo hacerlo, lea Validar tokens de acceso y Validar tokens de ID.
      </Info>

      Los tokens de acceso se usan para llamar al [endpoint Get User Info de la API de autenticación](https://auth0.com/docs/api/authentication#get-user-info) (si su aplicación del dispositivo
      solicitó el scope `openid`) o a la API especificada por el parámetro `audience`.
      Si llama a su propia API, su aplicación del dispositivo debe verificar el token de acceso antes de usarlo.

      Los tokens de ID contienen información del usuario que debe [decodificarse y extraerse](/es/docs/secure/tokens/id-tokens#id-token-payload). La API de autenticación
      solo devuelve un `id_token` si su aplicación del dispositivo solicitó el scope `openid`.

      Los tokens de actualización se usan para obtener un nuevo token de acceso o
      token de ID después de que haya expirado el anterior. La API de autenticación solo devuelve un `refresh_token` si
      la configuración **Allow Offline Access** está habilitada para la API especificada por el parámetro `audience`
      y su aplicación del dispositivo solicitó el scope `offline_access`.
    </Section>

    <Section id={sections[7].id} title={sections[7].title} stepNumber="8">
      Para llamar a tu API, la aplicación de tu dispositivo debe enviar el token de acceso como token Bearer en el
      encabezado `Authorization` de tu solicitud HTTP.

      ```bash lines
      curl --request GET \
        --url https://myapi.com/api \
        --header 'authorization: Bearer AUTH0_API_ACCESS_TOKEN' \
        --header 'content-type: application/json'
      ```
    </Section>

    <Section id={sections[8].id} title={sections[8].title} stepNumber="9">
      Para obtener un nuevo token de acceso para un usuario, la aplicación del dispositivo puede llamar al [endpoint POST /oauth/token de la Authentication API](https://auth0.com/docs/api/authentication#-post-oauth-token-) con el
      parámetro `refresh_token`.

      ```bash lines
      curl --request POST \
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=refresh_token \
        --data 'client_id={yourClientId}' \
        --data 'client_secret={yourClientSecret}' \
        --data refresh_token=AUTH0_REFRESH_TOKEN
      ```

      Si la solicitud se realiza correctamente, la aplicación de su dispositivo recibe una respuesta HTTP 200 con la siguiente carga útil:

      ```json lines
      {
        "access_token": "eyJ...MoQ",
        "expires_in": 86400,
        "scope": "openid offline_access",
        "id_token": "eyJ...0NE",
        "token_type": "Bearer"
      }
      ```

      Para obtener más información sobre los tokens de actualización, consulta [Tokens de actualización](https://auth0.com/docs/secure/tokens/refresh-tokens).
    </Section>

    <Section id={sections[9].id} title={sections[9].title} stepNumber="10">
      Los registros del Tenant se crean para cualquier interacción que se produzca y pueden usarse para solucionar problemas.

      ### Respuestas del token

      Mientras esperas a que el usuario autorice el dispositivo, es posible que recibas varias respuestas HTTP 4xx.

      #### Autorización pendiente

      Verás este error mientras esperas a que el usuario realice una acción. Sigue consultando con el `interval` sugerido que recuperaste en el paso anterior de este tutorial.

      ```json lines
      `HTTP 403`
      {
      "error": "authorization_pending",
      "error_description": "..."
      }
      ```

      #### Reduzca la frecuencia

      Está realizando sondeos con demasiada frecuencia. Reduzca la frecuencia y use el intervalo sugerido obtenido en el paso anterior de este
      tutorial. Para evitar este error debido a la latencia de la red, debe empezar a contar cada intervalo después de
      recibir la respuesta a la última solicitud de sondeo.

      ```json lines
      `HTTP 429`
      {
      "error": "slow_down",
      "error_description": "..."
      }
      ```

      #### Token expirado

      El usuario no ha autorizado el dispositivo con la suficiente rapidez, por lo que `device_code` ha expirado. Su
      aplicación debe notificarle que el flujo ha expirado y pedirle que lo reinicie.

      <Info>
        El error `expired_token` se devuelve exactamente una vez. Después de eso, el endpoint devuelve el
        error `invalid_grant`.
      </Info>

      ```json lines
      `HTTP 403`
      {
      "error": "expired_token",
      "error_description": "..."
      }
      ```

      #### Acceso denegado

      Si se deniega el acceso, recibirá:

      ```json lines
      `HTTP 403`
      {
      "error": "access_denied",
      "error_description": "..."
      }
      ```

      Esto puede ocurrir por varios motivos, entre ellos:

      * El usuario rechazó autorizar el dispositivo.
      * El servidor de autorización denegó la transacción.
      * Una Action configurada denegó el acceso.
    </Section>

    <Section id={sections[11].id} title={sections[11].title} stepNumber="12">
      Para usar el flujo de autorización de dispositivos, una aplicación para dispositivos debe:

      * Ser compatible con Server Name Indication (SNI)
      * Ser una aplicación nativa
      * Tener el [método de autenticación](/es/docs/secure/application-credentials#application-authentication-methods) configurado como **None**
      * Ser [conforme con OIDC](/es/docs/get-started/applications/application-settings#oauth)
      * No haberse creado mediante Dynamic Client Registration

      Además, el flujo de autorización de dispositivos no permite lo siguiente:

      * Conexiones sociales que usen claves de desarrollador de Auth0, a menos que utilice New Universal Login Experience
      * Acceder a parámetros de cadena de consulta desde una página de inicio de sesión alojada o Actions.
    </Section>
