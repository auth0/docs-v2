      Para usar los servicios de Auth0, deberá tener una aplicación configurada en el Auth0 Dashboard. La aplicación de Auth0 es
      donde configurará cómo quiere que funcione la autenticación para el proyecto que está desarrollando.

      ### Configurar una aplicación

      Use el selector interactivo para crear una nueva aplicación de Auth0 o seleccionar una aplicación existente que represente
      el proyecto con el que desea realizar la integración. A cada aplicación de Auth0 se le asigna un ID de cliente alfanumérico y único
      que el código de su aplicación usará para llamar a las API de Auth0 a través del SDK.

      Cualquier configuración que realice con esta guía de inicio rápido se actualizará automáticamente en su aplicación en el [Dashboard](https://manage.auth0.com/#/), que es donde
      podrá administrar sus aplicaciones en el futuro.

      Si prefiere ver una configuración completa, puede consultar una aplicación de ejemplo.

      ### Configurar URLs de callback

      Una URL de callback es una URL de su aplicación a la que desea que Auth0 redirija a los usuarios después de que se hayan
      autenticado. Si no se configura, los usuarios no volverán a su aplicación después de iniciar sesión.

      <Info>
        Si está siguiendo nuestro proyecto de ejemplo, establézcala en
        `http://localhost:3000/auth/auth0/callback`.
      </Info>

      ### Configurar URLs de cierre de sesión

      Una URL de cierre de sesión es una URL de su aplicación a la que desea que Auth0 redirija a los usuarios después de que hayan cerrado
      sesión. Si no se configura, los usuarios no podrán cerrar sesión desde su aplicación y recibirán un error.

      <Info>
        Si está siguiendo nuestro proyecto de ejemplo, establézcala en `http://localhost:3000`.
      </Info>

      ### Configurar orígenes web permitidos

      Un origen web permitido es una URL a la que desea permitir el acceso a su flujo de autenticación. Debe
      incluir la URL de su proyecto. Si no se configura correctamente, su proyecto no podrá renovar de forma silenciosa
      los tokens de autenticación, por lo que se cerrará la sesión de sus usuarios la próxima vez que visiten su aplicación o actualicen una
      página.

      <Info>
        Si está siguiendo nuestro proyecto de ejemplo, establézcala en `http://localhost:3000`.
      </Info>

      Use `omniauth-auth0`, una [estrategia personalizada de OmniAuth](https://github.com/intridea/omniauth#omniauth-standardized-multi-provider-authentication), para gestionar el flujo de autenticación.

      Agregue las siguientes dependencias a su `Gemfile`:

      ```gem lines
      gem 'omniauth-auth0', '~> 3.0'
      gem 'omniauth-rails_csrf_protection', '~> 1.0' # evita solicitudes de autenticación falsificadas
      ```

      Una vez que hayas agregado las gemas, instálalas con `bundle install`.

      Cree un controlador de Auth0 para gestionar la devolución de llamada de autenticación, la acción `logout` y los métodos para
      construir la URL de cierre de sesión.

      Ejecute el comando:
      `rails generate controller auth0 callback failure logout --skip-assets --skip-helper --skip-routes --skip-template-engine`.

      Dentro del método callback, asigne a la sesión activa el hash con la información del usuario, devuelto como
      `request.env['omniauth.auth']`.

      Para configurar el cierre de sesión, borre todos los objetos almacenados en la sesión llamando al método `reset_session`
      dentro de la acción `logout`. Luego, redirija al endpoint de cierre de sesión de Auth0. Para obtener más información sobre
      `reset_session`, consulte la [documentación de ActionController de Ruby on Rails](http://api.rubyonrails.org/classes/ActionController/Base.html#M000668).

    <Section id={sections[8].id} title={sections[8].title} stepNumber="9" isSingleColumn>
      Para mostrar el perfil del usuario, la aplicación debe proporcionar una ruta protegida. Puede usar un [Concern](https://guides.rubyonrails.org/getting_started.html#using-concerns) para controlar el acceso a rutas que pueden compartirse entre varios
      controladores. El concern debe redirigir automáticamente a Auth0 cuando el usuario no esté autenticado. De lo contrario, el
      concern debe devolver el perfil del usuario actual.

      Una vez que tenga un Concern, inclúyalo en cualquier controlador que requiera que el usuario haya iniciado sesión. Luego podrá acceder al
      usuario desde la sesión `session[:userinfo]`, como en el siguiente ejemplo:

      ```ruby lines
      class DashboardController < ApplicationController
      include Secured
      def show
      @user = session[:userinfo]

      end
      end
      ```

      Una vez que el usuario se haya cargado desde la sesión, úsalo para mostrar información en el frontend:

      ```html lines
      <div>
      <p>Normalized User Profile:<%= JSON.pretty_generate(@user[:info])%></p>
      <p>Full User Profile:<%= JSON.pretty_generate(@user[:extra][:raw_info])%></p>
      </div>
      ```

      <Note>
        ##### Verificación

        Agrega el concern `Secured` a tu aplicación y luego inclúyelo en el controlador que requiera un
        usuario autenticado para acceder. Verifica que un usuario autenticado tenga acceso a las acciones de ese
        controlador y que los usuarios no autenticados sean redirigidos a Auth0 para autenticarse.
      </Note>
