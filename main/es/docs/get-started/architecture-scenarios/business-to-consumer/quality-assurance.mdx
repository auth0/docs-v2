---

El aseguramiento de la calidad es importante para identificar problemas antes de que afecten a sus clientes y, según la naturaleza de su proyecto, hay varios tipos de pruebas de aseguramiento de la calidad que querrá considerar como parte de su integración con Auth0:

* ¿Su aplicación es fácil de entender y usar, incluso para personas con alguna discapacidad?
* ¿Su aplicación necesita funcionar en distintos navegadores y dispositivos?
* ¿Su aplicación necesita funcionar en entornos multinacionales o internacionales?
* ¿Cómo responderá su aplicación cuando se someta a cargas de producción inesperadas?
* ¿Cómo puede asegurarse de que su aplicación esté protegida frente a vulnerabilidades de seguridad?

Auth0 [Universal Login](/es/docs/authenticate/login/auth0-universal-login) y los widgets de interfaz de usuario asociados (como [Lock](/es/docs/libraries/lock)) ya se han diseñado y creado siguiendo las mejores prácticas de usabilidad y accesibilidad, y ofrecen compatibilidad inmediata y probada con una amplia variedad de [navegadores y dispositivos](/es/docs/troubleshoot/customer-support/product-support-matrix). También se ofrece soporte listo para usar para la [internacionalización](/es/docs/customize/internationalization-and-localization) (I18N), con extensibilidad integrada diseñada para situaciones personalizadas de varios idiomas y localización (L10N).

Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la [integración](#integration-testing) entre sus aplicaciones y Auth0, y para realizar [pruebas unitarias](#unit-testing) de módulos de extensibilidad individuales (como [Rules](/es/docs/customize/rules/debug-rules), [Hooks](/es/docs/customize/hooks/update-hooks) y scripts personalizados de base de datos). También se proporciona orientación sobre la [política de pruebas de penetración](/es/docs/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 para ayudarle a realizar pruebas de vulnerabilidades de seguridad, así como sobre cómo las pruebas con [Mock](#mock-testing) pueden aprovecharse junto con nuestra [política de pruebas de carga](/es/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) para ayudar a garantizar que sus aplicaciones funcionen correctamente bajo cargas inesperadas.

<div id="unit-testing">
  ## Pruebas unitarias
</div>

El objetivo de las pruebas unitarias es probar unidades individuales de código. Si crea código personalizado dentro de Auth0 en forma de Rules, Hooks y/o scripts de Custom DB, debería considerar usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar su código. Las empresas que han tenido más éxito con Auth0 han comprobado que resulta útil ejecutar estas pruebas unitarias antes de la [implementación automática](/es/docs/get-started/architecture-scenarios/business-to-consumer/deployment) de la configuración del inquilino de Auth0 y los recursos relacionados.

<div id="integration-testing">
  ## Pruebas de integración
</div>

Una práctica recomendada es configurar distintos inquilinos para desarrollo, pruebas y producción, como se explica en la guía de Arquitectura sobre [compatibilidad con SDLC](/es/docs/get-started/architecture-scenarios/business-to-consumer/architecture#sdlc-support). Auth0 le permite configurar variables disponibles desde la [extensibilidad](/es/docs/customize/extensions) personalizada; pueden considerarse variables de entorno para su inquilino de Auth0. En lugar de codificar de forma fija referencias que cambian al mover código entre los entornos de desarrollo, pruebas y producción, puede usar un nombre de variable configurado en el inquilino y al que haga referencia el código de extensibilidad personalizado. Esto facilita que el mismo código personalizado funcione, sin cambios, en distintos inquilinos, ya que puede hacer referencia a variables que se completarán con valores específicos del inquilino en el momento de la ejecución:

* Para usar variables en Rules, consulte cómo [configurar valores](/es/docs/customize/rules/configuration)
* Para usar variables en Hooks, consulte cómo configurar [secretos](/es/docs/customize/hooks/hook-secrets) en el editor
* Para usar variables en Actions, consulte Explorar flujos y desencadenadores
* Para usar variables en scripts de Custom DB, consulte los [parámetros de configuración](/es/docs/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Una práctica recomendada es usar variables para almacenar valores específicos del inquilino, así como cualquier secreto confidencial que no deba exponerse en su código personalizado. Si su código personalizado se implementa en GitHub, usar una variable específica del inquilino evita exponer valores confidenciales a través de su repositorio de GitHub.
</Info>

<div id="test-automation">
  ### Automatización de pruebas
</div>

Puede automatizar su proceso general de compilación incorporando la automatización de implementaciones, además de la automatización de pruebas. Esto puede utilizarse para implementar nuevas versiones de la configuración y/o del código personalizado en Auth0 y ejecutar pruebas automatizadas. Si las pruebas detectan algún fallo, las capacidades de automatización de implementaciones pueden utilizarse para volver a la última versión funcional. Para obtener más información, consulte la [guía sobre automatización de implementaciones](/es/docs/get-started/architecture-scenarios/business-to-consumer/deployment).

<div id="mock-testing">
  ## Pruebas con mocks
</div>

Como equilibrio entre la [política de pruebas de carga](/es/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y la necesidad de realizar pruebas de carga, es habitual que los clientes de Auth0 simulen los endpoints de Auth0. Esta práctica es valiosa para garantizar que su aplicación funcione con las interfaces previstas sin tener que limitar sus pruebas, y puede apoyarse en herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/).
