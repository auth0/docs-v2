Es importante definir desde el principio cómo se registrarán los usuarios, ya que las decisiones que tome aquí influirán en muchas de las decisiones que deberá tomar más adelante. Hemos observado que existe un conjunto habitual de patrones sobre cómo se incorporan los usuarios a un sistema, así como varios aspectos que conviene tener en cuenta al diseñar el flujo de trabajo.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Aunque Auth0 admite numerosos flujos de trabajo, los flujos web que usan Auth0 [Universal Login](/es/docs/authenticate/login/auth0-universal-login) para el registro se consideran una práctica recomendada tanto del sector como de Auth0, ya que ofrecen una funcionalidad óptima y la mejor seguridad.
</Callout>

Auth0 admite el registro de usuarios mediante distintos [proveedores de identidad](/es/docs/authenticate/identity-providers). Durante el registro, Auth0 aprovisiona el perfil del usuario para que contenga la información de su cuenta. Hay varios aspectos que se deben tener en cuenta al evaluar la funcionalidad y el flujo de trabajo:

* ¿Debe usar Auth0 como almacén de identidades?
* ¿Puede usar su propio almacén de identidades heredado con Auth0?
* ¿Cómo migra las identidades de los usuarios desde su almacén de identidades a Auth0?
* ¿Pueden sus usuarios registrarse con sus cuentas sociales existentes, como Google y Facebook?

Auth0 proporciona almacenamiento de identidades listo para usar que puede aprovechar para almacenar credenciales de usuario de forma segura. Consulte [autorregistro](#self-sign-up) para obtener más información. Si ya cuenta con un almacén de identidades heredado y desea delegar su administración, las capacidades de [migración de usuario](#user-migration) le ofrecen varias opciones para hacerlo.

Como alternativa, si tiene que mantener su almacén de identidades heredado —quizá porque tiene aplicaciones que aún no está listo para migrar o que no se pueden migrar—, puede usar la capacidad de [proxy del almacén de identidad](#identity-store-proxy). Permitir que sus clientes usen su propia identidad también es una opción atractiva y, aunque vemos que nuestros clientes no suelen hacerlo al principio, puede usar la capacidad de [registro social](#social-sign-up) para ofrecerla.

<div id="user-migration">
  ## Migración de usuarios
</div>

Además de alojar el [Perfil de usuario](/es/docs/get-started/architecture-scenarios/business-to-consumer/profile-management), Auth0 también puede tanto [actuar como proxy](#identity-store-proxy) de tu almacén de identidades heredado como proporcionar un reemplazo seguro alojado en Auth0. Ambas capacidades se admiten mediante el uso de [Conexiones de base de datos](/es/docs/authenticate/database-connections) de Auth0. Si decides usar Auth0 como reemplazo de tu almacén de identidades heredado, puedes [migrar usuarios](/es/docs/manage-users/user-migration) de una sola vez con una migración masiva o de forma progresiva con la migración automática.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los clientes suelen optar por un enfoque de dos etapas para la migración de usuarios: primero usan la migración automática para migrar tantos usuarios como sea posible y luego la migración masiva para los usuarios restantes. Consulta [Escenarios de migración de usuarios](/es/docs/manage-users/user-migration/user-migration-scenarios) para obtener más información.
</Callout>

Se prefiere la migración automática, ya que permite migrar a los usuarios de forma individual y también conservar su contraseña actual en casi todos los casos. Para la migración masiva, recomendamos usar la [Management API](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports) en lugar de la [extensión User Import/Export](/es/docs/manage-users/user-migration/user-import-export-extension) salvo en los casos más sencillos, ya que la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip> ofrece mayor flexibilidad y control.

Con la migración masiva, los usuarios normalmente tienen que **restablecer su contraseña una vez completada la migración**, salvo que las contraseñas estén almacenadas con hash en tu almacén de identidades heredado mediante bcrypt (o puedas generarlas en formato bcrypt). En este caso, es posible que puedas usar la migración masiva y **conservar las contraseñas de los usuarios** como parte del proceso, según el algoritmo bcrypt y la cantidad de rondas de salt utilizadas. Consulta [Ejemplos de esquema de importación masiva de bases de datos](/es/docs/manage-users/user-migration/bulk-user-import-schema) para obtener más información.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las llamadas a la Management API están sujetas a la [política de limitación de tasa de Auth0](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy). Debes tenerlo en cuenta y, para facilitarlo, Auth0 recomienda por lo general usar el [SDK de Auth0](/es/docs/libraries) adecuado para tu entorno de desarrollo en lugar de llamar directamente a nuestras API.
</Callout>

<div id="identity-store-proxy">
  ### Proxy del almacén de identidades
</div>

Los tipos de conexión de base de datos de Auth0 también se pueden configurar para actuar como proxy de un almacén de identidades existente (heredado). Si necesita conservar las identidades de los usuarios definidas en su propio almacén heredado —por ejemplo, si tiene una o más aplicaciones críticas para la empresa que no puede migrar a Auth0, pero que aún necesitan acceso a estas identidades—, puede integrarse fácilmente con Auth0. Consulte [Autentique usuarios con su base de datos](/es/docs/authenticate/database-connections/custom-db) para obtener más información.

<div id="self-sign-up">
  ## Autorregistro
</div>

El autorregistro aprovecha las [Conexiones de base de datos](/es/docs/authenticate/database-connections) de Auth0 para almacenar el ID del usuario, la contraseña y la información de identidad del nombre de usuario (opcional) recopilada de los nuevos usuarios durante el proceso de registro. Las políticas de las conexiones de base de datos que rigen aspectos como la [longitud mínima del nombre de usuario](/es/docs/authenticate/database-connections/require-username) o la [seguridad y complejidad de la contraseña](/es/docs/authenticate/database-connections/password-options) se pueden configurar a través de <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 [Universal Login](/es/docs/authenticate/login/auth0-universal-login), así como widgets de Auth0 como [Lock](https://auth0.com/lock), se integran con las Conexiones de base de datos para ofrecer, de forma predeterminada, una funcionalidad integral de interfaz de usuario para el registro. Estos componentes de UI son totalmente reactivos y, gracias a sus amplias opciones de configuración y personalización, puede implementar funcionalidades de autorregistro de usuarios, así como de inicio de sesión.
</Callout>

<div id="social-sign-up">
  ## Registro social
</div>

El registro social equivale a iniciar sesión mediante [autenticación social](/es/docs/get-started/architecture-scenarios/business-to-consumer/authentication); aquí no existe una distinción *per se*, ya que el [perfil](/es/docs/get-started/architecture-scenarios/business-to-consumer/profile-management) del usuario se crea automáticamente la primera vez que inicia sesión con una identidad social.
