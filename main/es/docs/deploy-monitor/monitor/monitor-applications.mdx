Sus aplicaciones y servicios dependen de Auth0. Monitorizar el estado de Auth0 le permite informar a los clientes sobre errores concretos o reducir el impacto en los usuarios en caso de que surja algún problema con Auth0.

Hay muchas formas de monitorizar Auth0, y cada enfoque complementa a los demás. Debe elegir la más adecuada en función de sus necesidades y de sus posibilidades de inversión.

<div id="synthetic-transactions">
  ## Transacciones sintéticas
</div>

El enfoque más sencillo para supervisar Auth0:

* Configure una solicitud periódica para realizar una transacción de autenticación.
* Si la solicitud se completa correctamente, Auth0 funciona bien.
* Si la solicitud falla, esto puede indicar:

  * un problema con Auth0
  * un problema específico del inquilino usado para la transacción sintética
  * o simplemente una solicitud fallida puntual.

Para las transacciones sintéticas, use un entorno lo más parecido posible a la configuración de su inquilino de producción. Incluso podría usar el mismo inquilino de producción. Dado que configurar transacciones sintéticas con flujos de redirección y proveedores externos puede resultar complicado, se recomienda usar [Resource Owner Password Grant](/es/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow). Este flujo no implica redirecciones del navegador ni requiere una interfaz de usuario.

Si usa [rules](/es/docs/customize/rules), [conexiones de base de datos personalizadas](/es/docs/authenticate/database-connections/custom-db) u otros puntos de extensibilidad, las transacciones sintéticas deben configurarse para usar las rules o los scripts de Custom DB, a fin de garantizar que ese aspecto del sistema funcione correctamente.

Herramientas como [Pingdom](http://pingdom.com/) facilitan la configuración de transacciones sintéticas.

<div id="check-period">
  ### Intervalo de comprobación
</div>

Recomendamos ejecutar transacciones sintéticas a intervalos de un minuto. Con este enfoque sencillo, esa frecuencia no consumirá gran parte de la cuota del límite de tasa de Auth0 y, al mismo tiempo, proporcionará respuestas a tiempo.

<div id="limitations-of-synthetic-transactions">
  ### Limitaciones de las transacciones sintéticas
</div>

Las transacciones sintéticas son una forma sencilla y económica de supervisar el estado de un inquilino de Auth0. Sin embargo, tienen algunas limitaciones:

* Las transacciones sintéticas no representan la experiencia real de sus usuarios finales. En su lugar, proporcionan una métrica aproximada.
* Es posible que las transacciones sintéticas no utilicen los mismos flujos que sus usuarios.
* Carecen de atomicidad (normalmente se ejecutan una vez por minuto) y no informan sobre los errores que podrían haber experimentado sus usuarios finales.

Si le interesa obtener datos más detallados, consulte [Seguimiento de errores](#error-tracking) y [Métricas y Logs](#metrics-and-logs).

<div id="error-tracking">
  ## Seguimiento de errores
</div>

Este enfoque es útil para hacer seguimiento de los errores en las llamadas existentes a Auth0. Consiste en reportar errores cada vez que falla una llamada a Auth0. [Sentry](https://sentry.io/) es una herramienta que se usa habitualmente en estos casos y que funciona tanto en entornos de frontend como de backend.

Este enfoque es útil porque te permite conocer los errores reales que están experimentando tus usuarios finales. Sin embargo, como solo haces seguimiento de los errores (y no de todas las solicitudes), no es posible obtener una idea precisa de cuántos usuarios finales se ven afectados: ¿es el 1 % o el 5 %? Además, no requiere que configures una &quot;llamada sintética&quot; independiente, que podría consumir parte de tu cuota del límite de tasa, especialmente si está mal configurada.

<div id="metrics-and-logs">
  ## Métricas y logs
</div>

Este enfoque es útil si las llamadas a Auth0 se realizan desde un backend que controlas. Este es el caso de:

* La mayoría de las llamadas a la [Management API de Auth0](https://auth0.com/docs/api/management/v2)
* Las llamadas a la [Authentication API](https://auth0.com/docs/api/authentication) desde Regular Web Applications y aplicaciones de máquina a máquina ([más información sobre los tipos de cliente](/es/docs/get-started/applications)).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Algunos productos de observabilidad también te permiten reportar métricas/logs desde aplicaciones frontend y móviles.
</Callout>

El enfoque consiste en usar métricas y/o logs para realizar un seguimiento de las tasas de error en las llamadas a Auth0. Las métricas/logs reportan tasas de error que describen con precisión lo que experimentan los usuarios finales, sin necesidad de configurar una &quot;llamada sintética&quot; independiente, que podría consumir parte de la cuota asociada a tu límite de tasa, especialmente si está mal configurada.

<div id="alerting">
  ## Alertas
</div>

Independientemente del enfoque de monitorización que utilice, es habitual recibir una alerta o una notificación cuando se producen errores con una frecuencia determinada. Esa frecuencia depende de su aplicación.

Cuando su equipo reciba una alerta de Auth0, le recomendamos añadir un enlace a [consultar](/es/docs/deploy-monitor/monitor/check-auth0-status) en la carga útil/guía práctica de la alerta. Esto les permitirá consultar rápidamente el canal oficial de estado de Auth0 para ver si el problema proviene de Auth0 o de su aplicación/servicio.
