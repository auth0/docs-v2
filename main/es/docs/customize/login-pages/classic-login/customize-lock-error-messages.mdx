<Warning>
  En este momento, los esfuerzos de desarrollo activo de Auth0 se centran en Universal Login, y Classic Login ya no se actualiza. Para personalizar tus páginas de inicio de sesión, Auth0 recomienda [Personalizaciones avanzadas para Universal Login](/es/docs/customize/login-pages/advanced-customizations).
</Warning>

Puedes personalizar los mensajes de error que se mostrarán en determinadas situaciones mediante la opción `languageDictionary`. La lista completa de los campos de `languageDictionary` disponibles para personalizar se encuentra en el [archivo del diccionario en inglés para Lock](https://github.com/auth0/lock/blob/master/src/i18n/en.js) del repositorio de GitHub. A continuación, se muestra un ejemplo de algunos mensajes de error personalizados:

{/* codeblockOld.header.login.logInButton codeblockOld.header.login.configureSnippet */}

export const codeExample = `// Ejemplos de mensajes de error personalizados en la opción languageDictionary
var options = {
  languageDictionary: {
    error: {
      login: {
        "lock.invalid_email_password": "Mensaje personalizado sobre credenciales no válidas",
        "lock.network": "Mensaje personalizado que indica un error de red y sugiere al usuario que compruebe la conexión",
        "lock.unauthorized": "Mensaje personalizado sobre un problema de permisos",
        "too_many_attempts": "Mensaje personalizado que indica que el usuario ha fallado al iniciar sesión demasiadas veces."
      },
      signUp: {
        "invalid_password": "Mensaje personalizado que indica que una contraseña no era válida",
        "user_exists": "Mensaje personalizado que indica que ya existe un usuario"
      }
    }
  }
};

// Inicialización de Auth0Lock
var lock = new Auth0Lock(
  '{yourClientId}',
  '{yourDomain}',
  options
);`;

<div id="custom-errors-in-rules">
  ## Errores personalizados en Rules
</div>

Si devuelve códigos de error personalizados desde una [rule](/es/docs/customize/rules) o un script personalizado de base de datos, puede manejar errores personalizados:

* En la URL de redirección de su aplicación, leyendo los parámetros de la cadena de consulta `error` y `error_mesage`.
* Redirigiendo al usuario de nuevo a sus páginas alojadas con un mensaje de error personalizado y mostrando ese mensaje con un [mensaje flash](/es/docs/libraries/lock/lock-api-reference).
