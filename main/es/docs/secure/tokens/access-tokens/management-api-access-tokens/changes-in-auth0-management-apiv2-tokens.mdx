title: Cambios en los tokens de Auth0 Management APIv2
---

Hace algún tiempo, cambiamos el proceso para obtener un token de Management APIv2. En este artículo se explica qué cambió, por qué se hizo y cómo puede evitarlo (no se recomienda).

<div id="what-changed-and-why">
  ## Qué cambió y por qué
</div>

<div id="the-user-experience">
  ### La experiencia de usuario
</div>

Hasta hace poco, se podía generar un token de Management APIv2 directamente desde el API Explorer de <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip>. Seleccionaba los alcances según el endpoint que quería invocar y obtenía un token desde esa misma página.

Ese método era muy sencillo, pero también muy inseguro. Por eso lo cambiamos.

El nuevo método usa el [flujo de credenciales de cliente](/es/docs/get-started/authentication-and-authorization-flow/client-credentials-flow). Para saber cómo usar el nuevo proceso, consulte [Tokens de acceso de Management API](/es/docs/secure/tokens/access-tokens/management-api-access-tokens).

#### Por qué cambió esto

Para generar el token, la Management API requería acceso a tu <Tooltip tip="Secreto del cliente: secreto que utiliza un cliente (aplicación) para autenticarse con el Servidor de autorización; solo deberían conocerlo el cliente y el Servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/es/docs/glossary?term=Client+Secret">Secreto del cliente</Tooltip> global (utilizado para firmar el token). Esta es información que no debería exponerse en navegadores web.

Además, el API Explorer no tiene forma de realizar la autorización. Esto significa que, si un usuario pudiera iniciar sesión y acceder al API Explorer, podría generar un token con cualquier scope, incluso si no tuviera permiso para contar con ese scope.

La nueva implementación no plantea esos riesgos. Una vez que completes la configuración inicial, puedes obtener un token visitando el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o haciendo una simple solicitud `POST` al [endpoint `/oauth/token` de nuestra Authentication API](https://auth0.com/docs/api/authentication#client-credentials).

Sin embargo, en lo que respecta al proceso manual, entendemos que cambiar de pantalla no siempre ofrece la mejor experiencia de usuario, por lo que estamos buscando formas de hacer que el nuevo flujo sea más intuitivo.

<div id="the-validity-period">
  ### Período de validez
</div>

Con el flujo anterior, los tokens no caducaban. Con el nuevo flujo, todos los tokens de Management APIv2 caducan de forma predeterminada a las 24 horas.

<div id="why-this-changed">
  #### Por qué ha cambiado esto
</div>

Tener un token que nunca caduca puede ser muy arriesgado si un atacante se hace con él. Si el token caduca en unas pocas horas, el atacante solo dispone de un breve margen de tiempo para acceder a tus recursos protegidos.

Para obtener un token, debes seguir únicamente el proceso descrito en [Management API Access Tokens](/es/docs/secure/tokens/access-tokens/management-api-access-tokens).
