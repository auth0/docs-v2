LoginEmailVerificationMembers
description Login Email Verification 画面を操作するために使用できるメンバー (プロパティとメソッド) を定義します。
この画面はメールアドレス確認プロセスの重要な一部で、通常はログイン時に、ユーザーがワンタイム code を入力して
メールアドレスの所有権を証明する必要があります。SDK では、この code の送信や
新しい code の要求を行えます。

この画面の `universal_login_context` (`window.universal_login_context`) には、次の内容が含まれます。

* `client`: Auth0 アプリケーションに関する情報。
* `organization` (省略可) : 認証対象が特定の組織である場合の詳細。
* `prompt`: 現在の認証プロンプトのコンテキスト (例: `login`) 。
* `screen`: UI テキストと一般的な画面情報。`login-email-verification` では、標準のものを除き、画面固有の `data` フィールドは特に定義されていません。
* `transaction`: 進行中のトランザクションの詳細。state や、前回までの試行で発生したエラー (例: `invalid-code`) が含まれます。

````ts Example
export interface LoginEmailVerificationMembers extends BaseMembers {
  /**
   * ユーザーが入力したメールアドレス確認コードを送信します。
   * このアクションは、ユーザーがメールで受け取ったコードを入力し、
   * 「続行」または「確認」ボタンをクリックする操作に対応します。SDK はこのコードを
   * Auth0 の `/u/login-email-verification` エンドポイントに POST します。
   *
   * コードが有効な場合、Auth0 は通常ユーザーを認証フローの次のステップにリダイレクトします。
   * コードが無効、期限切れ、またはその他のエラーが発生した場合、
   * Auth0 は通常 login-email-verification 画面を再レンダリングし、
   * SDK コンテキスト内の `transaction.errors` 配列が失敗の詳細
   * （例: エラーコード `invalid-code`）で更新されます。
   *
   * @param {ContinueWithCodeOptions} payload - ユーザーが入力した `code` 文字列を含むオブジェクト。
   *                                            拡張機能のために任意の `CustomOptions` を含めることもできます。
   * @returns {Promise<void>} フォームの送信が開始されたときに解決される Promise。
   *                          リダイレクトまたはページの再レンダリングが一般的な結果であるため、
   *                          解決時にデータを直接返しません。
   * @throws {Error} `payload.code` が指定されていないか文字列でない場合、
   *                 または `FormHandler` が送信中に回復不能な問題（例: ネットワークエラー）に
   *                 遭遇した場合にエラーをスローします。
   *                 Auth0 からの検証エラー（無効なコードなど）は JavaScript エラーとしてスローされず、
   *                 操作後に `this.transaction.errors` に反映されます。
   *
   * @example
   * ```typescript
   * // 'manager' は LoginEmailVerification のインスタンスと仮定します
   * const userInputCode = "123456";
   * try {
   *   await manager.continueWithCode({ code: userInputCode });
   *   // 成功した場合、ページがリダイレクトされます。ここではクライアント側の追加処理は不要です。
   * } catch (error) {
   *   // この catch は予期しないエラー用であり、Auth0 の検証エラー用ではありません。
   *   console.error("Failed to submit the verification code:", error);
   * }
   * // await の後、サーバー側の検証の問題について manager.transaction.errors を必ず確認してください。
   * if (manager.transaction.errors && manager.transaction.errors.length > 0) {
   *   manager.transaction.errors.forEach(err => {
   *     if (err.code === 'invalid-code') {
   *       // ユーザーに「入力したコードは無効です」と表示します。
   *     }
   *   });
   * }
   * ```
   */
  continueWithCode(payload: ContinueWithCodeOptions): Promise<void>;

  /**
   * 新しい確認コードをユーザーのメールアドレスに送信するよう要求します。
   * このアクションは通常、最初のメールが届かなかった、コードが期限切れになった、
   * または問題が疑われる場合に、ユーザーが「コードを再送信」ボタンをクリックしたときに呼び出されます。
   * SDK は再送信要求を示すアクションとともに Auth0 の `/u/login-email-verification` エンドポイントに POST します。
   *
   * このリクエストが正常に送信されると、Auth0 は新しいメールの送信を試みます。
   * ページが再レンダリングされる場合があります。問題が発生した場合（例: 同じメールアドレスへの
   * 再送信試行が多すぎる場合、エラーコード `too-many-emails` で識別）、`transaction.errors` が更新されます。
   *
   * @param {ResendCodeOptions} [payload] - 省略可能。リクエストとともに追加パラメーターを送信する必要がある場合の
   *                                        `CustomOptions` 用オブジェクト。
   * @returns {Promise<void>} 再送信リクエストが開始されたときに解決される Promise。
   *                          `continueWithCode` と同様に、Promise の解決時に直接データが返されるのではなく、
   *                          通常はページの再レンダリングまたは state の更新が発生します。
   * @throws {Error} `FormHandler` が回復不能な問題（例: ネットワークエラー）に遭遇した場合にスローします。
   *                 サーバー側のエラー（rate limit など）は `this.transaction.errors` に反映されます。
   *
   * @example
   * ```typescript
   * // 'manager' は LoginEmailVerification のインスタンスと仮定します
   * try {
   *   await manager.resendCode();
   *   // 必要に応じて、新しいコードが送信されたことをユーザーに通知するよう UI を更新します。
   *   alert("A new verification code has been dispatched to your email.");
   * } catch (error) {
   *   console.error("Failed to request a new code:", error);
   * }
   * // await の後、サーバー側の問題について manager.transaction.errors を確認してください。
   * if (manager.transaction.errors && manager.transaction.errors.length > 0) {
   *   manager.transaction.errors.forEach(err => {
   *     if (err.code === 'too-many-emails') {
   *       // 「メールの送信要求が多すぎます。数分待ってからお試しください。」と表示します。
   *     }
   *   });
   * }
   * ```
   */
  resendCode(payload?: ResendCodeOptions): Promise<void>;

  /**
   * この画面のタイムアウト管理を含む再送信機能を取得します
   * @param options 再送信機能の設定オプション
   */
  resendManager(options?: StartResendOptions): ResendControl;
}
````

<ParamField body="continueWithCode" type="Promise<void>">
  ユーザーが入力した確認コードを送信します。
  このアクションは、ユーザーがメールで受け取った確認コードを入力し、
  「続行」または「確認」ボタンをクリックする操作に対応します。SDK はその後、この `code` を
  Auth0 の `/u/login-email-verification` エンドポイントに POST します。

  `code` が有効な場合、Auth0 は通常、ユーザーを認証フローの次のステップへリダイレクトします。`code` が無効、
  期限切れ、または別のエラーが発生した場合、Auth0 は通常
  login-email-verification 画面を再表示し、SDK のコンテキスト内の
  `transaction.errors` 配列が失敗の詳細
   (例: エラーコード `invalid-code`) で更新されます。

  フォーム送信が開始されると解決される Promise です。
  リダイレクトまたはページの再表示が一般的な結果であるため、
  解決時にデータが直接返されることはありません。

  #### スロー

  `payload.code` が指定されていないか文字列ではない場合、
  または送信中に `FormHandler` で回復不能な問題 (例: ネットワークエラー) が発生した場合は、エラーをスローします。
  Auth0 からの検証エラー (無効な `code` など) は JavaScript エラーとしてスローされず、
  操作後に `this.transaction.errors` に反映されます。

  ```typescript Example
  // 'manager' は LoginEmailVerification のインスタンスであるとします
  const userInputCode = "123456";
  try {
    await manager.continueWithCode({ code: userInputCode });
    // 成功した場合、ページはリダイレクトされます。ここでは追加のクライアント側処理は不要です。
  } catch (error) {
    // この catch は予期しないエラー用であり、Auth0 の検証エラー用ではありません。
    console.error("Failed to submit the verification code:", error);
  }
  // await の後は、サーバー側の検証エラーがないか必ず manager.transaction.errors を確認してください。
  if (manager.transaction.errors && manager.transaction.errors.length > 0) {
    manager.transaction.errors.forEach(err => {
      if (err.code === 'invalid-code') {
        // 「入力した code は無効です」とユーザーに表示します。
      }
    });
  }
  ```

<ParamField body="resendCode" type="Promise<void>">
  ユーザーのメールアドレスに新しい確認コードを送信するようリクエストします。
  このアクションは通常、ユーザーが「Resend Code」ボタンをクリックしたときに呼び出されます。たとえば、
  最初のメールを受信していない場合、確認コードの有効期限が切れている場合、または何らかの問題があると思われる場合です。
  SDK は、再送リクエストを示す action を付けて、Auth0 の `/u/login-email-verification` エンドポイントに POST します。

  このリクエストが正常に送信されると、Auth0 は新しいメールの送信を試みます。
  ページが再レンダリングされる場合があります。問題がある場合 (例: 同じメールアドレスに対する再送試行が多すぎる場合。
  これはエラー code `too-many-emails` で識別されます) 、`transaction.errors` が更新されます。

  再送リクエストが開始されると resolve される Promise です。
  `continueWithCode` と同様に、通常は Promise の解決結果として直接データが返されるのではなく、
  ページの再レンダリングまたは state の更新が発生します。

  #### スロー

  `FormHandler` で回復不能な問題 (例: ネットワークエラー) が発生した場合にスローされます。
  サーバー側のエラー (レート制限など) は `this.transaction.errors` に反映されます。

  ```typescript Example
  // 'manager' が LoginEmailVerification のインスタンスであると仮定します
  try {
    await manager.resendCode();
    // 必要に応じて、新しい確認コードが送信されたことをユーザーに知らせるために UI を更新します。
    alert("新しい確認コードがメールアドレスに送信されました。");
  } catch (error) {
    console.error("新しい code のリクエストに失敗しました:", error);
  }
  // await の後、サーバー側の問題について manager.transaction.errors を確認します。
  if (manager.transaction.errors && manager.transaction.errors.length > 0) {
    manager.transaction.errors.forEach(err => {
      if (err.code === 'too-many-emails') {
        // 「メールのリクエスト回数が多すぎます。数分待ってから再試行してください。」と表示します。
      }
    });
  }
  ```
