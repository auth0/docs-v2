LoginEmailVerificationMembers
description は、Login Email Verification 画面を操作するために使用できるメンバー (プロパティとメソッド) を定義します。
この画面は、通常ログイン時に行われるメールアドレス確認プロセスの重要な一部であり、ユーザーはワンタイム code を入力して
メールアドレスの所有権を証明する必要があります。SDK を使用すると、この code を送信したり、
新しい code を要求したりできます。

この画面の `universal_login_context` (`window.universal_login_context`) には、以下が含まれます。

* `client`: Auth0 アプリケーションに関する情報。
* `organization` (省略可能) : 認証が特定の組織に対して行われる場合の詳細。
* `prompt`: 現在の認証プロンプトのコンテキスト (例: &#39;login&#39;) 。
* `screen`: UI テキストと一般的な画面情報。`login-email-verification` では、標準のものを除き、画面固有の `data` フィールドは定義されていません。
* `transaction`: 進行中のトランザクションの詳細。state や、前回までの試行で発生したエラー (例: &quot;invalid-code&quot;) が含まれます。

````ts Example
export interface LoginEmailVerificationMembers extends BaseMembers {
  /**
   * ユーザーが入力したメールアドレス確認コードを送信します。
   * このアクションは、ユーザーがメールで受け取ったcodeを入力し、
   * 「続行」または「確認」ボタンをクリックする操作に対応します。SDKはこのcodeを
   * Auth0の `/u/login-email-verification` エンドポイントにPOSTします。
   *
   * codeが有効な場合、Auth0は通常ユーザーを認証フローの次のステップにリダイレクトします。
   * codeが無効、期限切れ、またはその他のエラーが発生した場合、
   * Auth0は通常login-email-verification画面を再レンダリングし、
   * SDKコンテキスト内の `transaction.errors` 配列がエラーの詳細
   * （例：エラーコード `invalid-code`）で更新されます。
   *
   * @param {ContinueWithCodeOptions} payload - ユーザーが入力した `code` 文字列を含むオブジェクト。
   *                                            拡張機能のために任意の `CustomOptions` を含めることもできます。
   * @returns {Promise<void>} フォームの送信が開始されたときに解決されるPromise。
   *                          リダイレクトまたはページの再レンダリングが一般的な結果であるため、
   *                          解決時にデータを直接返しません。
   * @throws {Error} `payload.code` が指定されていないか文字列でない場合、
   *                 または `FormHandler` が送信中に回復不能な問題（例：ネットワークエラー）に
   *                 遭遇した場合にエラーをスローします。
   *                 Auth0からの検証エラー（無効なcodeなど）はJavaScriptエラーとしてスローされず、
   *                 操作後に `this.transaction.errors` に反映されます。
   *
   * @example
   * ```typescript
   * // 'manager' は LoginEmailVerification のインスタンスと仮定します
   * const userInputCode = "123456";
   * try {
   *   await manager.continueWithCode({ code: userInputCode });
   *   // 成功した場合、ページがリダイレクトされます。ここではクライアント側の追加処理は不要です。
   * } catch (error) {
   *   // このcatchは予期しないエラー用であり、Auth0の検証エラー用ではありません。
   *   console.error("Failed to submit the verification code:", error);
   * }
   * // awaitの後、サーバー側の検証エラーについて manager.transaction.errors を必ず確認してください。
   * if (manager.transaction.errors && manager.transaction.errors.length > 0) {
   *   manager.transaction.errors.forEach(err => {
   *     if (err.code === 'invalid-code') {
   *       // ユーザーに「入力したcodeは無効です」と表示します。
   *     }
   *   });
   * }
   * ```
   */
  continueWithCode(payload: ContinueWithCodeOptions): Promise<void>;

  /**
   * 新しい確認コードをユーザーのメールアドレスに送信するよう要求します。
   * このアクションは通常、ユーザーが「コードを再送信」ボタンをクリックしたときに呼び出されます。
   * 最初のメールが届かなかった、codeが期限切れになった、または問題が疑われる場合などが該当します。
   * SDKは再送信リクエストを示すアクションとともに、Auth0の `/u/login-email-verification`
   * エンドポイントにPOSTします。
   *
   * このリクエストが正常に送信されると、Auth0は新しいメールの送信を試みます。
   * ページが再レンダリングされる場合があります。問題が発生した場合（例：同じメールアドレスへの
   * 再送信試行が多すぎる場合、エラーコード `too-many-emails` で識別）、
   * `transaction.errors` が更新されます。
   *
   * @param {ResendCodeOptions} [payload] - 省略可能。リクエストとともに追加パラメーターを
   *                                        送信する必要がある場合の `CustomOptions` 用オブジェクト。
   * @returns {Promise<void>} 再送信リクエストが開始されたときに解決されるPromise。
   *                          `continueWithCode` と同様に、Promiseの解決時に直接データを返すのではなく、
   *                          通常はページの再レンダリングまたはstateの更新が発生します。
   * @throws {Error} `FormHandler` が回復不能な問題（例：ネットワークエラー）に遭遇した場合にスローします。
   *                 サーバー側のエラー（rate limitなど）は `this.transaction.errors` に反映されます。
   *
   * @example
   * ```typescript
   * // 'manager' は LoginEmailVerification のインスタンスと仮定します
   * try {
   *   await manager.resendCode();
   *   // 必要に応じて、新しいcodeが送信されたことをユーザーに通知するようUIを更新します。
   *   alert("A new verification code has been dispatched to your email.");
   * } catch (error) {
   *   console.error("Failed to request a new code:", error);
   * }
   * // awaitの後、サーバー側の問題について manager.transaction.errors を確認してください。
   * if (manager.transaction.errors && manager.transaction.errors.length > 0) {
   *   manager.transaction.errors.forEach(err => {
   *     if (err.code === 'too-many-emails') {
   *       // 「メールの送信リクエストが多すぎます。数分お待ちください。」と表示します。
   *     }
   *   });
   * }
   * ```
   */
  resendCode(payload?: ResendCodeOptions): Promise<void>;

  /**
   * この画面の再送信機能とタイムアウト管理を取得します
   * @param options 再送信機能の設定オプション
   */
  resendManager(options?: StartResendOptions): ResendControl;
}
````

<ParamField body="continueWithCode" type="Promise<void>">
  ユーザーが入力したメールアドレス確認コードを送信します。
  このアクションは、ユーザーがメールで受け取った code を入力し、
  「続行」または「確認」ボタンをクリックする操作に対応します。SDK はその後、この code を
  Auth0 の `/u/login-email-verification` エンドポイントに POST します。

  code が有効な場合、Auth0 は通常、ユーザーを認証フローの次のステップにリダイレクトします。code が無効、
  期限切れ、または別のエラーが発生した場合、Auth0 は通常 login-email-verification 画面を再描画し、
  SDK のコンテキスト内の `transaction.errors` 配列が、失敗の詳細
   (例: エラー code `invalid-code`) で更新されます。

  フォーム送信が開始されると解決される Promise です。
  解決時に直接データは返されません。通常はリダイレクトまたは
  ページの再描画が行われるためです。

  #### Throws

  `payload.code` が指定されていない場合、または文字列ではない場合、
  あるいは送信中に `FormHandler` で回復不能な問題 (例: ネットワークエラー) が発生した場合にエラーをスローします。
  Auth0 からの検証エラー (無効な code など) は JavaScript エラーとしてスローされず、
  操作後に `this.transaction.errors` に反映されます。

  ```typescript Example
  // 'manager' が LoginEmailVerification のインスタンスであると仮定します
  const userInputCode = "123456";
  try {
    await manager.continueWithCode({ code: userInputCode });
    // 成功した場合、ページはリダイレクトされます。ここでは追加のクライアント側アクションは不要です。
  } catch (error) {
    // この catch は予期しないエラー用であり、Auth0 の検証エラー用ではありません。
    console.error("確認コードの送信に失敗しました:", error);
  }
  // await の後は、サーバー側の検証エラーがないか manager.transaction.errors を必ず確認してください。
  if (manager.transaction.errors && manager.transaction.errors.length > 0) {
    manager.transaction.errors.forEach(err => {
      if (err.code === 'invalid-code') {
        // 「入力した code は無効です」とユーザーに表示します。
      }
    });
  }
  ```

<ParamField body="resendCode" type="Promise<void>">
  新しい確認コードをユーザーのメールアドレスに送信するよう要求します。
  このアクションは通常、ユーザーが「コードを再送信」ボタンをクリックしたときに呼び出されます。たとえば、
  最初のメールを受信していない場合、code の有効期限が切れた場合、または問題があると考えられる場合です。
  SDK は、再送信リクエストを示すアクションとともに Auth0 の `/u/login-email-verification` エンドポイントに
  POST します。

  このリクエストが正常に送信されると、Auth0 は新しいメールの送信を試みます。
  ページが再描画される場合もあります。問題がある場合 (例: 同じメールアドレスに対する再送信試行が多すぎる場合で、
  エラー code `too-many-emails` によって識別される) 、`transaction.errors` が更新されます。

  再送信リクエストが開始されると解決される Promise です。
  `continueWithCode` と同様に、通常は Promise の解決時に直接データが返されるのではなく、
  ページの再描画または state の更新が行われます。

  #### Throws

  送信中に `FormHandler` で回復不能な問題 (例: ネットワークエラー) が発生した場合にスローされます。
  サーバー側のエラー (レート制限など) は `this.transaction.errors` に反映されます。

  ```typescript Example
  // 'manager' が LoginEmailVerification のインスタンスであると仮定します
  try {
    await manager.resendCode();
    // 必要に応じて、新しい code が送信されたことをユーザーに知らせるよう UI を更新します。
    alert("新しい確認コードがメールアドレスに送信されました。");
  } catch (error) {
    console.error("新しい code の要求に失敗しました:", error);
  }
  // await の後は、サーバー側の問題がないか manager.transaction.errors を確認してください。
  if (manager.transaction.errors && manager.transaction.errors.length > 0) {
    manager.transaction.errors.forEach(err => {
      if (err.code === 'too-many-emails') {
        // 「メールの要求回数が多すぎます。数分待ってからもう一度お試しください。」と表示します。
      }
    });
  }
  ```
