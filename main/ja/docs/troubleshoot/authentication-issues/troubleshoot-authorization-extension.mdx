---

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 では、ロールベースアクセス制御 (RBAC) を実装する方法が 2 つ用意されており、API 独自の内部アクセス制御システムの代わりに使用することも、組み合わせて使用することもできます。

  * [Authorization Core](/ja/docs/manage-users/access-control/rbac)
  * [Authorization Extension](/ja/docs/customize/extensions/authorization-extension)

  Authorization Core feature set は、Authorization Extension と同等の機能を備え、パフォーマンスとスケーラビリティを向上させるとともに、Authorization Extension より柔軟な RBAC システムを提供します。

  現時点では、どちらも RBAC の主要な機能を実装しており、API で定義されたカスタム スコープを、permissions としてユーザーに割り当てられたもののみに制限できます。
</Callout>

以下では、Authorization Extension の設定時に発生する可能性がある問題と、その原因を特定するためのヒントを紹介します。

<div id="the-authentication-results-in-a-token-contain-group-information-but-not-roles-or-permissions-information">
  ## 認証結果のトークンにはグループ情報は含まれるが、ロールや権限の情報は含まれない
</div>

この現象が発生する場合、あるアプリケーションに対してロールと権限を作成していても、ユーザーは別のアプリケーションを使って認証している可能性があります。たとえば、Website A に対してすべてのロールと権限を作成したとします。しかし、Auth0 で Website B 用の別の Web サイトアプリケーションも作成したとします。その後、アプリで Website A 用ではなく Website B 用の `client_id` と `client_secret` を使用している場合です。

また、いずれかのユーザーを含む接続で、<Tooltip tip="Auth0 Dashboard: サービスを構成するための Auth0 のメイン製品。" cta="用語集を表示" href="/ja/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> の **Try Connection** ボタンをクリックした場合にも、これが発生することがあります。これにより Auth0 のグローバルアプリケーションを使用した認証フローが実行されますが、これは拡張機能で設定したアプリケーションとは同じではありません。

<div id="application-is-not-shown-in-the-drop-down-menu-when-setting-up-the-extension">
  ## 拡張機能の設定時に、アプリケーションがドロップダウンメニューに表示されない
</div>

Authorization Extension でサポートされているアプリケーションタイプは次のとおりです。

* ネイティブアプリ
* シングルページ Web アプリ
* 通常の Web アプリ

タイプが割り当てられていないアプリケーションと Machine to Machine Application はサポートされていません。

<div id="after-upgrading-to-v2-users-get-an-error-upon-login">
  ## v2 へのアップグレード後、ユーザーがログイン時にエラーを受け取る
</div>

`You are not allowed to access this application` というエラーが表示される場合、原因は古い Rule との競合である可能性が高いです。永続化設定を無効にし、既存の Rule を削除したうえで、設定を再度有効にしてから、もう一度テストしてください。
