description: 契約者向けに、Auth0 のテナントにおけるエンティティ上限ポリシーについて説明します。
title: エンティティ上限ポリシー
---

Auth0 におけるエンティティとは、アプリケーション、接続、組織、API <Tooltip tip="リソースサーバー: 保護対象リソースをホストするサーバー。リソースサーバーは保護対象リソースへのリクエストを受け入れ、応答します。" cta="用語集を表示" href="/ja/docs/glossary?term=resource+servers">リソースサーバー</Tooltip>、Permissions、ロール、Actions などのテナント構成要素を指します。

<Warning>
  エンティティ上限は、プラットフォームの健全性を維持するために適用されるハードリミットです。これは価格設定の一部ではありませんが、料金プランのグループによって異なる場合があります。各料金プランには別途制限があり、特定のプランを契約したときに付与される MAU や Enterprise 接続 などの利用枠を指します。 (料金プランごとの制限については、[Auth0 Pricing](https://auth0.com/pricing/) を参照してください。)
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  &quot;unlimited&quot; という用語は、ハードリミットがないことを意味しますが、Okta はしきい値に対してシステム上の制限を設けています。これらの制限について不明な点がある場合は、サポートにお問い合わせください。
</Callout>

<div id="global-subscription-plan-limits">
  ## 全サブスクリプションプラン共通の制限
</div>

このセクションでは、すべてのサブスクリプションプランに共通して適用される制限を示します。

<div id="organizations">
  ### 組織
</div>

| Entity                              | Limit   |
| ----------------------------------- | ------- |
| テナントあたりの組織数                         | 100,000 |
| 組織あたりのメンバー数                         | 100,000 |
| 組織あたりの接続数                           | 10      |
| 組織メンバーあたりのロール割り当て数                  | 50      |
| エンタープライズグループあたりのロール割り当て数         | 10      |
| エンタープライズグループに割り当て可能な一意のロール数      | 100     |
| ロールを割り当て可能なエンタープライズグループ数        | 10,000  |
| 組織あたりの M2M クライアントグラント数 (パブリッククラウド)  | 100     |
| 組織あたりの M2M クライアントグラント数 (プライベートクラウド) | 1000    |
| カスタムトークン交換プロファイル数                   | 100     |

Enterprise プランのお客様は、[サポートに連絡](https://support.auth0.com/)することで、テナントあたりの組織数および組織あたりのメンバー数の上限引き上げをリクエストできます。

これらの上限は、パブリッククラウドではテナントあたり 2,000,000 組織、組織あたり 2,000,000 メンバーまで引き上げることができます。プライベートクラウドのインスタンスでは、これらのエンティティに上限はありません。

<div id="authorization-core-role-based-access-control-rbac">
  ### Authorization Core のロールベースアクセス制御 (RBAC)
</div>

| Entity                     | Limit  |
| -------------------------- | ------ |
| テナントあたりのロール                | 1000   |
| API (リソースサーバー) あたりのスコープ    | 1000   |
| ユーザーあたりのロール                | 50     |
| エンタープライズグループあたりのロール        | 10     |
| エンタープライズグループに割り当て可能な一意のロール | 100    |
| ロールを割り当て可能なエンタープライズグループ    | 10,000 |
| ユーザーあたりのPermissions        | 1000   |
| ロールあたりのPermissions         | 1000   |

上記のエンタープライズグループおよびロールの上限は、組織機能を使用しない場合の RBAC に適用されます。組織機能を使用すると、これらの上限はテナント単位ではなく組織単位で適用されます。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  ユーザーあたりのPermissions数の上限は、直接割り当てられたPermissionsにのみ適用される点に注意してください。

  技術的には、Permissionsを複数の異なるロールに割り当ててから、それらのロールをユーザーに割り当てることで、ここに記載されている数を超えるPermissionsをユーザーが持つことも可能です。
</Callout>

| Entity                        | Limit |
| ----------------------------- | ----- |
| アプリケーションごとのユーザーあたりのリフレッシュトークン | 200   |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  <Tooltip tip="リフレッシュトークン: ユーザーに再度ログインさせることなく、新しいアクセストークンを取得するために使用されるトークンです。" cta="用語集を見る" href="/ja/docs/glossary?term=Refresh+tokens">リフレッシュトークン</Tooltip> の上限に達した状態で新しいリフレッシュトークンを作成すると、
  システムはその
  アプリケーションにおけるそのユーザーの最も古いトークンを取り消すか削除します。

  取り消されたトークンおよび期限切れのトークンは、この上限にはカウントされません。
</Callout>

<div id="actions">
  ### Actions
</div>

| エンティティ                                | 制限  |
| ------------------------------------- | --- |
| テナントあたりのActions数                      | 100 |
| テナントあたりのトリガーごとのActions数               | 20  |
| テナントあたりのActions Modules数              | 100 |
| ActionあたりのActions Modules数            | 3   |
| Actionあたりのバージョン数                      | 50  |
| Action Moduleあたりのバージョン数               | 50  |
| Actionあたりのシークレット数                     | 30  |
| Action Moduleあたりのシークレット数              | 30  |
| Actionあたりの依存関係 (NPM Modules) 数        | 10  |
| Action Moduleあたりの依存関係 (NPM Modules) 数 | 10  |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  ActionsおよびActions Modulesの制限には、デプロイ済みと未デプロイのエンティティの両方が含まれます。これらを削除すると、該当する制限のカウント対象から外れます。

  Action VersionsおよびAction Module Versionsの制限には、ドラフトとアクティブの両方が含まれます。制限に達すると、以降はバージョンを追加するたびに、Auth0によって最も古い未使用のバージョンが削除されます。
</Callout>
