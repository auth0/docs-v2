Auth0 では、以下の領域に関する運用ポリシーを定めています。

<div id="billing">
  ## 請求
</div>

このポリシーでは、住所変更、クレジットカードに関する問題、サブスクリプションの解約、返金、請求書の発行、売上税、ユーザー数の増減、テナントの変換について説明しています。詳しくは、[請求ポリシー](/ja/docs/troubleshoot/customer-support/operational-policies/billing-policy)を参照してください。

<div id="data-export-and-transfer">
  ## データのエクスポートと転送
</div>

Auth0 からデータをエクスポートするには、[Export/Import Extension](/ja/docs/customize/extensions/authorization-extension/import-and-export-authorization-extension-data) または <Tooltip tip="Management API: お客様が管理タスクを実行するための製品です。" cta="用語集を表示" href="/ja/docs/glossary?term=Management+API">Management API</Tooltip> を使用できます。詳細については、[データのエクスポート](/ja/docs/troubleshoot/customer-support/manage-subscriptions/export-data) を参照してください。

Auth0 は、ある Auth0 テナントから別の Auth0 テナントへのデータ転送を行いません。これは、Public Cloud と Private Cloud の両方のお客様に適用されます。Auth0 テナント内のすべてのデータは常にお客様の管理下にあり、Management API を通じていつでも利用できます。サービスの利用を停止する場合は、データをエクスポートできます。詳細については、[データのエクスポートおよび転送ポリシー](/ja/docs/troubleshoot/customer-support/operational-policies/data-export-and-transfer-policy) を参照してください。

<div id="endpoints">
  ## エンドポイント
</div>

詳細については、[Public Cloud Service のエンドポイント](/ja/docs/troubleshoot/customer-support/operational-policies/public-cloud-service-endpoints)および[SSO (シングルサインオン) の API エンドポイント](/ja/docs/authenticate/single-sign-on/api-endpoints-for-single-sign-on)を参照してください。

<div id="load-and-penetration-testing">
  ## 負荷テストとペネトレーションテスト
</div>

負荷テストを実施できるのは、Enterprise サブスクリプションをご購入のお客様に限られます。Enterprise サブスクリプションをご利用のお客様は、Auth0 の本番テナントに対して、年間 1 回の負荷テスト (再実施は最大 2 回まで) を申請できます。パフォーマンステストおよび負荷テストは、Auth0 による事前の書面での承認がある場合にのみ許可されます。承認後は、当社が承認したテナントのみを対象としてテストを実施できます。詳細については、[負荷テストポリシー](/ja/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) および [ペネトレーションテストポリシー](/ja/docs/troubleshoot/customer-support/operational-policies/penetration-testing-policy) を参照してください。

これらのテストを実施するには、事前に [サポートチケット](https://support.auth0.com) を送信して当社に通知してください。Auth0 では、テストの開始予定日の少なくとも 14 日前までに通知する必要があります。

<div id="migrations">
  ## 移行
</div>

Auth0 では、製品提供に反復的なアプローチを採用しています。これには、新しい機能を導入し、継続的に改善していくための反復的な[製品リリース ライフサイクル](/ja/docs/troubleshoot/product-lifecycle)が含まれます。

Auth0 製品の構築において、Auth0 は次の方針を重視しています。

* お客様に価値を早期から継続的に提供し、そのフィードバックを基に改善を重ねること。
* お客様を深く理解し、あらゆる意思決定でその視点を考慮すること。
* より適切な判断を行えるよう、データを継続的に収集および分析すること。
* 機能を追加する際には、製品全体の現在の姿、理想的な姿、将来の姿を見据えて設計すること。

これらの目標を達成するため、Auth0 では、製品提供に反復的なアプローチを採用しています。これには、新しい機能を導入し、継続的に改善していくための反復的な製品リリース ライフサイクルが含まれます。

* [製品リリース ステージ](/ja/docs/troubleshoot/product-lifecycle/product-release-stages)
* [移行プロセス](/ja/docs/troubleshoot/product-lifecycle/migration-process)
* [非推奨化と移行](/ja/docs/troubleshoot/product-lifecycle/deprecations-and-migrations)
* [過去の移行](/ja/docs/troubleshoot/product-lifecycle/past-migrations)

<div id="rate-limits">
  ## レート制限
</div>

設定を短時間に繰り返し更新したり、過度にポーリングしたり、高い並行度で API 呼び出しを行ったりすると、アプリにレート制限が適用される場合があります。詳しくは [レート制限ポリシー](/ja/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy) を参照してください。

<div id="entity-limits">
  ## エンティティの上限
</div>

Auth0 のエンティティとは、アプリケーション、接続、Rules、API の <Tooltip tip="リソースサーバー: 保護されたリソースをホストするサーバー。リソースサーバーは、保護されたリソースへのリクエストを受け取り、それに応答します。" cta="用語集を表示" href="/ja/docs/glossary?term=resource+servers">リソースサーバー</Tooltip> などのテナント設定要素を指します。Auth0 では、サブスクリプション レベル グループのプラットフォームに応じて、保持できるエンティティの数が制限されます。詳細については、[エンティティ制限ポリシー](/ja/docs/troubleshoot/customer-support/operational-policies/entity-limit-policy) を参照してください。

<div id="sanitize-har-files">
  ## HAR ファイルのサニタイズ
</div>

[Auth0 Support](https://support.auth0.com) に問い合わせる際、トラブルシューティングのために、サニタイズした HAR ファイルの提出が必要になる場合があります。

<Warning>
  HAR ファイルを他者 (Auth0 を含む) と共有する前に、次のような機密データをすべて削除または難読化していることを確認してください。

  * 機密性の高いユーザー情報
  * 個人を特定できる情報 (PII)
  * 機密性の高いアプリケーション情報

  詳細については、[Auth0 Community](https://community.auth0.com) の以下の記事を参照してください。

  * [HTTP トレースのサニタイズ](https://community.auth0.com/t/sanitizing-http-traces/119488)
  * [HTTP トレース ファイルを自動的にサニタイズする方法](https://community.auth0.com/t/how-to-sanitize-a-http-trace-file-automatically/120583)
  * [機密情報を手動でマスキングする方法](https://community.auth0.com/t/how-to-manually-redact-sensitive-information/122554)
  * [HAR ファイルが大きすぎてサポートケースにアップロードできない](https://community.auth0.com/t/har-file-is-too-large-to-upload-to-the-support-case/122488)
</Warning>

<div id="tenant-restoration">
  ## テナントの復元
</div>

<Warning>
  削除したテナントは**復元できません**。また、新しいテナントの作成時に削除済みテナントの名前を再利用することも**できません**。
</Warning>

**テナントを削除する前に、代替手段として以下のリソースを確認してください。**

* [テナント管理者の変更](/ja/docs/get-started/manage-dashboard-access) - テナントの所有者を変更する場合
* [テナントの削除またはリセット](/ja/docs/troubleshoot/customer-support/manage-subscriptions/delete-or-reset-tenant) - テナント構成をリセットする場合

すでにテナントを削除していて、特定のドメイン名を使用する必要がある場合は、新しいテナントで[カスタムドメイン](/ja/docs/customize/custom-domains)を設定することをお勧めします。

<div id="private-cloud-restoration">
  ## Private Cloud の復元
</div>

Private Cloud の復元により、お客様はデータの損失や破損からスペースを復旧できます。このオプションは、追加の SKU として提供されます。

お客様は、過去 14 日以内のバックアップを使用して、本番 Private Cloud スペースの復元を [サポートチケット](https://support.auth0.com/) からリクエストできます。復元は、Auth0 が保持するバックアップデータを使用して実施されます。Private Cloud スペース全体が、お客様の希望する復元時点に最も近い利用可能なバックアップの状態に復元されます。Private Cloud 契約に記載された災害復旧の RPO が適用されます。後方互換性のない機能や非推奨化により、復元できる範囲に制限が生じる場合があります。復元後の環境に対する機能検証は、お客様の責任で実施してください。

この SKU を購入したお客様は、非本番の Private Cloud スペースに対して、年 1 回のテスト復元を利用できます。

<div id="unsupported-requests">
  ## サポート対象外のリクエスト
</div>

サポートチームは、可能な限りお手伝いできるよう努めています。ただし、現在のところ、以下のリクエストには対応できません。

* 非本番アカウントから本番アカウントへのデータ移行
* テナント管理者の削除
* テナント名の変更
* 接続名の変更
* 以前に削除したテナント名の再利用
* テナントのリージョン間移行 (例: US から EU への移行)
* アドホックな使用状況データレポート
* テナント内で削除または変更されたデータや設定の復元 (以下を含む)

  * データベース接続、およびそのユーザーとパスワード
  * ユーザー、そのユーザープロファイル情報、メタデータ、ロールの割り当て
  * ロールと権限
  * アプリケーション
  * SSO 統合
  * API
  * 接続
  * Actions
  * Rules
  * Hooks
  * Extensions
  * メールテンプレート
  * [標準保持期間](/ja/docs/deploy-monitor/logs/log-data-retention)の経過後のテナントログ

<div id="software-updates">
  ## ソフトウェアの更新
</div>
