description: APIのベストプラクティスと標準について説明します。
title: APIのベストプラクティス
---

Auth0 サービスと統合する場合は、現在のネットワーク標準に沿って API を設定することをお勧めします。以下の DNS 名前解決、トランスポート層セキュリティ (TLS) の設定、および HTTP 最適化戦略を確認してください。これらは、インフラストラクチャの変更にアプリケーションが対応し、Auth0 サービスへの安定した接続を維持できるように設計されています。DNS、TLS、HTTP に関するこれらの標準に従うことで、Auth0 のグローバル配信ネットワークとの互換性を確保し、適切な再試行ロジックによってリクエストの信頼性を高め、認証トラフィックのセキュリティを維持できます。

<div id="dns-and-hostname-resolution">
  ## DNS とホスト名解決
</div>

<div id="use-auth0-provided-hostnames">
  ### Auth0 が提供するホスト名を使用する
</div>

Auth0 への接続には、基盤となる IP アドレスではなく、常に提供されたホスト名を使用してください。Auth0 のインフラストラクチャでは、負荷分散、クラウドおよび CDN のルーティング、リージョンのフェイルオーバーのために、DNS ベースのトラフィック制御を利用しています。クラウドインフラストラクチャの命名は時間の経過とともに変わる可能性があるため、DNS ベースのフィルタリングの使用は推奨されません。

<div id="respect-dns-time-to-live-ttl">
  ### DNS の Time to Live (TTL) を尊重する
</div>

DNS が返す [DNS TTL](https://www.ateam-oracle.com/dns-ttls-considerations) は必ず尊重してください。TTL 値は、アプリケーションがインフラストラクチャの変更にすばやく対応できるよう、低めに設定されていることがよくあります (例: 30～300 秒) 。

<div id="dns-caching">
  ### DNS キャッシュ
</div>

可能であれば、オペレーティングシステムの DNS リゾルバーを使用してください。独自の (アプリケーション レベルの) リゾルバーを使用する場合は、応答のキャッシュ期間が TTL を超えないようにしてください。

<div id="support-multiple-aaaaa-records">
  ### 複数の A/AAAA レコードをサポートする
</div>

Auth0 は、特定のホスト名に対して複数の IP アドレスを返すことがあります。API は、ラウンドロビンまたはランダムな割り当てによって、返された IP アドレス間で接続を分散する必要があります。いずれかのアドレスに到達できない場合は、すぐにフェイルオーバーするよう設定してください。

<div id="transport-layer-security-tls">
  ## トランスポート層セキュリティ (TLS)
</div>

Auth0 では、TLS を使用した HTTP リクエストのみをサポートしています。最新バージョンの使用を推奨します。

* **TLS 1.3** を強く推奨
* **TLS 1.2**
* **TLS 1.1 以下はサポート対象外**

<div id="use-modern-cipher-suites">
  ### 最新の暗号スイートを使用する
</div>

クライアントの暗号スイート構成を過度にカスタマイズすることは避けてください。コンプライアンス上の要件 (例: FIPS) がある場合を除き、ランタイムの既定の安全な設定を使用してください。サポートする暗号スイートが限られているクライアントは拒否される可能性があります。

利用可能な暗号スイートを確認するには、[TLS (SSL) Versions and Ciphers](/ja/docs/customize/custom-domains/self-managed-certificates/tls-ssl)を参照してください。

<div id="support-server-name-indication-sni">
  ### Server Name Indication (SNI) をサポート
</div>

API では、Auth0 へのすべての HTTPS トラフィックで、TLS の SNI 拡張を使用する必要があります。SNI を使用しないトラフィックは受け付けられません。

<div id="honor-certificate-validation">
  #### 証明書の検証を必ず行う
</div>

**次のことはしないでください**:

* 証明書の検証を無効にする
* 証明書をピン留めする

Auth0 は証明書を自動的かつ頻繁に更新・置換します。証明書のピン留めは、サービスの中断を招く可能性があります。

<div id="allow-intermediate-certificate-rotation">
  #### 中間証明書のローテーションを許可する
</div>

証明書チェーンは定期的に変更されることがあります。TLS クライアントが次の要件を満たしていることを確認してください。

* デフォルトで、OS のトラストストア内の有効な公開認証局を信頼する
* 定期的な更新の一環として、OS のトラストストアを更新する
* 中間証明書をピン留めしない
* 無効なチェーンのみを拒否する

<div id="enable-tls-session-resumption-and-reuse">
  #### TLS セッションの再開と再利用を有効にする
</div>

HTTP/2 で TLS セッションチケットまたはセッション ID を使用できるようにします。これにより、TLS ハンドシェイクのオーバーヘッドを削減できます。

<div id="http">
  ## HTTP
</div>

<div id="use-a-modern-http-protocol">
  ### 最新の HTTP プロトコルを使用する
</div>

[HTTP/2](https://www.rfc-editor.org/rfc/rfc9113) または [HTTP/3](https://httpwg.org/specs/rfc9114.html) を使用してください。これらは接続の多重化を提供し、HTTP ヘッダー圧縮を使用でき、必要な TCP 接続数も少なくて済みます。クライアントが HTTP/2 を自動的にサポートしている場合は、HTTPS エンドポイントを使用し、HTTP/2 以上はクライアント側でネゴシエーションさせてください。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  必要な場合を除き、HTTP/1.1 は避けてください。HTTP/1.1 は完全にサポートされていますが、効率は低くなります。必要な場合を除き、使用は推奨されません。
</Callout>

<div id="enable-and-reuse-persistent-connections">
  ### 永続接続を有効にして再利用する
</div>

HTTP 接続を再利用してリクエストのレイテンシを削減し、信頼性を向上できるように API を設定します。推奨設定は次のとおりです。

* 接続は最大 300 秒間維持する
* アイドル接続の最大数は適切な値に制限する
* 既存の接続を活用するため、リクエストごとに新しい HTTP API オブジェクトを作成しない
* 接続の健全性を保つため、定期的に接続を再確立する

<div id="request-retries">
  ### リクエストの再試行
</div>

再試行しても安全な操作、つまり[冪等なリクエスト](https://developer.mozilla.org/en-US/docs/Glossary/Idempotent)については、リクエストの再試行を実装することを推奨します。

一時的なネットワークエラーまたはサーバー側エラーが原因で再試行する場合は、HTTP `Retry-After` レスポンスヘッダーを確認してください。このヘッダーにより、アプリケーションが次のリクエストを送信するまでに待機すべき時間を判断できます。

例: `Retry-After: 300`

ヘッダーが存在する場合は、指定された秒数だけ待機してからリクエストを再試行してください。`Retry-After` に従うことで、適正な利用が確保され、障害の連鎖やレート制限を防ぐことができます。

`Retry-After` が存在しない場合は、再試行間隔の決定に[指数バックオフ](https://en.wikipedia.org/wiki/Exponential_backoff)を使用してください。試行回数は3回を超えないでください。

<div id="auth0-sdks">
  ## Auth0 SDKs
</div>

環境またはアプリケーションで Auth0 SDKs を使用する際は、以下のベストプラクティスに従うことを推奨します。

<div id="use-official-auth0-sdks">
  ### 公式の Auth0 SDK を使用する
</div>

お使いのプラットフォームと言語では、[公式または掲載されている](docs/libraries) SDK を優先してください。カスタムの HTTP アプリケーション統合や API 統合を独自に構築するのではなく、公式の Auth0 SDK を使用してください。これらのライブラリはベストプラクティスに沿って保守されており、低レベルのネットワーク処理やセキュリティ準拠に伴う複雑さを抽象化します。

<div id="choose-the-applicable-sdk">
  ### 該当する SDK を選択する
</div>

Auth0 は、さまざまな統合パターン向けの専用ライブラリを提供しています。アーキテクチャ上のユースケースに適した SDK を選択してください。

* **認証および認可 SDK**: これらの SDK は、エンドユーザーのログインやトークンの取得に使用します。アプリケーションの種類に応じて SDK を選択してください。
  * **シングルページアプリケーション (SPA)**: JS/TS ライブラリは、ブラウザー内でトークンのライフサイクルと PKCE フローを処理します (例: [React](https://github.com/auth0/auth0-react)、[Vue](https://github.com/auth0/auth0-vue)、[Angular](https://github.com/auth0/auth0-angular)) 。
  * **従来型の Web アプリケーション**: サーバーサイドのライブラリは、セッションと Cookie ベースのトランザクションを管理します (例: [Node.js](https://github.com/auth0/node-auth0)、[ASP.NET Core](https://github.com/auth0/auth0-aspnetcore-authentication)、[PHP](https://github.com/auth0/auth0-PHP)) 。
  * **ネイティブ/モバイルアプリケーション**: プラットフォームネイティブのライブラリは、システムブラウザーやセキュアストレージと連携します ([Swift](https://github.com/auth0/Auth0.swift)、[Kotlin](https://github.com/auth0/Auth0.android)、[React Native](https://github.com/auth0/react-native-auth0)) 。
* **バックエンドサービスおよび API**: API で受信したアクセストークンを検証します。これらのライブラリは、暗号学的署名の検証と [JSON Web Key Sets (JWKS)](docs/secure/tokens/json-web-tokens/json-web-key-sets#json-web-key-sets) のキャッシュを処理します。
* **Management API SDKs**: 新しいユーザーの作成やクライアントシークレットのローテーションなどの管理タスクをプログラムで実行します。レート制限ヘッダーと再試行ロジックを処理します。

公式 SDK を使用することで、統合で次の機能が自動的にサポートされます。

* 回復性に優れた HTTP 接続プーリングと keep-alive の設定。
* 指数バックオフを用いた再試行ロジックの正しい実装と、`Retry-After` ヘッダーの尊重。
* 最新の TLS およびセキュリティ標準への準拠。
