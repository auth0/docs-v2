title: エラーメッセージを確認する
---

以下のいずれかの場所に表示されるエラーメッセージを確認してください。

* ブラウザーと HTML ページのレスポンス
* 開発者ツールのネットワークタブとコンソールタブ
* <Tooltip tip="認可サーバー: ユーザーがアクセスできる範囲の定義に関与する一元化されたサーバーです。たとえば、認可サーバーはユーザーが利用できるデータ、タスク、機能を制御できます。" cta="用語集を表示" href="/ja/docs/glossary?term=Authorization+Server">認可サーバー</Tooltip> のレスポンス
* 非推奨エラー

<div id="check-browser-errors">
  ## ブラウザーエラーを確認する
</div>

HAR ファイルは、ウェブブラウザーとウェブサーバーのやり取りを JSON 形式のログとして記録したものです。認証が想定どおりに機能していない場合は、[HAR ファイルを生成して分析する](/ja/docs/troubleshoot/troubleshooting-tools/generate-and-analyze-har-files)ことで、問題を特定できます。

<Warning>
  HAR ファイルを誰か (Auth0 を含む) と共有する前に、次のような機密データは必ずすべて削除するか、判別できないようにしてください。

  * 機密性の高いユーザー情報
  * 個人を特定できる情報 (PII)
  * 機密性の高いアプリケーション情報

  詳しくは、[Auth0 Community](https://community.auth0.com) の以下の記事を参照してください。

  * [HTTP トレースのサニタイズ](https://community.auth0.com/t/sanitizing-http-traces/119488)
  * [HTTP トレースファイルを自動的にサニタイズする方法](https://community.auth0.com/t/how-to-sanitize-a-http-trace-file-automatically/120583)
  * [機密情報を手動でマスクする方法](https://community.auth0.com/t/how-to-manually-redact-sensitive-information/122554)
  * [HAR ファイルが大きすぎてサポートケースにアップロードできない](https://community.auth0.com/t/har-file-is-too-large-to-upload-to-the-support-case/122488)
</Warning>

<div id="check-login-screen">
  ## ログイン画面を確認する
</div>

Lock のログインウィジェットでは、ユーザー名やパスワードが正しくない場合など、特定の問題が発生するとエラーメッセージが表示されます。Auth0 の標準エラーページを使用している場合は、**詳細情報** リンクを確認してください。

<div id="check-logs">
  ## ログを確認する
</div>

Auth0 は、Dashboard 管理者の操作、ユーザー認証の成功と失敗、パスワード変更リクエストなどを含む[ログデータ](/ja/docs/deploy-monitor/logs)を保存します。ログは[Dashboard](https://manage.auth0.com/#/logs)で確認できます。

一部のエラーはログに表示されません。たとえば、外部の<Tooltip tip="Identity Provider (IdP): デジタルアイデンティティを保存・管理するサービス。" cta="用語集を表示" href="/ja/docs/glossary?term=Identity+Provider">IDプロバイダー</Tooltip>でエラーが発生し、認証が完了しないままユーザーが Auth0 に戻らない場合、ログにはエントリが記録されません。

Auth0 のログはエクスポートして自分で保存することも、外部のログサービスに自動的にプッシュすることもできます。この機能は、データ保持要件やログ分析要件への対応に役立ちます。Auth0 Extension をインストールして設定すれば、ログを Sumo Logic や Loggly などの別のプロバイダーに自動的にエクスポートできます。詳しくは、Auth0 Marketplace の[Log Streaming ソリューション](https://marketplace.auth0.com/features/log-streaming)をご覧ください。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  問題のトラブルシューティングに役立つ各ログイベントの詳細については、[Monitoring ページ](/ja/docs/deploy-monitor/monitor)を参照してください。
</Callout>

また、<Tooltip tip="Management API: 顧客が管理タスクを実行できるようにする製品。" cta="用語集を表示" href="/ja/docs/glossary?term=Management+API">Management API</Tooltip>を使用してログをエクスポートし、保存することもできます。利用可能なエンドポイントは 2 つあり、それぞれ少し異なる情報を提供します。

<div id="search-all-logs-get_logs-endpoint">
  ### すべてのログを検索する /get_logs エンドポイント
</div>

[ログイベント検索エンドポイント](https://auth0.com/docs/api/management/v2#!/Logs/get_logs)では、指定した検索条件に一致するログエントリを取得できます。検索条件を指定しない場合は、利用可能なすべてのエントリの一覧が返されます。

検索条件は **q** パラメーターで指定し、特定のフィールドは **fields** パラメーターで取得できます。

API にアクセスするには、[Management API v2 トークン](https://auth0.com/docs/api/management/v2/tokens)が必要です。

このリクエスト例では、ログイン成功のすべてのログを取得します (ログイン成功のイベント略語は `s` です) 。各ログエントリから取得するフィールドは、**date**、**description**、**client&#95;id**、**log&#95;id** です。

使用できる検索条件とイベント略語の一覧の詳細については、[ログイベント検索エンドポイント](https://auth0.com/docs/api/management/v2#!/Logs/get_logs)を参照してください。

<div id="get-a-single-log-entry-get_logs_by_id-endpoint">
  ### 単一のログエントリを取得する /get_logs_by_id エンドポイント
</div>

[ID を指定してログイベントを取得するエンドポイント](https://auth0.com/docs/api/management/v2#!/Logs/get_logs_by_id) は、指定した ID に対応するログエントリを取得します。

このサンプルリクエストは、ID `90020180129170850881585554625888895190928456277777449010` のログエントリを 1 件取得します。

<div id="check-logs-for-deprecation-errors">
  ## 非推奨に関するエラーがないかログを確認する
</div>

Auth0 の機能が非推奨になると、アプリケーションでその機能が使用されていることを示すエラーや通知がテナントログに記録されることがあります。機能が非推奨になっている可能性を示す、[特定の非推奨エントリをログで検索](/ja/docs/troubleshoot/basic-issues/check-deprecation-errors)できます。

<div id="rate-limits-and-other-errors">
  ## レート制限とその他のエラー
</div>

Auth0 では、[レート制限を超過した](/ja/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy)ときに報告されるエラーに対して、固有のエラーコードを提供しています。レート制限エラーを確認できるようにログの自動スキャンを設定し、ユーザーに大きな影響が及ぶ前に、レート制限に達するアクティビティへ事前に対処することをおすすめします。Auth0 はその他の種類のエラーについてもエラーコードを公開しているため、認証エラーに加えて、Auth0 Management API 呼び出しによるエラーについてもログをスキャンすると役立ちます (Management API のエラーコードは、[Management API Explorer](https://auth0.com/docs/api/management/v2) で各呼び出しの下に表示されます) 。

また、Rule 内からユーザープロフィールを取得するために Management API を呼び出すことは、レート制限エラーの一般的な原因である点にも注意してください。これは、そのような API 呼び出しがログインのたびに実行されるうえ、定期的なセッションチェックでも実行される可能性があるためです。

<div id="check-actions-real-time-logs-error-console">
  ## Actions リアルタイムログのエラーコンソールを確認する
</div>

Rules、Hooks、カスタムデータベース スクリプトには、`console.log()` ステートメントを記述できます。これらのステートメントの出力は、Actions リアルタイムログで確認できます。Actions リアルタイムログを使用している場合は、**Auth0 Dashboard &gt; Monitoring &gt; Actions Logs** に移動すると、このログコンソールを開けます。
