オンラインリフレッシュトークン (ORT) は、[シングルページアプリケーション](/ja/docs/get-started/auth0-overview/create-applications/single-page-web-apps) (SPA) などのブラウザーベースのアプリケーション向けに特別に設計された[リフレッシュトークン](/ja/docs/secure/tokens/refresh-tokens)の一種です。

ORT は、Intelligent Tracking Prevention (ITP) などのブラウザのプライバシー機能によってサードパーティ Cookie へのアクセスがブロックされ、従来の[サイレント認証](/ja/docs/authenticate/login/configure-silent-authentication#configure-silent-authentication)が信頼できなくなる場合でも、SPA でユーザーのセッションを維持できます。

ORT はユーザーの Auth0 [セッション](/ja/docs/manage-users/sessions)に関連付けられており、そのセッションが有効な間のみ使用できます。これにより、ユーザーがログアウトすると、そのセッションを使用しているすべてのアプリケーションも一緒にログアウトされます。

<div id="how-it-works">
  ## 仕組み
</div>

ORT をリクエストすると、次のように処理されます。

1. アプリケーションは、認可リクエストに `online_access` スコープを含めます。
2. Auth0 は、ユーザーの現在のセッションに関連付けられた ORT を返します。
3. ORT を新しいアクセストークンに交換するたびに、Auth0 はセッションのアイドルタイムアウトを延長します。
4. 基盤となる Auth0 セッションがアクティブである限り、トークンは有効なままです。

<div id="comparison-of-orts-versus-refresh-tokens">
  ## ORT とリフレッシュトークンの比較
</div>

| **特徴**                                                                                             | **ORTs**                                                                                          | **リフレッシュトークン**                                                                        |
| -------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| **要求されるスコープ**                                                                                      | `online_access`                                                                                   | `offline_access`                                                                      |
| **セッションとの紐付け**                                                                                     | Auth0 セッションに紐付けられる                                                                                | セッションとは独立                                                                             |
| **有効期間**                                                                                           | [セッションの有効期間](/ja/docs/manage-users/sessions/configure-session-lifetime)によって決まる                       | [トークンの有効期間](/ja/docs/secure/tokens/refresh-tokens/configure-refresh-token-expiration)で設定 |
| **トークンローテーション**                                                                                    | ローテーションしない                                                                                        | [ローテーション](/ja/docs/secure/tokens/refresh-tokens/configure-refresh-token-rotation)をサポート   |
| **[所持証明の実証](/ja/docs/secure/sender-constraining/demonstrating-proof-of-possession-dpop)  (DPoP) の要件** | パブリッククライアント (SPA、ネイティブ) の場合は必須                                                                    | 任意 (リソースサーバーの設定)                                                                      |
| **失効**                                                                                             | [セッション全体を失効させる](/ja/docs/manage-users/sessions/manage-sessions-actions#revoke-sessions-with-actions) | [トークンのみを失効させる](/ja/docs/secure/tokens/refresh-tokens/revoke-refresh-tokens)              |
| **Management API**                                                                                 | 表示されない                                                                                            | Management API で表示される                                                                 |
| **セッションのアイドルタイムアウト**                                                                               | 交換のたびに延長される                                                                                       | セッションに影響しない                                                                           |

<div id="key-behavioral-differences">
  ### 主な動作の違い
</div>

**セッションの延長:** ORT を交換すると、Auth0 はセッションのアイドルタイムアウトをリセットし、ユーザーセッションを維持します。Auth0 はセッションの[絶対有効期間](/ja/docs/manage-users/sessions/session-lifecycle#absolute-timeout-maximum)を延長しません。

**DPoP の適用:** パブリッククライアント (SPA など) では、ORT の使用時に Auth0 は DPoP を必須とします。ORT は、ユーザーの認証済みセッションへのアクセスを許可するセッション Cookie と同等のセキュリティレベルで機能するため、同等に厳重に保護する必要があります。DPoP は、トークンの窃取やリプレイ攻撃を防ぎます。

**ステートレス:** ORT はセッションに紐付けられているため、個別に参照または管理することはできません。Management API で確認したり、個別に失効させたりすることもできません。

<div id="best-practices">
  ## ベストプラクティス
</div>

1. **トークンはメモリに保存する:** ORT を `localStorage` や `sessionStorage` に保存しないでください。セキュリティを高めるには、JavaScript メモリ内に保持してください。ORT は Auth0 セッションに紐付けられているため、ユーザーが次回アクセスしたときにセッションクッキーによってコンテキストが再確立され、アプリは新しい ORT を要求できます。
2. **セッションの有効期限切れを適切に処理する:** セッションの有効期限が切れた場合に備えて、適切なエラーハンドリングを実装してください。ユーザーの作業内容を失わないようにしながら、再認証にリダイレクトしてください。
3. **パブリッククライアントでは DPoP を使用する:** パブリッククライアントでは、ORT に DPoP が必要です。実装で DPoP プルーフが正しく生成・使用されていることを確認してください。
4. **トークンの構造に依存しない:** ORT は不透明な文字列として扱ってください。解析したり、内部構造に依存したりしないでください。内部構造は変更される可能性があります。
5. **セッションの有効期間を確認する:** アイドルタイムアウトと絶対タイムアウトの両方を把握しておいてください。絶対タイムアウトは、リフレッシュトークンの交換では延長できません。

<div id="limitations">
  ## 制限事項
</div>

<div id="unsupported-flows">
  ### サポートされていないフロー
</div>

ORT は、インタラクティブなブラウザーセッションを作成するフローでのみ使用できます。次のフローでは ORT を発行できません。

| **フロー**                                                                                                               | **理由**                                           |
| --------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------ |
| [Device Authorization Flow](/ja/docs/get-started/authentication-and-authorization-flow/device-authorization-flow)        | セッションはブラウザーで作成される一方、トークンは別のデバイスで使用されるため          |
| [Implicit Flow](/ja/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post)                 | OAuth 仕様で implicit flow におけるリフレッシュトークンが禁止されているため |
| [Resource Owner Password Grant](/ja/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow) | ブラウザーセッションが作成されないため                              |
| [Client Credentials Flow](/ja/docs/get-started/authentication-and-authorization-flow/client-credentials-flow)            | ユーザーもセッションも関与しないため                               |
| [Custom Token Exchange](/ja/docs/get-started/authentication-and-authorization-flow/token-exchange-flow)                  | トークンを紐付けるセッションがないため                              |

これらのフローのいずれかで `online_access` スコープを要求しても、そのスコープはエラーなく無視され、リフレッシュトークンは発行されません。

<div id="scope-conflicts">
  ### スコープの競合
</div>

同一の認可リクエストで `online_access` と `offline_access` の両方を要求することはできません。両方のスコープが含まれている場合、Auth0 はエラーを返します。

```json
{
  "error": "invalid_request",
  "error_description": "Cannot request both online_access and offline_access scopes"
}
```

ユースケースに応じて、適切なスコープを選択してください。

* SSO の継続性を活用できるブラウザーベースのアプリケーションには、`online_access` を使用します。
* ブラウザーセッションに依存せずにトークンを機能させる必要があるアプリケーションには、`offline_access` を使用します。
