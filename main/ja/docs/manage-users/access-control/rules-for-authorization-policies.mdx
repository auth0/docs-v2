事前設定された認可ポリシーに [Rules](/ja/docs/customize/rules) を追加すると、ユーザーアクセスを許可または拒否する条件をさらに細かく制御できます。Rule には、適切なロジックに基づいて認可の判断を行うカスタムコードが含まれます。他の Rule と組み合わせることで、さまざまな状況での動作を定義するのに役立ちます。

Rules では、ユーザーの部署、時刻、場所、またはその他のユーザー属性や API 属性 (username、セキュリティクリアランス、API 名など) といった、ユーザーに保存されている属性の任意の組み合わせに基づいてアクセスを制限できます。

たとえば、非営利組織で細かなアクセス制御を実現するために Rules を使用している場合、ニューデリーオフィスの研究開発部門で勤務する W2 従業員にのみ、あるアプリケーションへのアクセスを許可できます。

認可ポリシーでの Rule 実装のサンプルについては、[Sample Use Cases: Rules with Authorization](/ja/docs/manage-users/access-control/sample-use-cases-rules-with-authorization) を参照してください。

<div id="rules-in-the-authorization-process">
  ## 認可プロセスにおける Rules
</div>

Rules は、実行される順序に応じて、権限が <Tooltip tip="アクセストークン: API へのアクセスに使用される、不透明な文字列または JWT 形式の認可資格情報。" cta="用語集を見る" href="/ja/docs/glossary?term=Access+Token">アクセストークン</Tooltip> に追加される前に、認可の判定結果を変更できます。Rules を組み込んだ基本的なプロセスは次のとおりです。

1. ユーザーがアプリケーションへの認証を試みます。
2. Auth0 はリクエストを選択された IDプロバイダー に送信します。
3. IDプロバイダーがユーザーの認証情報が有効であることを確認すると、作成済みのすべての Rules が Dashboard で設定された順序で実行されます。
4. いずれの Rule でもユーザーのアクセスが制限されていなければ、ユーザーにはアプリケーションへのアクセスが許可されます。
