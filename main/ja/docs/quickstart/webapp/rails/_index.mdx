    <Section id={sections[0].id} title={sections[0].title} stepNumber="1" isSingleColumn>
      Auth0 のサービスを利用するには、Auth0 Dashboard でアプリケーションを設定しておく必要があります。Auth0 アプリケーションでは、
      開発中のプロジェクトで認証をどのように機能させるかを設定します。

      ### アプリケーションを設定する

      インタラクティブなセレクタを使用して、新しい Auth0 アプリケーションを作成するか、
      連携するプロジェクトに対応する既存のアプリケーションを選択します。Auth0 の各アプリケーションには、
      SDK を介して Auth0 API を呼び出す際にアプリケーションコードで使用する、英数字の一意なクライアントID が割り当てられます。

      このクイックスタートで行った設定は、[Dashboard](https://manage.auth0.com/#/) 内のアプリケーションにも自動的に反映されます。以降のアプリケーション管理も、
      ここで行えます。

      設定済みの完全な構成を確認したい場合は、代わりにサンプルアプリケーションを参照することもできます。

      ### コールバック URL を設定する

      コールバック URL とは、認証後に Auth0 がユーザーをリダイレクトする先の、
      アプリケーション内 URL です。これが設定されていない場合、ユーザーはログイン後にアプリケーションへ戻れません。

      <Info>
        サンプルプロジェクトに沿って進めている場合は、これを
        `http://localhost:3000/auth/auth0/callback` に設定してください。
      </Info>

      ### ログアウト URL を設定する

      ログアウト URL とは、ログアウト後に Auth0 がユーザーをリダイレクトする先の、
      アプリケーション内 URL です。これが設定されていない場合、ユーザーはアプリケーションからログアウトできず、エラーが発生します。

      <Info>
        サンプルプロジェクトに沿って進めている場合は、これを `http://localhost:3000` に設定してください。
      </Info>

      ### 許可する Web オリジンを設定する

      許可する Web オリジンとは、認証フローへのアクセスを許可する URL です。ここには
      プロジェクトの URL を含める必要があります。これが正しく設定されていない場合、プロジェクトは認証トークンを
      サイレントに更新できないため、次回ユーザーがアプリケーションにアクセスしたとき、またはページを更新したときに
      ログアウトされます。

      <Info>
        サンプルプロジェクトに沿って進めている場合は、これを `http://localhost:3000` に設定してください。
      </Info>

      認証フローの処理には、カスタムの [OmniAuth strategy](https://github.com/intridea/omniauth#omniauth-standardized-multi-provider-authentication) である `omniauth-auth0` を使用します。

      次の依存関係を `Gemfile` に追加します。

      ```gem lines
      gem 'omniauth-auth0', '~> 3.0'
      gem 'omniauth-rails_csrf_protection', '~> 1.0' # 偽造された認証リクエストを防ぐ
      ```

      gem を追加したら、`bundle install` を実行して gem をインストールします。

      認証コールバック、`logout` アクション、およびログアウト URL を構築するメソッドを処理する Auth0 コントローラーを作成します。

      次のコマンドを実行します。
      `rails generate controller auth0 callback failure logout --skip-assets --skip-helper --skip-routes --skip-template-engine`.

      callback メソッド内で、`request.env['omniauth.auth']` として返されるユーザー情報のハッシュを現在のセッションに代入します。

      ログアウトを設定するには、`logout` アクション内で `reset_session` メソッドを呼び出し、セッション内に保存されているすべてのオブジェクトをクリアします。次に、Auth0 のログアウトエンドポイントにリダイレクトします。`reset_session` の詳細については、
      [Ruby on Rails ActionController documentation](http://api.rubyonrails.org/classes/ActionController/Base.html#M000668) を参照してください。

    <Section id={sections[8].id} title={sections[8].title} stepNumber="9" isSingleColumn>
      ユーザープロファイルを表示するには、アプリケーションで保護されたルートを用意する必要があります。複数の
      コントローラー間で共有できるルートへのアクセスを制御するには、[Concern](https://guides.rubyonrails.org/getting_started.html#using-concerns) を使用できます。ユーザーが未認証の場合、
      Concern は自動的に Auth0 にリダイレクトするようにする必要があります。認証済みの場合は、
      Concern が現在のユーザープロファイルを返すようにする必要があります。

      Concern を用意したら、ログイン済みユーザーを必要とする任意のコントローラーにこれを含めます。すると、次の例のように
      セッション `session[:userinfo]` からユーザーにアクセスできます。

      ```ruby lines
      class DashboardController < ApplicationController
      include Secured
      def show
      @user = session[:userinfo]

      end
      end
      ```

      ユーザーがセッションから読み込まれたら、その情報を使ってフロントエンドに表示します。

      ```html lines
      <div>
      <p>Normalized User Profile:<%= JSON.pretty_generate(@user[:info])%></p>
      <p>完全なユーザープロファイル:<%= JSON.pretty_generate(@user[:extra][:raw_info])%></p>
      </div>
      ```

      <Note>
        ##### チェックポイント

        アプリに `Secured` concern を追加し、認証済みユーザーのみがアクセスできるコントローラーにこれを組み込みます。認証済みのユーザーがその
        コントローラー内のアクションにアクセスできること、また未認証のユーザーが認証のために Auth0 にリダイレクトされることを確認します。
      </Note>
