  <Content>
    このチュートリアルでは、Device Authorization Flowを使用して、入力が制限されたデバイスからAPIを呼び出す方法を説明します。アカウント用に設定されたサンプルを使ってこのクイックスタートを進めるには、ログインすることをお勧めします。

    インタラクティブに操作を試すには、[Device Flow プレイグラウンド](https://auth0.github.io/device-flow-playground/)をご利用ください。

    <Section id={sections[0].id} title={sections[0].title} stepNumber="1">
      * ネイティブアプリケーションを登録します。
      * **OIDC Conformant** トグルが有効になっていることを確認します。詳細については、OIDC-Conformant Authentication を参照してください。
      * アプリケーションの grant types に **Device Code** を追加します。詳細については、Update Grant Types を参照してください。
      * リフレッシュトークンを有効にする場合は、アプリケーションの grant types に **Refresh Token** を追加します。
      * アプリケーションに対して少なくとも 1 つの接続を
        設定し、有効にします。
      * Auth0 に API を登録します。
      * リフレッシュトークンを使用する場合は、**Allow Offline Access** を有効にします。詳細については、API Settings を参照してください。
      * Device User Code Settings を設定して、ランダムに生成されるユーザー code の
        文字セット、形式、長さを定義します。
    </Section>

      デバイスアプリケーションが `device_code` と `user_code` を受け取ったら、
      ユーザーに `verification_uri` へアクセスして `user_code` を入力するよう案内します。

    <Section id={sections[4].id} title={sections[4].title} stepNumber="5">
      デバイスアプリケーションは、ユーザーがアクティブ化するまでの間、Authentication API の [POST /oauth/token エンドポイント](https://auth0.com/docs/api/authentication#-post-oauth-token-) を定期的に呼び出し、
      レスポンスを適切に処理する必要があります。

      <Info>
        ネットワーク遅延の問題を避けるため、デバイスアプリケーションは `interval` の値 (秒単位) 以上、または
        正常なレスポンスを受信するまでのいずれか長い方の時間だけ待機するようにしてください。
      </Info>

      ```bash
      curl --request POST \ 
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=urn:ietf:params:oauth:grant-type:device_code \
        --data device_code=AUTH0_SCOPES \
        --data 'client_id={yourClientId}'
      ```
    </Section>

    <Section id={sections[6].id} title={sections[6].title} stepNumber="7">
      ユーザーがデバイスアプリケーションを承認すると、そのアプリケーションは HTTP 200 レスポンスと次のペイロードを受け取ります。

      ```json lines
      {
        "access_token": "eyJz93a...k4laUWw",
        "refresh_token": "GEbRxBN...edjnXbL",
        "id_token": "eyJ0XAi...4faeEoQ",
        "token_type": "Bearer",
        "expires_in": 86400
      }
      ```

      <Info>
        トークンは保存する前に検証してください。方法については、Validate Access Tokens と Validate ID Tokens を参照してください。
      </Info>

      アクセストークンは、Authentication API の [Get User Info endpoint](https://auth0.com/docs/api/authentication#get-user-info) の呼び出し (デバイス
      アプリケーションが `openid` スコープを要求した場合) や、`audience`
      パラメータで指定した API の呼び出しに使用されます。独自の API を呼び出す場合は、使用前にデバイスアプリケーションでアクセストークンを検証する必要があります。

      IDトークンには、[デコードして取り出す](/ja/docs/secure/tokens/id-tokens#id-token-payload) 必要があるユーザー情報が含まれています。Authentication API
      は、デバイスアプリケーションが `openid` スコープを要求した場合にのみ `id_token` を返します。

      リフレッシュトークンは、以前のトークンの有効期限が切れた後に、新しいアクセストークンまたは
      IDトークンを取得するために使用されます。Authentication API は、`audience` パラメータで指定された API で **Allow Offline Access** 設定が有効になっており、
      かつデバイスアプリケーションが `offline_access` スコープを要求した場合にのみ `refresh_token` を返します。
    </Section>

    <Section id={sections[7].id} title={sections[7].title} stepNumber="8">
      API を呼び出すには、デバイスアプリケーションは HTTP リクエストの
      `Authorization` ヘッダーで、アクセストークンを Bearer トークンとして送信する必要があります。

      ```bash lines
      curl --request GET \
        --url https://myapi.com/api \
        --header 'authorization: Bearer AUTH0_API_ACCESS_TOKEN' \
        --header 'content-type: application/json'
      ```
    </Section>

    <Section id={sections[8].id} title={sections[8].title} stepNumber="9">
      ユーザーの新しいアクセストークンを取得するには、デバイスアプリケーションから
      `refresh_token` パラメーターを指定して、Authentication API の [POST /oauth/token エンドポイント](https://auth0.com/docs/api/authentication#-post-oauth-token-) を呼び出すことができます。

      ```bash lines
      curl --request POST \
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=refresh_token \
        --data 'client_id={yourClientId}' \
        --data 'client_secret={yourClientSecret}' \
        --data refresh_token=AUTH0_REFRESH_TOKEN
      ```

      リクエストが成功すると、デバイスアプリケーションは次のペイロードを含む HTTP 200 レスポンスを受け取ります。

      ```json lines
      {
        "access_token": "eyJ...MoQ",
        "expires_in": 86400,
        "scope": "openid offline_access",
        "id_token": "eyJ...0NE",
        "token_type": "Bearer"
      }
      ```

      リフレッシュトークンの詳細については、[Refresh Tokens](https://auth0.com/docs/secure/tokens/refresh-tokens)をご覧ください。
    </Section>

    <Section id={sections[9].id} title={sections[9].title} stepNumber="10">
      テナントログは、発生したあらゆる操作について作成され、問題のトラブルシューティングに使用できます。

      ### トークンレスポンス

      ユーザーがデバイスを認可するのを待っている間に、いくつかの異なる HTTP 4xx レスポンスが返されることがあります。

      #### 認可待ち

      このエラーは、ユーザーが操作を行うのを待っている間に表示されます。このチュートリアルの前の手順で取得した推奨の `interval` を使用して、ポーリングを続けてください。

      ```json lines
      `HTTP 403`
      {
      "error": "authorization_pending",
      "error_description": "..."
      }
      ```

      #### 間隔を空けてください

      ポーリングの間隔が短すぎます。間隔を空け、この
      チュートリアルの前の手順で取得した推奨間隔を使用してください。ネットワーク遅延が原因でこのエラーが発生するのを防ぐには、各間隔の計測を
      直前のポーリングリクエストに対するレスポンスを受信した後に開始してください。

      ```json lines
      `HTTP 429`
      {
      "error": "slow_down",
      "error_description": "..."
      }
      ```

      #### 期限切れのトークン

      ユーザーが十分な時間内にデバイスを承認しなかったため、`device_code` の有効期限が切れています。アプリケーションでは、フローの有効期限が切れたことをユーザーに通知し、フローを再開するよう促す必要があります。

      <Info>
        `expired_token` エラーは 1 回だけ返されます。その後、このエンドポイントは
        `invalid_grant` エラーを返します。
      </Info>

      ```json lines
      `HTTP 403`
      {
      "error": "expired_token",
      "error_description": "..."
      }
      ```

      #### アクセス拒否

      アクセスが拒否された場合は、次のレスポンスが返されます。

      ```json lines
      `HTTP 403`
      {
      "error": "access_denied",
      "error_description": "..."
      }
      ```

      これは、次のようなさまざまな理由で発生することがあります。

      * ユーザーがデバイスへの認可を拒否した。
      * 認可サーバーがトランザクションを拒否した。
      * 設定された Action がアクセスを拒否した。
    </Section>

    <Section id={sections[11].id} title={sections[11].title} stepNumber="12">
      Device Authorization Flow を使用するには、デバイスアプリケーションが次の要件を満たしている必要があります。

      * Server Name Indication (SNI) をサポートしていること
      * Native Application であること
      * [Authentication Method](/ja/docs/secure/application-credentials#application-authentication-methods) が **None** に設定されていること
      * [OIDC-conformant](/ja/docs/get-started/applications/application-settings#oauth) であること
      * Dynamic Client Registration を使用して作成されていないこと

      さらに、Device Authorization Flow では次のことは許可されません。

      * New Universal Login Experience を使用している場合を除き、Auth0 の開発者キーを使用するソーシャル接続
      * ホストされたログインページまたは Actions からクエリ文字列パラメーターにアクセスすること。
    </Section>
