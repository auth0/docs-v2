      デバイスアプリケーションがユーザーによる有効化を待っている間は、Authentication API の [POST /oauth/token エンドポイント](https://auth0.com/docs/api/authentication#-post-oauth-token-) を一定間隔で呼び出し、
      レスポンスを適切に処理する必要があります。

      <Info>
        ネットワーク遅延の問題を回避するため、デバイスアプリケーションは少なくとも `interval` の値 (秒単位) だけ、または
        正常なレスポンスを受信するまでのいずれか長い方の時間だけ待機してください。
      </Info>

      ```bash
      curl --request POST \ 
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=urn:ietf:params:oauth:grant-type:device_code \
        --data device_code=AUTH0_SCOPES \
        --data 'client_id={yourClientId}'
      ```

    <Section id={sections[6].id} title={sections[6].title} stepNumber="7" isSingleColumn>
      ユーザーがデバイスアプリケーションを認可すると、デバイスアプリケーションは HTTP 200 レスポンスと次のペイロードを受け取ります。

      ```json lines
      {
        "access_token": "eyJz93a...k4laUWw",
        "refresh_token": "GEbRxBN...edjnXbL",
        "id_token": "eyJ0XAi...4faeEoQ",
        "token_type": "Bearer",
        "expires_in": 86400
      }
      ```

      <Info>
        トークンは保存する前に検証してください。方法については、[アクセストークンの検証](https://auth0.com/docs/secure/tokens/access-tokens/validate-access-tokens)および[IDトークンの検証](https://auth0.com/docs/secure/tokens/id-tokens/validate-id-tokens)を参照してください。
      </Info>

      アクセストークンは、Authentication API の [Get User Info エンドポイント](https://auth0.com/docs/api/authentication#get-user-info) (デバイス
      アプリケーションが `openid` スコープを要求した場合) または `audience`
      パラメーターで指定された API を呼び出すために使用されます。独自の API を呼び出す場合は、使用する前にデバイスアプリケーションでアクセストークンを検証する必要があります。

      IDトークンには、[デコードして抽出する](/ja/docs/secure/tokens/id-tokens#id-token-payload)必要があるユーザー情報が含まれています。Authentication API
      は、デバイスアプリケーションが `openid` スコープを要求した場合にのみ `id_token` を返します。

      リフレッシュトークンは、以前のトークンの有効期限が切れた後に、新しいアクセストークンまたは
      IDトークンを取得するために使用されます。Authentication API は、`audience` パラメーターで指定された API で
      **Allow Offline Access** 設定が有効になっており、かつデバイスアプリケーションが `offline_access` スコープを要求した場合にのみ `refresh_token` を返します。


    <Section id={sections[8].id} title={sections[8].title} stepNumber="9" isSingleColumn>
      ユーザーの新しいアクセストークンを取得するには、デバイスアプリケーションから
      `refresh_token` パラメーターを指定して、Authentication API の [POST /oauth/token エンドポイント](https://auth0.com/docs/api/authentication#-post-oauth-token-) を呼び出すことができます。

      ```bash lines
      curl --request POST \
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=refresh_token \
        --data 'client_id={yourClientId}' \
        --data 'client_secret={yourClientSecret}' \
        --data refresh_token=AUTH0_REFRESH_TOKEN
      ```

      リクエストが成功すると、デバイスアプリケーションは次のペイロードを含む HTTP 200 レスポンスを受け取ります。

      ```json lines
      {
        "access_token": "eyJ...MoQ",
        "expires_in": 86400,
        "scope": "openid offline_access",
        "id_token": "eyJ...0NE",
        "token_type": "Bearer"
      }
      ```

      リフレッシュトークンの詳細については、[Refresh Tokens](https://auth0.com/docs/secure/tokens/refresh-tokens)を参照してください。

    <Section id={sections[9].id} title={sections[9].title} stepNumber="10" isSingleColumn>
      テナントログは、発生したあらゆるやり取りについて作成され、
      問題のトラブルシューティングに利用できます。

      ### トークンレスポンス

      ユーザーがデバイスを認可するのを待っている間に、いくつかの異なる HTTP 4xx レスポンスを受け取ることがあります。

      #### 認可待ち

      ユーザーが操作を行うのを待っている間は、このエラーが表示されます。このチュートリアルの前の手順で取得した推奨の
      `interval` を使用して、ポーリングを継続してください。

      ```json lines
      `HTTP 403`
      {
      "error": "authorization_pending",
      "error_description": "..."
      }
      ```

      #### 間隔を空けてください

      ポーリングの間隔が短すぎます。間隔を空け、この
      チュートリアルの前の手順で取得した推奨間隔を使用してください。ネットワーク遅延が原因でこのエラーを受け取らないようにするには、最後のポーリングリクエストへのレスポンスを
      受信してから、各間隔の計測を開始してください。

      ```json lines
      `HTTP 429`
      {
      "error": "slow_down",
      "error_description": "..."
      }
      ```

      #### 期限切れのトークン

      ユーザーがデバイスを承認するまでに時間がかかりすぎたため、`device_code` の有効期限が切れています。アプリケーションは、フローの有効期限が切れたことをユーザーに通知し、フローを再開するよう促す必要があります。

      <Info>
        `expired_token` エラーは、1 回だけ返されます。その後、エンドポイントは
        `invalid_grant` エラーを返します。
      </Info>

      ```json lines
      `HTTP 403`
      {
      "error": "expired_token",
      "error_description": "..."
      }
      ```

      #### アクセス拒否

      アクセスが拒否された場合、次が返されます。

      ```json lines
      `HTTP 403`
      {
      "error": "access_denied",
      "error_description": "..."
      }
      ```

      これは、次のようなさまざまな理由で発生する可能性があります。

      * ユーザーがデバイスの認可を拒否した。
      * 認可サーバーがトランザクションを拒否した。
      * 設定済みのActionがアクセスを拒否した。

    <Section id={sections[11].id} title={sections[11].title} stepNumber="12" isSingleColumn>
      デバイス認可フローを使用するには、デバイスアプリケーションが次の要件を満たしている必要があります。

      * Server Name Indication (SNI) をサポートしていること
      * ネイティブアプリケーションであること
      * [認証方法](/ja/docs/secure/application-credentials#application-authentication-methods) が **None** に設定されていること
      * [OIDC 準拠](/ja/docs/get-started/applications/application-settings#oauth) であること
      * 動的クライアント登録によって作成されていないこと

      さらに、デバイス認可フローでは次のことは許可されません。

      * New Universal Login Experience を使用している場合を除き、Auth0 開発者キーを使用するソーシャル接続
      * ホストされたログインページまたは Actions からクエリ文字列パラメーターにアクセスすること
