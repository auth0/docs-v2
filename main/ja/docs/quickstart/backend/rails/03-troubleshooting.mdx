このドキュメントは、API から想定外のレスポンスが返された場合に、設定をトラブルシューティングする際に役立ちます。アカウント向けに設定されたサンプルを使ってこのクイックスタートを進められるよう、ログインしておくことをお勧めします。

JWT の検証が正しく設定されていれば、リクエストの送信時に API から適切なレスポンスを取得できます。ただし、API から 401 (Unauthorized) レスポンスが返される場合は、JWT ミドルウェアの設定が、渡された JWT と一致していないことが原因です。

このドキュメントは、JWT ミドルウェアの設定をトラブルシューティングする際に役立ちます。

<div id="how-does-a-token-get-validated">
  ## トークンはどのように検証されますか？
</div>

JWT を検証する際は、さまざまな点を確認する必要があります。

1. **トークンの形式は正しいですか？** つまり、このトークンは JSON Web Token (JWT) の構造に準拠していますか？ JWT の構造の詳細については、[JWT の構造に関するこのセクション](/ja/docs/secure/tokens/json-web-tokens#what-is-the-json-web-token-structure-) を参照してください。
2. **トークンは改ざんされていませんか？** JWT の最後の部分は署名です。この署名は、トークンが実際に送信者によって署名されており、途中で一切変更されていないことを検証するために使用されます。
3. **トークンは有効期間内に受信されましたか？** JWT は、指定された一定期間のみ有効です (`exp` クレームで表されます) 。
4. **トークンは想定した発行元から送られてきていますか？** これには 2 つの要素があります

   * **署名の検証**: 発行元が発行したキーを使用して、JWT が正しく署名されていることを確認できますか？
   * **Issuer 値**: Issuer は `iss` クレームで定義されます。ここでも、このクレームはアプリケーションが想定している値と一致していますか？
5. **トークンは現在のアプリケーション向けですか？** つまり、JWT の `aud` クレームはアプリケーションが想定している値と一致していますか？

<div id="inspecting-a-token">
  ## トークンを確認する
</div>

JWT を手早く確認するには、[JWT.io](https://jwt.io/) Web サイトを使用します。便利なデバッガーが用意されており、JWT の形式が正しいかどうかをすばやく確認できるほか、さまざまなクレームの値も確認できます。

<Frame>![JWT.io で JWT をデバッグする](https://cdn2.auth0.com/docs/1.14550.0/media/articles/server-apis/aspnet-core-webapi/jwt-io-debugger-rs256.png)</Frame>

上のスクリーンショットでは、トークンが **RS256** アルゴリズムで署名されていることがわかります。トークンの **Issuer** は **[https://jerrie.auth0.com/](https://jerrie.auth0.com/)**、**対象者** は **[https://rs256.test.api](https://rs256.test.api)** です。

つまり、JWT 検証の構成で設定するこれらの値は、Issuer の末尾のスラッシュも含めて、**完全に** 一致している必要があります。たとえば次のようになります

```
audience = "https://rs256.test.api",
domain = "https://jerrie.auth0.com/"
};
```

HS256 で署名されたトークンの場合、デバッガービューは少し異なります。
