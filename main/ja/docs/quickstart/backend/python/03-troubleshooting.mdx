このドキュメントでは、API から想定外のレスポンスが返された場合に、設定をトラブルシューティングする方法を説明します。このクイックスタートは、ご利用のアカウント向けに設定された例を使って進められるよう、ログインした状態で進めることをお勧めします。

JWT の検証が正しく設定されていれば、リクエスト時に API から適切なレスポンスを取得できます。ただし、API から 401 (Unauthorized) レスポンスが返される場合は、JWT ミドルウェアの設定が渡された JWT と一致していないことが原因です。

このドキュメントは、JWT ミドルウェアの設定をトラブルシューティングする際に役立ちます。

<div id="how-does-a-token-get-validated">
  ## トークンはどのように検証されますか？
</div>

JWT を検証する際には、いくつか確認すべき点があります。

1. **トークンの形式は正しいですか？** つまり、このトークンは JSON Web Token (JWT) の構造に準拠していますか？ JWT の構造について詳しくは、[JWT の構造に関するこちらのセクション](/ja/docs/secure/tokens/json-web-tokens#what-is-the-json-web-token-structure-) を参照してください。
2. **トークンは改ざんされていませんか？** JWT の最後の部分は署名です。署名は、そのトークンが実際に送信者によって署名されており、途中で一切変更されていないことを検証するために使用されます。
3. **トークンは有効期間内に受信されましたか？** JWT は、指定された期間内でのみ有効です (`exp` クレームで表されます) 。
4. **トークンは想定された発行元から送信されていますか？** これには 2 つの要素があります

   * **署名の検証**: 発行元が発行したキーを使用して、JWT が正しく署名されていることを確認できますか？
   * **発行者 値**: 発行者 は `iss` クレームで定義されます。このクレームは、アプリケーションが想定している値と一致していますか？
5. **トークンは現在のアプリケーションを対象としていますか？** つまり、JWT の `aud` クレームは、アプリケーションが想定している値と一致していますか？

<div id="inspecting-a-token">
  ## トークンの確認
</div>

JWT を手早く確認するには、[JWT.io](https://jwt.io/) Web サイトを使用する方法があります。便利なデバッガーが用意されており、JWT の形式が正しいかをすばやく確認できるほか、各種クレームの値も確認できます。

<Frame>![JWT.io で JWT をデバッグする](https://cdn2.auth0.com/docs/1.14550.0/media/articles/server-apis/aspnet-core-webapi/jwt-io-debugger-rs256.png)</Frame>

上のスクリーンショットから、このトークンが **RS256** アルゴリズムで署名されていることがわかります。トークンの **発行者** は **[https://jerrie.auth0.com/](https://jerrie.auth0.com/)**、**対象者** は **[https://rs256.test.api](https://rs256.test.api)** です。

つまり、JWT 検証の構成登録におけるこれらの値は **完全に** 一致している必要があります。たとえば、発行者の末尾のスラッシュも含めて一致していなければなりません。

```
audience = "https://rs256.test.api",
domain = "https://jerrie.auth0.com/"
};
```

HS256 で署名されたトークンの場合、デバッガー画面は少し異なります。
