    このガイドでは、[Django](https://www.djangoproject.com/) で構築した新規または既存の Python API に Auth0 を統合する方法を説明します。

    Auth0 Dashboardでまだ APIを作成していない場合は、インタラクティブセレクターを使用して新しい Auth0 APIを作成するか、統合するプロジェクトに対応する既存の APIを選択できます。

    または、入門ガイドをご覧ください。Auth0 Dashboardを使って最初のAPIを設定する手順を説明しています。

    Auth0 のすべての API は、アクセストークンの検証時にアプリケーションコードがオーディエンスとして使用する API 識別子を用いて設定されます。

    <Section id={sections[0].id} title={sections[0].title} stepNumber="1">
      Permissions を使用すると、指定したアクセストークンでユーザーに代わってリソースへどのようにアクセスできるかを定義できます。たとえば、
      ユーザーのアクセスレベルが manager の場合は `messages` リソースへの読み取りアクセスを付与し、
      administrator の場合はそのリソースへの書き込みアクセスを付与できます。

      許可する権限は、Auth0 Dashboard の [APIs](https://manage.auth0.com/#/apis) セクションにある **Permissions** ビューで定義できます。次の
      例では、`read:messages` スコープを使用します。

      ### Django アプリケーションを作成する

    <Section id={sections[2].id} title={sections[2].title} stepNumber="3">
      [Authlib](https://github.com/lepture/authlib) というライブラリを使用して、[ResourceProtector](https://docs.authlib.org/en/latest/flask/1/resource-server.html) を作成します。これは、指定した
      バリデーターでリソース (API ビュー) を保護する [Django view decorator](https://docs.djangoproject.com/en/4.0/topics/http/decorators/) の一種です。

      このバリデーターは、リソースに渡されたアクセストークンの
      署名とクレームが有効であることを確認して検証します。

      AuthLib の `JWTBearerTokenValidator` は、いくつか調整を加えることで、
      [アクセストークンの検証](https://auth0.com/docs/secure/tokens/access-tokens/validate-access-tokens) に関する Auth0 の要件に適合するように使用できます。

      `Auth0JWTBearerTokenValidator` を作成するには、`domain` と
      `audience` (API Identifier) を渡す必要があります。これにより、トークン署名の検証に必要な公開鍵を取得し、
      それを `JWTBearerTokenValidator` クラスに渡します。

      次に、クラスの `claims_options` をオーバーライドして、トークンの `expiry`、
      `audience`、および `issue` クレームが Auth0 の要件に従って検証されるようにします。

      インタラクティブパネルのコードを使用して、`apiexample/validator.py` ファイルを作成します。
    </Section>

    <Section id={sections[3].id} title={sections[3].title} stepNumber="4">
      次に、`apiexample/views.py` に 3 つの API ビューを作成します。

      * `/api/public`: 認証不要のパブリックエンドポイント。
      * `/api/private`: 有効なアクセストークン JWT が必要なプライベートエンドポイント。
      * `/api/private-scoped`: 指定された `scope` を含む有効なアクセストークン JWT が必要な
        プライベートエンドポイント。

      保護されたルートには `require_auth` デコレータを付けます。これは、先ほど作成した `Auth0JWTBearerTokenValidator` を使用する `ResourceProtector`
      です。

      `Auth0JWTBearerTokenValidator` を作成するには、テナントのドメインと、先ほど作成した API の API
      Identifier を渡します。

      `private_scoped` ルートの `require_auth` デコレータは、追加の引数
      `"read:messages"` を受け取ります。これにより、先ほど作成した Permission (スコープ) がアクセストークンに含まれているかを確認します。
    </Section>

    <Section id={sections[4].id} title={sections[4].title} stepNumber="5">
      前の手順では、`views.py` ファイルにメソッドを追加しました。次に、URLパターンをビューにマッピングできるDjangoの[URLディスパッチャー](https://docs.djangoproject.com/en/4.0/topics/http/urls/)を使用して、これらのメソッドをURLにマッピングします。

      URLパターンを `apiexample/urls.py` ファイルに追加します。
