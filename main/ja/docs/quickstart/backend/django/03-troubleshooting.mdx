このドキュメントでは、API から予期しないレスポンスが返された場合の設定のトラブルシューティング方法を説明します。アカウント用に構成された例を使用してこのクイックスタートを進められるよう、ログインしておくことをお勧めします。

JWT の検証を正しく設定していれば、リクエスト時に API から適切なレスポンスを取得できます。ただし、API から 401 (Unauthorized) レスポンスが返される場合は、JWT ミドルウェアの設定が渡された JWT と一致していないことが原因です。

このドキュメントは、JWT ミドルウェア設定のトラブルシューティングに役立ちます。

<div id="how-does-a-token-get-validated">
  ## トークンはどのように検証されますか？
</div>

JWT を検証する際には、考慮すべき点がいくつかあります。

1. **トークンは正しい形式ですか？** つまり、このトークンは JSON Web Token (JWT) の構造に準拠していますか？ JWT の構造の詳細については、[JWT の構造に関するこのセクション](/ja/docs/secure/tokens/json-web-tokens#what-is-the-json-web-token-structure-)を参照してください。
2. **トークンは改ざんされていませんか？** JWT の最後の部分は署名です。この署名は、トークンが実際に送信者によって署名されたものであり、途中で一切変更されていないことを検証するために使用されます。
3. **トークンは有効期間内に受信されましたか？** JWT は指定された一定期間のみ有効です (`exp` クレームで表されます) 。
4. **トークンは想定された発行元から送信されていますか？** これには 2 つの要素があります

   * **署名の検証**: JWT が、発行元によって発行された鍵を使用して正しく署名されていることを確認できますか？
   * **Issuer 値**: Issuer は `iss` クレームで定義されます。ここでも、このクレームがアプリケーションで想定している値と一致しているかを確認します。
5. **トークンは現在のアプリケーション向けのものですか？** つまり、JWT の `aud` クレームはアプリケーションが想定している値と一致していますか？

<div id="inspecting-a-token">
  ## トークンの確認
</div>

JWT を手早く確認する方法の 1 つは、[JWT.io](https://jwt.io/) の Web サイトを使うことです。便利なデバッガーが用意されており、JWT の形式が正しいかどうかをすばやく確認できるほか、各種クレームの値も確認できます。

<Frame>![JWT.io で JWT をデバッグする](https://cdn2.auth0.com/docs/1.14550.0/media/articles/server-apis/aspnet-core-webapi/jwt-io-debugger-rs256.png)</Frame>

上のスクリーンショットでは、トークンが **RS256** アルゴリズムで署名されていることがわかります。トークンの **Issuer** は **[https://jerrie.auth0.com/](https://jerrie.auth0.com/)** で、**対象者** は **[https://rs256.test.api](https://rs256.test.api)** です。

つまり、JWT 検証設定に登録するこれらの値は、**完全に** 一致している必要があります。たとえば、Issuer の末尾のスラッシュも含めて、次のように厳密に一致していなければなりません

```
audience = "https://rs256.test.api",
domain = "https://jerrie.auth0.com/"
};
```

HS256 で署名されたトークンの場合、デバッガービューは少し異なります。
