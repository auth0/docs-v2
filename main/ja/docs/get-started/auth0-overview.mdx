title: Auth0 の概要
---

Auth0 は、アプリケーションに認証および認可サービスを追加するための、柔軟ですぐに導入できるソリューションです。独自のユーザー認証・認可ソリューションを構築する場合に伴うコスト、時間、リスクを、チームや組織は回避できます。

Auth0 のユースケースをいくつか見てみましょう。

* 優れたアプリを構築し、そこにユーザー認証と認可を追加したいとします。ユーザーは、識別子 (username、メールアドレス、または電話番号) とパスワード、またはソーシャルアカウント (Facebook や X など) でログインできる必要があります。さらに、ログイン後にユーザーのユーザープロファイルを取得して UI をカスタマイズし、認可ポリシーを適用したいと考えています。
* API を構築し、それを [OAuth 2.0](/ja/docs/authenticate/protocols/oauth) で保護したいとします。
* 複数のアプリを所有しており、[シングルサインオン (SSO)](/ja/docs/authenticate/single-sign-on) を実装したいとします。
* JavaScript のフロントエンドアプリとモバイルアプリを構築し、その両方から API に安全にアクセスさせたいとします。
* [Security Assertion Markup Language (SAML)](/ja/docs/authenticate/protocols/saml/saml-configuration) を使用してユーザーを認証する必要がある Web アプリがあるとします。
* パスワードには限界があると考えており、メールアドレスまたは SMS で送信されるワンタイムコードでユーザーがログインできるようにしたいとします。
* ユーザーのメールアドレスの 1 つが、あるサイトの公開データ侵害で漏えいした場合に通知を受け取りたいとします。さらに、そのユーザーに通知したり、パスワードをリセットするまでアプリへのログインをブロックしたりしたいとします。
* DDoS 攻撃を回避するため、ログイン失敗を繰り返す不審な IP アドレスを先回りしてブロックしたいとします。
* 大規模な組織の一員であり、既存のエンタープライズディレクトリサービスをフェデレーションして、従業員が既存のエンタープライズ認証情報を使用してさまざまな社内アプリケーションやサードパーティアプリケーションにログインできるようにしたいとします。
* 独自のユーザー管理ソリューションを実装したくない、あるいは実装方法がわからないとします。パスワードリセット、ユーザーの作成、プロビジョニング、ブロック、削除、それらすべてを管理するための UI が必要です。アプリそのものに集中したいだけです。
* ユーザーが機密データにアクセスしようとするときに、[多要素認証 (MFA)](/ja/docs/secure/multi-factor-authentication) を必須にしたいとします。
* SOC2、GDPR、PCI DSS、HIPAA など、増え続けるコンプライアンス要件への対応に役立つ ID ソリューションを探しているとします。
* サイトやアプリケーション上のユーザー行動を追跡したいとします。このデータを使用してファネルを作成し、ユーザー維持率を測定し、サインアップフローを改善する計画です。
* リソースに対する[関係性](https://docs.fga.dev/authorization-concepts)や、組織内での[役割](/ja/docs/manage-users/access-control/rbac)に基づいて、ユーザーのリソースへのアクセスを許可する堅牢な認可ポリシーが必要だとします。
