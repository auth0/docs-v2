validatedOn: 2026-02-17
---

Auth0 は、組織のブランド要件やユーザーの期待に沿ったルックアンドフィールにカスタマイズできます。Auth0 の各要素にブランディングを適用することで、顧客に一貫したユーザー体験を提供できるほか、信頼できる安全なプロバイダーの製品を利用しているという安心感を与えられます。

Auth0 は、[internationalization (I18N)](/ja/docs/customize/internationalization-and-localization) と <Tooltip tip="ローカライゼーション: 特定の地域や言語に合わせてコンテンツを調整すること。" cta="用語集を見る" href="/ja/docs/glossary?term=Localization">ローカライゼーション</Tooltip> (L10N) をサポートしています。これらはいずれも、国際的な顧客層に向けてブランディングを行う場合に重要です。Auth0 Lock UI ウィジェットなど、標準で提供される要素は複数言語のサポートがあらかじめ有効になっており、必要なものが[まだ用意されていない場合](/ja/docs/customize/internationalization-and-localization/lock-internationalization)でも、さらに言語を追加できる拡張性が組み込まれています。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  ほぼすべてのアプリケーションでは、何らかの形で国際化および/またはローカライゼーションが必要になります。Auth0 を使えば簡単に追加できますが、最初から考慮しておく必要があります。たとえばローカライゼーションを後付けで対応するのは、着手が遅れると大変な作業になることがあります。
</Callout>

どの項目をブランディングするか、またどのようにブランディングするのが最適かを検討する際は、確認しておきたい点がいくつかあります。

* ログインページをブランディングする必要がありますか？
* ログインページをローカライズする必要がありますか？
* メールを単にブランドに合わせるだけでなく、ユーザーの設定に応じて内容を変えられるようにするには、どのようにカスタマイズできますか？
* ユーザーがログインページを表示したときに、引き続き自社のドメイン上にいるとどう認識できるようにしますか？
* 追加のブラウザーセキュリティ (例: Extended Validation の実装) を提供するには何が必要ですか？
* エラー発生時にユーザーをどこへ誘導したいですか？

Auth0 では、[Universal Login](#universal-login-and-login-pages) や [パスワードリセット](#password-reset-page-customization) などの Auth0 ページをカスタマイズおよび設定するうえで、非常に高い柔軟性が提供されています。そのため、必要な UX のルックアンドフィールをほぼ自由に構成できます。多くの場合は、標準の体験を少し変更するだけで十分です。一方で、ブランド価値やブランド認知の観点から、より広範なカスタマイズが必要になる場合もあります。この柔軟性は Auth0 ページだけでなく、拡張性によって [メールテンプレート](#email-template-customization) にも適用できます。さらに、Auth0 の [カスタムドメイン](#custom-domain-naming) 機能により、安全性とセキュリティの面でユーザーに信頼感と安心感を与え、利用者の認知をいっそう高めることができます。

Auth0 はエラー発生時に既定の情報を提供しますが、標準の情報は、ユーザーに提供すべき文脈情報が不足しているため、やや分かりにくいことがあります。ただし、Auth0 の [エラーページのカスタマイズ](#error-page-customization) に関するガイダンスを利用すれば、自社のサポート窓口を通じて、より状況に即した情報を提供できるため、その問題を軽減できます。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  問題が発生したユーザーに役立つ情報を提供するために、わかりやすい表示名とロゴを設定し、さらに組織のサポート用メールアドレスと URL も提供してください。方法については、[ダッシュボードのテナント設定](/ja/docs/get-started/tenant-settings#settings) を参照してください。
</Callout>

<div id="universal-login-and-login-pages">
  ## Universal Login とログインページ
</div>

[Universal Login](/ja/docs/authenticate/login/auth0-universal-login) は、ユーザー認証の推奨方法であり、ログインページを中心に構成されています。ログインページは、組織のブランディング要件に合わせてカスタマイズできます。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Universal Login ページのスクリプトをカスタマイズする場合は、バージョン管理を使用することを強く推奨します。これを行うには、[デプロイの自動化](/ja/docs/get-started/architecture-scenarios/business-to-consumer/deployment) または [別の戦略](/ja/docs/customize/login-pages/classic-login/version-control) のいずれかを使用して、スクリプトを Auth0 テナントにデプロイしてください。
</Callout>

<div id="custom-domain-naming">
  ## カスタムドメイン名
</div>

デフォルトでは、テナントに関連付けられた URL には、その名前と、場合によってはリージョン固有の識別子が含まれます。たとえば、米国のテナントでは `https://example.auth0.com` のような URL になり、ヨーロッパのテナントでは `https://example.eu.auth0.com` のようになります。[カスタムドメイン](/ja/docs/customize/custom-domains) を使用すると、組織のブランドに合った名前を使って、ユーザーに一貫したエクスペリエンスを提供できます。

<Warning>
  Auth0 の各テナントに設定できるカスタムドメイン名は 1 つだけです。そのため、どうしてもドメイン名ごとに独立したブランディングが必要な場合は、複数の Auth0 テナントを本番環境にデプロイする[アーキテクチャ](/ja/docs/get-started/architecture-scenarios/business-to-consumer/architecture)が必要になります。
</Warning>

さらに、<Tooltip tip="カスタムドメイン: 特別な名前、またはバニティ名を持つサードパーティのドメイン。" cta="用語集を表示" href="/ja/docs/glossary?term=Custom+Domain">カスタムドメイン</Tooltip>機能を使用すると、証明書管理プロセスを完全に制御できます。デフォルトでは Auth0 が標準の SSL 証明書を提供しますが、カスタムドメインを設定すると、Extended Validation (EV) SSL 証明書などを使用して、訪問者により大きな安心感を与えるブラウザ上の視覚的な表示を提供できます。

一般に、認証に一元化されたドメインを使用しているお客様が、最も効果的に運用できています。これは、特に複数の製品やサービスブランドを提供している企業で顕著です。一元化されたドメインを使用することで、エンドユーザーに一貫したユーザーエクスペリエンスを提供できるだけでなく、Auth0 で複数の本番テナントを維持する必要性も最小限に抑えられます。

<div id="email-template-customization">
  ## メールテンプレートのカスタマイズ
</div>

Auth0 では、ユーザーへの通知や、安全なアイデンティティ管理に必要な機能 (たとえば、メールアドレスの確認、アカウント復旧、ブルートフォース攻撃からの保護) を実現するために、メールが広く使用されています。Auth0 には、これらに対応するテンプレートも複数用意されています。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  メールテンプレートをカスタマイズする前に、[Email Provider](/ja/docs/get-started/architecture-scenarios/business-to-consumer/operations#email-provider-setup) を設定してください。
</Callout>

デフォルトでは、メールテンプレートには標準的な文言と Auth0 のブランディングが含まれています。ただし、これらのテンプレートはほぼあらゆる要素を設定できるため、任意の文言やユーザー体験を反映したり、優先言語やアクセシビリティ オプションなどを変更したりできます。

メールテンプレートは、[Liquid syntax](/ja/docs/customize/email/email-templates/use-liquid-syntax-in-email-templates) を使用してカスタマイズします。ユーザー設定に基づいてテンプレートをカスタマイズする場合は、ユーザープロファイル内の [metadata](/ja/docs/manage-users/user-accounts/metadata) や、アプリケーション固有のメタデータにもアクセスできます。

<div id="password-reset-page-customization">
  ## パスワードリセットページのカスタマイズ
</div>

[パスワードリセット](/ja/docs/customize/login-pages/classic-login/customize-password-reset-page)ページは、ユーザーがパスワード変更機能を利用する際に使用されます。ログインページと同様に、組織固有のブランディング要件を反映するようにカスタマイズできます。

<div id="error-page-customization">
  ## エラーページのカスタマイズ
</div>

ユーザーのインタラクティブなワークフロー (たとえば、サインアップやログイン) 中に問題が発生した場合、Auth0 は内部的な原因を示すエラーメッセージを表示します。デフォルトのメッセージはやや分かりにくく、特にエンドユーザーにとっては、必要なコンテキスト情報が不足していることが多いため、なおさらです。そのため、不足している文脈に応じた情報をユーザーに直接提供できるよう、[エラーページをカスタマイズする](/ja/docs/customize/login-pages/custom-error-pages)ことをお勧めします。さらに、エラーページをカスタマイズすることで、Auth0 ではなく独自のブランディングを表示できるほか、次に何をすべきかについてユーザーに役立つ情報も提供できます。こうした情報には、FAQ へのリンクや、自社のサポートチームまたはヘルプデスクへの問い合わせ方法などが含まれる場合があります。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  標準では、Auth0 提供のエラーページをカスタマイズするためのユーザーインターフェースはありませんが、設定には [Management API のテナント設定エンドポイント](https://auth0.com/docs/api/management/v2#!/Tenants/patch_settings) を使用できます。独自のエラーページを作成してホストできる場合は、Auth0 がホストするページの代わりに、そのページへユーザーをリダイレクトすることもできます。
</Callout>

<div id="guardian-multi-factor-page-customization">
  ## Guardian の多要素認証ページのカスタマイズ
</div>

多要素認証ページは、[Universal Login Settings](https://manage.auth0.com/#/login_settings) セクションで Universal Login のブランディングオプションを調整することでカスタマイズできます。

さらに細かくカスタマイズする必要がある場合は、組織固有の UX 要件に合わせて、[HTML コンテンツ全体](/ja/docs/secure/multi-factor-authentication/customize-mfa/customize-mfa-classic-login) をカスタマイズすることもできます。
